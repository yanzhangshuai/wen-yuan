# fix: threestage 全书解析收口与空角色诊断

## 背景

当前书籍导入解析存在两条业务链路：

- `threestage`：全书三阶段解析
- `sequential`：逐章解析

用户已明确本任务采用方案 1：

1. `threestage` 作为全书解析的唯一主架构继续推进。
2. 不处理按章分析链路，保持现状。
3. 重点解决 `threestage` 的历史兼容残留、读写模型不一致，以及异常全书结果不可观测的问题。

同时，书籍 `61c34092-5295-4438-af28-46b5b0e9d558` 的最新 `threestage` 作业虽然成功，但前台没有任何角色，属于必须解释并修复的业务异常。

## Goal

建立一个可实施的设计，指导后续把 `threestage` 收口为全书解析单一路径，并修复“作业成功但角色为 0”的问题。

## 业务结论

- 当前 `threestage` 仍带有旧架构残留，不适合继续长期双语义并存。
- 书籍 `61c34092-5295-4438-af28-46b5b0e9d558` 不是“没解析出角色”，而是“写到了 `persona/persona_mentions/biography_records`，但读取仍依赖 `profiles`，因此展示为 0”。
- 该书全书结果本身也异常：56 章里只有第 31 章落出有效数据，这说明除了读写模型错位，还存在可观测性与阶段执行诊断不足。

## Requirements

### 1. 架构收口

- 明确 `threestage` 是全书解析唯一主路径。
- 清点并去除所有“旧 `twopass`/旧 persona-profile 语义仍是主读取口径”的兼容设计。
- 明确 `Profile` 在未来是移除、降级为投影，还是仅保留历史数据兼容，不再作为主业务真相。

### 2. 角色读取真相统一

- 明确书籍人物的单一真相源。
- 后续实现必须保证：只要 Stage B 已晋级出 persona，列表页和详情页不能再出现“0 角色”。
- 读模型、聚合查询、统计口径必须与三阶段写模型一致。

### 3. 异常作业诊断能力

- 为 `threestage` 增加阶段级可观测性要求。
- 能够定位：
  - Stage A 是否处理了全部章节
  - Stage B 是否实际消费了全部 mention
  - Stage C 是否只在部分章节归因
  - 每阶段失败、跳过、降级的原因

### 4. 针对异常书籍的验证闭环

- 以书籍 `61c34092-5295-4438-af28-46b5b0e9d558` 为回归样本。
- 后续实现后，至少应验证：
  - 角色列表不再为 0
  - 作业阶段日志完整可见
  - 能解释为何之前只有第 31 章产出数据

## Non-Goals

- 不修改 `sequential` / 按章分析架构。
- 不把按章分析并入本次收口任务。
- 不在本 PRD 中承诺“当前已稳定达到 85%+ 角色与事迹准确率”。
- 不做与本问题无关的 UI 重设计。
