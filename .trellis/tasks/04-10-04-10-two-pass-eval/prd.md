# brainstorm: 两遍式架构评估与优化

## Goal

基于已实现的两遍式（Two-Pass）架构，量化评估准确率和 token 消耗，分析跨书通用性，并探索更优方案。

### 成本对比

| | 原始架构 | 两遍式架构 | 变化 |
|---|---------|-----------|------|
| **总调用次数** | ~148 | ~156 | +5.4% |
| **总 tokens** | ~2,332K | ~2,208K | **-5.3%** |
| **总成本** | ~¥4.39 | ~¥2.34 | **-46.7%** |

## 准确率估算

### 原始架构痛点

| 错误类型 | 根因 | 两遍式是否消除 |
|---------|------|-------------|
| entityId 数字选错 → 跨人物合并 | buildEntityIdMap 用数组下标, LLM 从200行列表选编号 | ✅ Pass 1 不传 profiles, 无 entityId |
| rosterMap 无条件信任 0.97 | PersonaResolver 不校验名字相似度 | ✅ 已加名义核验 |
| 级联污染（错误 alias 传播） | 每章累积写入, 前序错误污染后序 | ✅ Pass 2 一次性全局消歧, 无累积 |
| 垃圾实体（整句话当人名） | AI 幻觉无长度过滤 | ✅ name ≤ 10 字符过滤 |
| 同名异人错误合并（娄三/娄四） | 编辑距离太近 + 姓氏相同 → 被强制合并 | ⚠️ 部分 — 规则层仍可能误组, 但 LLM 消歧层可以拆分 |

### 准确率预估

| 错误类型 | 原始架构影响量 | 两遍式预期 |
|---------|--------------|-----------|
| 跨人物错误合并 | ~30-50 个合并错误 | **<5 个** (Pass 2 LLM 判断, 可拆分) |
| 垃圾实体 | ~10-20 个 | **~0-2 个** (长度过滤 + 类别过滤) |
| 遗漏实体 | ~5-10 个 | **~5-10 个** (与原始类似, Pass 1 独立提取能力相当) |
| 关系错配 | 因人物合并导致 ~20% 关系错配 | **<3%** (人物对齐正确后, 关系自然正确) |

**entityF1 预估**: 0.74 → **0.85-0.90**  
**relationF1 预估**: 0.68 → **0.80-0.85**

## 跨书通用性分析

### 当前状态

1. **体裁预设已覆盖 7 种**：明清官场、武侠、宫廷家族、英雄传奇、历史演义、家族世情、神魔小说
2. **但选择机制是硬编码**：`resolveBookLexiconConfig()` 始终返回 `明清官场` 预设, 参数 `_bookTitle` 被忽略
3. **DB 没有 genre 字段**：Book 模型无体裁字段, 无法存储/选择预设
4. **Pass 1 prompt 是通用的**：`buildIndependentExtractionPrompt` 只传书名+正文, 无体裁特化逻辑, 通用性好
5. **Pass 2 消歧是通用的**：Union-Find + 编辑距离 + LLM 判断, 不依赖体裁

### 各书挑战分析

| 书籍 | 人物数 | 特殊挑战 | 当前系统覆盖度 |
|------|--------|---------|-------------|
| 儒林外史 | ~220 | 章回独立, 人物称谓多变 | ✅ 较好 |
| 红楼梦 | ~400+ | 亲属称呼极度复杂（贾母/老太太/史太君是同一人）、丫鬟改名（袭人/花珍珠） | ⚠️ 家族世情预设有, 但 resolveBookLexiconConfig 不会选中 |
| 三国演义 | ~1000+ | 人物极多, 字(子龙)/号(卧龙)/谥(武侯)系统, 同名异人（多个"张飞"） | ⚠️ 历史演义预设有, 但 LLM 消歧批量会很大 |
| 水浒传 | ~108+ | 绰号系统（及时雨=宋江）, 军职称号 | ⚠️ 英雄传奇预设有 |
| 西游记 | ~200+ | 法号/本相/神魔称号, 变化身份 | ⚠️ 神魔小说预设有 |
| 金瓶梅 | ~200+ | 类红楼, 家族辈分 + 称呼场景多 | ⚠️ 需独立预设 |

### 跨书通用性瓶颈

1. **体裁选择未打通**：有预设但不会被激活（`_bookTitle` 被忽略）
2. **人物规模天花板**：三国演义 1000+ 人物 → Pass 2 候选组爆炸, LLM 消歧调用可能达 50-100 批
3. **领域知识缺失**：字/号/谥系统需要额外规则（如"诸葛亮"="孔明"="卧龙"不能仅靠编辑距离）
4. **称谓上下文**：红楼梦中"太太"在不同章节指不同人, 仅靠全局消歧无法处理

## 可优化方向

### A. 激活体裁预设绑定（低成本, 高收益）

- 给 Book 增加 `genre` 字段
- `resolveBookLexiconConfig()` 根据 genre 选择预设
- 新书上传或开始解析时, 用 LLM 一次性判定体裁（或用户选择）
- **预期收益**：准确率 +2-5% (减少泛化称谓误杀)，无额外成本

### B. Pass 1 章节摘要注入（中成本, 中收益）

- 在 Pass 1 结束后, 为每章生成 100 字摘要
- Pass 3 的 CHUNK_EXTRACTION prompt 注入本章摘要而非 profiles 列表
- **预期收益**：减少 Pass 3 prompt token（摘要 < profiles），提高上下文理解
- **额外成本**：每章多 ~200 tokens 摘要生成

### C. 字号/谥号知识库（中成本, 特定书籍高收益）

- 维护一个可扩展的"字号映射表"配置文件（如 `诸葛亮 = [孔明, 卧龙, 武侯]`）
- Pass 2 消歧时先查知识库, 知识库命中直接合并, 不走 LLM
- **预期收益**：三国演义等书准确率 +10-20%, 减少 LLM 消歧调用 30-50%
- **局限**：需要人工维护, 仅适用于经典名著

### D. 渐进式消歧（高成本, 大规模书籍专用）

- Pass 2 分两轮: (1) 规则确定合并 → (2) LLM 只处理剩余模糊候选
- 增加"跨章共现分析": 两个名字在同一段落/对话中共现 → 不是同一人
- **预期收益**：三国演义等超大书籍 token 节省 40-60%
- **风险**：实现复杂度高

### E. 全书级别的增量消歧（替代方案, 根本性改进）

- 不做一次性 Pass 2, 改为"滑窗式"消歧:
  - 每处理 5 章, 做一次局部消歧
  - 最后做一次全局对齐
- **优点**：内存友好, 可处理超长书（100+ 章）
- **缺点**：更复杂, 可能引入滑窗边界错误

## Open Questions

1. **你优先关注成本还是准确率？** — 目前成本已降 ~47%, 如果准确率足够（F1>0.85）, 是否还需要继续压缩成本？
2. **你计划支持哪些书？** — 如果只是儒林外史 + 红楼梦, 方案 A+C 足够；如果要支持三国演义等超大规模书, 需要方案 D

## Technical Notes

- 文件 `resolveBookLexiconConfig()` 在 ChapterAnalysisService.ts:1401, 当前硬编码返回 `明清官场`
- DB schema 中 Book 没有 genre 字段
- GENRE_PRESETS 已有 7 个预设, 但无选择逻辑
- Pass 1 prompt (`buildIndependentExtractionPrompt`) 不含体裁特化, 通用性好
- Pass 2 `buildCandidateGroups()` 用 Union-Find + 编辑距离, 对字号体系效果有限
