## 一、产品愿景

**文渊**将枯燥的长篇古文小说转化为可视化、可交互的人物命运图谱。
它不是阅读器，而是"文学考古实验室"——通过 AI 逻辑推理，从百万字非结构化文本中挖掘人物社会网络、命运轨迹与时空变迁。

## 二、目标用户

| 角色 | 核心需求 |
| --- | --- |
| 文学研究者 | 梳理《红楼梦》、《儒林外史》等复杂人物网与籍贯迁徙 |
| 学生 / 爱好者 | 通过可视化直观理解剧情走向与人物关系 |
| 内容创作者 | 以图谱作为创作素材或逻辑校验工具 |
| 管理员 | 审核 AI 解析结果、维护数据质量 |

## 四、设计系统规范

### 4.1 布局与断点

- **目标设备**：PC 桌面优先，最小支持宽度 `1280px`
- 书库网格列数：`≥1536px` → 6 列；`1280–1535px` → 5 列；`<1280px` → 不做适配（布局允许出现横向滚动条）
- 内容区最大宽度：`1440px`（居中）

### 4.2 主题系统

支持"可扩展主题体系"，当前内置**三套主题**（含一套完整暗色模式）。主题通过 CSS 设计变量（Design Tokens）解耦，原则是：**所有视觉差异只写在主题 token 层，业务组件不感知主题**。

#### 4.2.5 主题切换规范

- 切换入口：顶部导航右上角**下拉选择器**（前台与后台保持一致），三套主题均可选：`古风 / 简约 / 暗色`
- 切换动画：`transition: color 300ms ease, background-color 300ms ease`，next-themes 注入 `data-theme` 属性，禁止页面闪烁（SSR 侧需注入初始主题脚本）
- 偏好持久化：写入 `localStorage`，key 为 `wen-yuan-theme`
- 新增主题：只允许新增主题 token 文件，不允许修改业务组件结构

#### 4.2.6 暗色模式策略

`theme-03`（星空暗色模式）即为系统的完整暗色主题，与其他两套亮色主题并列，由用户手动选择，**不**自动跟随系统 `prefers-color-scheme: dark`。  
后续如需添加自动跟随，只需为每套主题扩展 `dark` 变量集（业务组件不改）；当前 M0 阶段不实现此能力。

### 4.3 动画规范

| 场景 | 方案 | 时长 / 曲线 |
| --- | --- | --- |
| 主题切换 | CSS transition | `300ms ease` |
| 书籍卡片 Hover 抬起 | Framer Motion | `200ms ease-out`，`y: -8px`，`shadow` 扩散 |
| 页面路由切换 | Framer Motion `AnimatePresence` | `250ms ease-in-out`，淡入淡出 |
| 图谱节点渐入 | D3 transition / Framer | `400ms ease`，opacity 0→1 |
| 对话框 / 面板滑入 | Framer Motion | `200ms ease-out`，`x: 16px` → 0 |
| 加载骨架屏闪动 | CSS `@keyframes shimmer` | `1.5s linear infinite` |

全局禁用 `prefers-reduced-motion` 时所有上述动画归零（使用 Tailwind `motion-safe:` 工具类或 Framer `useReducedMotion`）。

### 4.4 无障碍基线

- 最低目标：**WCAG 2.1 AA** 对比度（正文 4.5:1，大字 / 图标 3:1）
- 两套主题在 `--color-fg` / `--color-bg` 组合上必须通过对比度验收
- 所有可交互元素须有明确的 `:focus-visible` 轮廓（可定制颜色以匹配主题）
- 图谱中纯颜色区分的节点/边必须同时提供形状或标签辅助（避免纯色盲障碍）

### 4.5 全局 UI 状态模式

| 状态 | 实现方式 | 阶段 |
| --- | --- | --- |

### 4.6 字体与图标系统

#### 4.6.1 字体加载方案

| 用途 | 字族 | 加载方式 | Weight |
| --- | --- | --- | --- |
| theme-01 标题 / 正文 | `"Noto Serif SC"` | Google Fonts CDN `https://fonts.googleapis.com/css2?family=Noto+Serif+SC:wght@400;500;600;700&display=swap` | 400 / 500 / 600 / 700 |
| theme-02 标题 / 正文 | `"Inter"` | Google Fonts CDN `https://fonts.googleapis.com/css2?family=Inter:wght@400;500;600;700&display=swap` | 400 / 500 / 600 / 700 |
| theme-03 标题 / 正文 | `"Inter"` | 同 theme-02（共用 CDN 链接，无需重复加载） | 400 / 500 / 600 / 700 |
| 代码 / 等宽 | `"JetBrains Mono"` | Google Fonts CDN `https://fonts.googleapis.com/css2?family=JetBrains+Mono:wght@400;500&display=swap` | 400 / 500 |

**Fallback 字体栈**

```css
/* theme-01 */
font-family: "Noto Serif SC", "Songti SC", "SimSun", "STSong", serif;

/* theme-02 */
font-family: "Inter", -apple-system, BlinkMacSystemFont, "Segoe UI", "Helvetica Neue", sans-serif;

/* 代码 */
font-family: "JetBrains Mono", "Fira Code", "Cascadia Code", ui-monospace, monospace;
```

**加载策略**

- 字体 CDN 链接统一写在 `src/app/layout.tsx` 的 `<head>` 中（`<link rel="preconnect">` + `<link rel="stylesheet">`）
- 使用 `font-display: swap` 避免 FOIT（内容不可见闪烁）
- 网络不可用时自动降级到 Fallback，不影响功能

#### 4.6.3 图标库规范

- **主图标库**：`lucide-react`（已安装）
- **使用原则**：
  - 所有图标使用 `lucide-react` 导出的 React 组件，不使用 SVG 内联或 `<img>` 引用
  - 尺寸统一通过 `size` prop 控制，默认 `size={16}`；工具栏图标 `size={18}`；页面级大图标 `size={24}`
  - 颜色继承父元素 `currentColor`，不硬编码色值，保证主题联动
  - 图标必须搭配 `aria-label` 或包裹在带 `title` 的 `<button>` 内，保障可访问性
- **中文书法装饰图标**（仅 theme-01 使用）：使用 SVG 自绘印章、毛笔笔划元素，放在 `src/components/ui/decorative/` 下，不引入第三方图标字体
- **禁止**：不引入 Font Awesome、iconfont 字体图标、`react-icons` 等额外图标库

### 5.1 首页 / 书库（Library）

**路由：** `/`  
**设计基调：** 仿真书架，现代博物馆感，第一眼“一亮”

#### 5.1.1 视觉呈现

- **仿真书籍卡片**：每本书以“正常实体书比例”展示（推荐宽高比 `2:3`，可带书脊厚度 3D 视觉），卡片包含：
  - 封面图（主视觉，优先占据卡片主体区域）
  - 书名（可放封面下沿或书脊，采用绝对定位）
  - 作者 / 朝代 / 统计信息（默认隐藏，按 Hover 或详情浮层展开）
  - 前台不展示“待解析 / 解析中 / 失败”等状态 Badge
- **网格布局**：书架木纹背景，书籍按“影响力”或“导入时间”排列
- **Hover 效果**：书籍抬起 + 阴影扩散动画（Framer Motion）
- **空状态**：前台显示“暂无可阅读书籍”提示，不提供导入入口 CTA

#### 5.1.2 交互

- 点击**已完成**的书籍 → 进入人物图谱页
- 未解析成功（`PENDING` / `PROCESSING` / `ERROR`）书籍在前台显示为灰度卡片，且**不可点击**
- 前台不提供“重新解析 / 删除 / 导入”入口，相关操作全部收敛到 `/admin/*`
- 默认前台顶部导航保持精简，仅包含：`[文渊 Logo]` `[书库]` `[主题下拉选择]`
- 管理员相关入口不在默认前台导航中常驻显示；进入 `/admin/*` 后由独立后台导航承载 `管理审核`、`模型设置` 等功能

#### 5.1.3 数据

- API：`GET /api/books` → 返回书籍列表（含状态、统计数量、封面与可读性标记）
- 状态字段：`PENDING` / `PROCESSING` / `COMPLETED` / `ERROR`
- 前台渲染口径：
  - 仅 `COMPLETED` 书籍可点击进入图谱
  - 非 `COMPLETED` 书籍统一灰度展示
  - 解析状态文案、进度细节仅在运营端展示

#### 5.1.4 书库卡片数据来源说明

这是本版新增的必需能力，用于让用户理解卡片上每个数字与状态从哪里来。

- `章节数`：来自管理员确认后的 `Chapter` 记录数
- `人物数`：来自当前书下有效人物档案数，不计 `REJECTED` / 软删除数据
- `最近解析时间`：最近一次解析任务完成时间；失败任务也计入最近任务时间
- `当前模型`：当前书默认模型；若最近一次任务使用了覆盖模型，则优先展示最近一次任务模型
- `失败摘要`：最近一次失败任务的错误摘要截断显示

产品要求：

- 卡片上应有“数据说明”入口，形式可为 Tooltip、Popover 或 Drawer
- 数据说明文案必须与接口真实统计口径一致

#### 5.1.5 可执行说明

- MVP v1 必须交付可用的书库列表、封面主视觉、真实书籍比例卡片和数据来源说明
- 前台不显示解析状态和进度；未解析成功卡片灰度不可点击
- 3D 书脊、木纹书架、动效属于本版正式需求；工程上可分阶段细化视觉质量，但不能从 PRD、交互稿或排期口径中删除

### 5.2 书籍导入（Import Flow）

**入口：** `/admin/books/import`（运营端）  
**实现方式：** `admin` 后台独立页面或后台布局内抽屉

约束：

- 所有“导入书籍 / 启动解析 / 重解析”动作只允许在 `/admin/*` 完成
- 前台（viewer）不提供导入入口与解析操作入口

#### 5.2.1 导入流程（4 步向导）

```text
Step 1: 上传文件
Step 2: 基本信息填写
Step 3: 章节预览 & 切分确认
Step 4: 启动 AI 解析
```

**Step 1 — 文件上传**
- 支持格式：`.txt`（MVP 必须）、`.epub`（可选）
- 拖拽上传区域 + 点击选择
- 文件大小限制：50MB
- 上传后显示文件名 + 大小
- 上传成功后，原始文件必须经统一 Storage Provider 保存，不允许只停留在内存或仅写入数据库文本字段

**Step 2 — 基本信息**

- 字段：书名（必填）、作者（可选）、朝代（可选）、简介（可选）
- 文件上传后 AI 优先从文本前言识别书名、作者、朝代、简介；书名识别失败时回退使用文件名（去扩展名）
- 所有字段用户均可修改

**Step 3 — 智能章节切分预览**

- 系统自动识别章节标题：
  - 中式：`第X回`、`第X章`、`楔子`、`后记`
  - 英式：`Chapter X`
- 预览表格：`序号 | 章节类型 | 标题 | 字数`
- 用户可手动调整：合并章节、修改标题、标记楔子 / 正文 / 后记
- 切分逻辑分两级：
  - 自动（正则 + 模型辅助）
  - 手动修正

**Step 4 — 启动解析**

**模型选择（必须在此步骤选定）：**

| 模型 | 提供商 | 定位 | 适用场景 |
| --- | --- | --- | --- |
| DeepSeek V3 | DeepSeek | 逻辑推理强、古文理解佳 | 默认推荐，精准解析 |
| DeepSeek R1 | DeepSeek | 慢思考推理 | 复杂关系二次复核 |
| 通义千问 Max (qwen-max) | 阿里云 | 中文语义理解强 | 人物别名消歧 |
| 通义千问 Plus (qwen-plus) | 阿里云 | 速度与质量均衡 | 中等篇幅书籍 |
| 豆包 Pro (doubao-pro) | 字节跳动 | 国内大模型，性价比高 | 批量快速扫描 |
| Gemini Flash | Google | 速度极快 | 备用 / 海外场景 |

- UI 展示为**模型卡片选择**，每张卡片显示：模型名、提供商 Logo、速度评级（🐇/🐢）、古文能力评级（⭐~⭐⭐⭐）、大致费用等级
- 用户也可在**系统模型管理页**配置各模型的 API Key 和自定义 BaseURL（支持接入私有化部署的兼容接口）
- 解析粒度：`全书` / `选定章节`
- 点击"开始解析"后关闭导入窗口，书籍进入 `PROCESSING` 状态，解析时使用的模型名称记录在 `Book` 记录上供审计追溯

##### 导入文件与静态资源存储规则

这是本版新增的必需能力，用来统一管理原始书籍文件、封面图与后续图片资源。

- M0 默认使用**本地文件系统存储**，不默认启用本地 S3 服务
- 存储访问统一走 `src/server/providers/storage/` 抽象层，风格与 `src/server/providers/ai/` 保持一致
- 当前推荐 Provider：
  - `local`：默认启用，适合个人项目、单机部署和低运维成本场景
  - `oss`：后续扩展 Provider，用于阿里云 OSS
- 业务模块不得直接写死本地路径、MinIO SDK 或 OSS SDK；导入、封面、图片都必须经 Storage Provider
- 本地路径建议统一为：
  - `storage/books/<bookId>/source/...`
  - `storage/books/<bookId>/cover/...`
  - `storage/books/<bookId>/images/...`
- 数据库存储口径：
  - 至少保存 `sourceFileUrl`
  - 同时保存 `sourceFileKey`、`sourceFileName`、`sourceFileMime`、`sourceFileSize`
- 本地存储也必须可通过链接访问：
  - 推荐由统一资源访问路由代理（如 `/api/assets/[...key]`）
  - `sourceFileUrl` 存可访问 URL，`sourceFileKey` 存稳定对象 key
  - 禁止把服务器绝对路径直接暴露给前端
- `rawContent` 继续保留在数据库，用于章节切分、重解析和服务端兜底；它不替代原始文件存储

当前不默认推荐“本地先跑一个 S3 服务”的原因：

- 对个人项目来说会增加服务启动、权限、端口和配置维护成本
- 当前阶段主链路优先级是“导入 -> 解析 -> 图谱 -> 审核”，不应被额外基础设施阻塞
- 先抽象 Provider，再切 OSS，能保留未来扩展性而不增加现在的运维负担

#### 5.2.2 解析进度展示

- 运营端显示解析进度（轮询 `/api/books/:id/status`）
- 进度阶段：`文本清洗 → 章节切分 → 实体提取 → 关系建模 → 完成`
- 解析失败时，运营端展示错误摘要 + `重试` 按钮
- 前台书库不展示解析进度与失败摘要，仅按“可阅读 / 不可阅读”状态展示卡片

#### 5.2.3 AI 解析内容（后端输出）

每章解析：

- **实体识别**：**书中出现的每一个人物必须全量提取，不得遗漏**，包括：
  - 有名有姓的人物（如“范进”“周进”）
  - 仅有称号 / 官职 / 绰号而无名字的人物（如“那老翁”“严监生的内侄”“那钦差大人”），以称号作为标准名，标注 `nameType: TITLE_ONLY`
  - 匿名群体性人物（如“众邻居”“几个秀才”）**忽略，不提取**
- **姓名、别名**（如“周进 = 周学道”）、性别、籍贯、官职
- **传记事件**：时间点、地点、事件、职位（四元组）
- **关系提取**：人物 A → 关系类型 → 人物 B + 原文证据片段
- **人物标签**：性格 / 社会角色标签（如“伪君子”“真名士”“市井小人”）
- **讽刺指数**：0–10 分，附 AI 点评

关系类型（20+）：
`父子` `母子` `兄弟` `夫妻` `姻亲` `师生` `同年` `荐举` `债主` `债务人` `友好` `敌对` `下属` `上司` `同僚` `欣赏` `嘲讽` `同盟` `竞争` `其他`

#### 5.2.4 AI 解析准确度要求

##### 目标：主要人物识别率 ≥ 95%，主要关系识别率 ≥ 95%

##### 准确率定义（可验收）

| 维度 | 定义 | 目标 |
| --- | --- | --- |
| 主要人物识别率 | 全书出场 ≥ 3 次的人物被正确提取的比例 | ≥ 95% |
| 别名消歧准确率 | 同一人物的多个称谓被正确归并的比例（如"周进 = 周学道"） | ≥ 90% |
| 主要关系识别率 | 全书明确出现的亲属 / 师生 / 同年等主要关系被正确提取的比例 | ≥ 95% |
| 关系误报率 | AI 提取的关系中，经人工核对为错误的比例 | ≤ 10% |
| 原文证据引用率 | 每条关系附带有效原文证据片段的比例 | ≥ 90% |

> 验收方式：以《儒林外史》前 20 回为基准测试集，人工标注后与 AI 输出对比计算。

##### 达到 95% 准确度的工程机制

1. **分章解析 + 全局上下文注入**
   - 每章解析时，Prompt 中携带“已识别人物列表”作为上下文，避免跨章节遗漏 / 重复识别
   - 每 10 章做一次全局汇总，AI 对人物别名进行跨章消歧

2. **结构化 Prompt + 强约束输出**
   - 要求 AI 以 JSON Schema 格式输出，字段缺失时必须填 `null` 而非省略
   - Prompt 中提供古文人名识别规则示例（Few-shot），降低模型幻觉

3. **置信度字段**
   - AI 为每个实体 / 关系打置信度分（0.0–1.0），低于 0.7 的结果自动标记为“待重点核对”
   - 图谱中低置信度节点 / 边额外显示警告图标

4. **指代消解二次扫描**
   - 全书解析完成后，自动扫描“名字相似度 > 80%”的 Persona，归入“合并建议”队列
   - 管理审核页优先展示合并建议，引导人工确认

5. **重解析机制**
   - 支持对单章或全书发起“重解析”，可切换更强模型对低置信度章节进行二次提取
   - 两次解析结果取并集，冲突时标记为“待仲裁”

6. **人工校对闭环**
   - DRAFT 数据经人工 VERIFY 后，优质样本可沉淀为该书的 Few-shot 示例，用于后续章节解析

#### 5.2.5 重解析粒度定义

这是本版新增的必需能力。

- 粒度：
  - `整书重跑`
  - `单章重跑`
- 模型策略：
  - 沿用当前书默认模型
  - 本次任务指定覆盖模型
- 覆盖策略：
  - `仅覆盖未审核草稿`（推荐默认）
  - `覆盖全部草稿`
- 版本策略：
  - `不保留历史草稿版本`
  - `保留任务批次与历史草稿版本`

约束：

- 已 `VERIFIED` 数据不得被重解析自动覆盖
- 被覆盖数据应归档或软删除，不做硬删除
- 重解析入口可出现在书库卡片菜单、审核页和图谱页管理操作中

#### 5.2.6 原文锚点输出要求

为支撑审核、证据链和高亮回跳，AI 输出至少需要保留以下锚点之一：

1. `chapterId + paraIndex`
2. `chapterId + evidenceText`
3. `chapterId + paraIndex + evidenceText`

推荐统一采用：

- `chapterId`
- `paraIndex`
- `evidenceText`
- `confidence`

#### 5.2.7 可执行说明

- MVP v1 主链路只强制启用 `.txt`
- `.epub` 保留在产品能力定义中，但可以放入后续实现阶段
- “全书解析任务”是 MVP v1 必做；“选定章节解析”不从产品文档删除，但在实现上优先通过“单章重解析”满足
- 导入与解析入口默认在运营端，不在前台开放

### 5.7 登录系统

**路由：** `/login`  
**默认状态：** 未登录用户以 `viewer` 身份只读访问全站；只有管理员动作与 `/admin/*` 需要登录

#### 5.7.1 设计原则

- **默认 viewer**：未登录用户可访问书库、图谱、人物详情与原文阅读，不需要任何认证
- **管理员后台集中入口**：所有后台管理页面统一收敛在 `/admin/*` 路由下，作为独立后台模块管理
- **运营动作后台化**：书籍导入、解析触发、重解析与模型管理只在 `/admin/*` 提供
- **按需登录**：点击管理员操作按钮，或直接访问 `/admin/*` 页面时，统一跳转 `/login?redirect=<当前路径>`
- **单一 admin 账号**：无注册、无找回密码、无多租户
- **账号入库**：账号存储于 `users` 表；初始管理员通过 seed 从环境变量写入
- **实现简单**：不引入第三方 Auth 库，使用项目内鉴权模块完成登录态校验

#### 5.7.2 角色权限表

| 功能 | admin | viewer |
| --- | --- | --- |
| 浏览书库、查看图谱 | ✅ | ✅ |
| 点击节点查看人物详情 | ✅ | ✅ |
| 打开原文阅读 / 高亮回跳 | ✅ | ✅ |
| 导入书籍 / 触发 AI 解析 | ✅ | ❌ → 触发登录 |
| 发起重解析（单章 / 整书） | ✅ | ❌ → 触发登录 |
| 图谱内联校对（编辑节点/关系） | ✅ | ❌ → 触发登录 |
| 管理审核页（确认/拒绝/合并） | ✅ | ❌ → 触发登录 |
| 手动新增 / 删除人物与关系 | ✅ | ❌ → 触发登录 |
| 模型管理（API Key 配置） | ✅ | ❌ → 触发登录 |

#### 5.7.3 登录页 / 登录 Modal

- 居中卡片：邮箱或用户名 + 密码输入框 + 登录按钮
- 凭证错误时统一显示“账号或密码错误”，不区分具体原因
- 登录成功 → 写 `httpOnly` Cookie（JWT，7 天有效期）→ 跳回 `redirect` 参数页面
- `redirect` 必须完整保留原始路径与查询参数，并在跳转链路中完成 URL 编码 / 解码
- 无 `redirect` 时默认跳转 `/`

#### 5.7.4 鉴权机制

- **Next.js Middleware**：读取 JWT Cookie，有效则注入 `x-auth-role: admin`，无效或缺失则注入 `x-auth-role: viewer`
- **Admin Layout 守卫**：`/admin/*` 使用独立 layout，在服务端再次校验登录态与角色，避免仅依赖 Middleware
- **受保护页面**：所有 `/admin/*` 页面，viewer 访问直接重定向 `/login?redirect=<目标路径>`
- **普通页面的管理员操作**：Route Handler 调用 `requireAdmin(auth)` 返回 `403`，前端收到后跳转 `/login?redirect=<当前路径>`
- **Redirect 规范**：统一传递“当前 `pathname + search`”，写入 URL 前编码，登录页消费后解码并校验为站内路径
- JWT 仅包含 `{ role: "admin", iat, exp }`
- 环境变量：`ADMIN_USERNAME`、`ADMIN_EMAIL`、`ADMIN_NAME`（可选）、`ADMIN_PASSWORD`、`JWT_SECRET`、`APP_ENCRYPTION_KEY`
- 登出：清除 Cookie，保留当前页，角色降级为 `viewer`


**API 路由鉴权策略（混合模式）：**

| 层级 | 路径 | 保护机制 |
| --- | --- | --- |
| Tier 1 | `/api/admin/*` | Middleware `matcher` 自动拦截（viewer 重定向 `/login`） |
| Tier 2 | `/api/<resource>/*` | Route Handler 内显式调用 `requireAdmin()` 守卫写操作 |

Middleware `matcher` 配置：`['/admin/:path*', '/api/admin/:path*']`
#### 5.7.5 顶部导航

## 九、MVP 范围界定（全量需求，按模块分批实现）

### 9.1 范围表达原则

- `5.1` 到 `5.7` 均为正式需求，全部在本版 M0 中完成
- 不设 M1 拆批，所有功能（含视觉增强、性能优化、图谱校对、路径查找、epub）均需交付
- 允许按功能模块分批实现，但不允许把任何一条正式需求后置或降级为"文档备注"

### 9.2 M0（全量目标）

M0 目标：

`admin 导入 txt/epub -> 元数据确认 -> 章节切分预览 -> 全书解析任务 -> 前台书库可读浏览 -> 单书图谱浏览 -> 原文阅读/高亮回跳 -> 管理员审核队列 -> 图谱内联校对 -> 手动人物管理 -> 路径查找 -> 沉浸式视觉`

M0 必须包含：

1. 运营端书籍导入（`.txt` / `.epub`）+ 元数据确认 + 章节切分预览（含手动修正）
2. 全书解析任务（含进度阶段、失败摘要、重试入口），进度与错误信息在运营端展示
3. 书库列表与卡片数据来源说明（章节数、人物数、最近解析时间、当前模型、失败摘要）
4. 书库卡片以封面为主视觉，信息可折叠或隐藏至二级交互；未解析成功卡片灰度且不可点击
5. 单书图谱浏览（基础力导向渲染）+ 人物详情面板 + 章节时间轴
6. 原文阅读 / 高亮回跳（关系、事件、审核条目都可回跳）
7. 管理员审核队列（人物草稿、关系草稿、传记事件）
8. 合并建议队列 + 实体合并工具（不是仅算法描述）
9. 重解析粒度首版：整书重跑、单章重跑、指定模型、覆盖策略、版本策略
10. 模型设置关键能力：API Key 脱敏展示、BaseURL、启用禁用、默认模型、联通性测试
11. 登录与鉴权：默认 viewer、按需登录、`/admin/*` 双重守卫
12. 统一静态资源存储底座：M0 默认 `local` Provider，统一接口，预留 `oss` Provider
13. 主题系统首版：下拉选择"古朴中国风 / 简约风 / 星空暗色"三套，变量化解耦，支持后续新增主题
14. 图谱内联校对（右键节点 / 边直接编辑，编辑态表单，状态确认 / 拒绝）
15. 手动人物管理（新增 / 编辑 / 删除人物、关系、传记事件，`recordSource: MANUAL`）
16. 3D 书架视觉细化、图谱沉浸式视觉（玻璃拟态详情面板、节点发光、边动画）与更完整交互动效
17. 路径查找（Neo4j 图算法）与图谱布局切换 / 导出 / 全屏
18. Canvas 渲染、语义缩放、视口裁剪、Web Worker 布局计算（大图性能保障）
19. `.epub` 格式支持（解析与导入流程与 `.txt` 一致）

补充说明：

- 允许按功能模块分批实现（建议顺序：主链路闭环 → 内联校对与手动管理 → 路径查找 → 图形性能增强 → 视觉精修），所有条目均在本版 M0 范围内，不后置

## 十二、技术注意事项

### 12.1 双数据库与图谱策略

- **双数据库策略**：Prisma（PostgreSQL）存储结构化数据；Neo4j 存储图关系，用于路径查找和社区检测
- M0 以 PostgreSQL + 前端 D3 布局打通主闭环；Neo4j 路径查找在 Phase 3–4 正式接入（已纳入 M0 范围，见 §9.2 第 17 条）
- `Book.status` 继续保留 `String`，避免过早收死状态枚举
- `visual_config` 用于缓存前端节点坐标，避免每次重新计算布局

### 12.2 AI 解析工程约束

- 大型书籍（50+ 章节）按章节并行调用 AI，各模型受速率限制，需要任务队列
- 分章解析 + 全局上下文注入是基础策略，不允许只做“纯单章孤立解析”
- 每 10 章做一次跨章别名消歧
- 解析结果必须包含置信度与证据锚点，才能支撑审核、回跳与合并建议

### 12.3 静态资源存储约束

- 默认存储方案：`local` Provider，本地文件系统落盘
- 不默认启用本地 S3 兼容服务；只有在明确需要对象存储语义一致性、多机部署或签名 URL 能力时，再进入 M1 评估
- 存储抽象目录固定为 `src/server/providers/storage/`
- 业务模块只能依赖统一存储接口，不直接读写 `storage/` 路径
- 本地文件可通过站内链接访问，推荐统一代理路由（如 `/api/assets/[...key]`），避免暴露物理路径
- 当前最小接口建议：
  - `putObject`
  - `deleteObject`
  - `getObjectUrl`
- 本地与线上都通过环境变量选择 Provider，不在业务代码里写死判断
- 建议环境变量：
  - `STORAGE_PROVIDER=local`
  - `STORAGE_LOCAL_ROOT=./storage`
  - `STORAGE_PUBLIC_BASE_URL=http://localhost:3000`
- 阿里云 OSS 预留环境变量：
  - `OSS_ENDPOINT`
  - `OSS_BUCKET`
  - `OSS_ACCESS_KEY_ID`
  - `OSS_ACCESS_KEY_SECRET`
  - `OSS_REGION`
  - `OSS_PUBLIC_BASE_URL`

### 12.5 大体量图谱渲染方案

采用 **D3 Canvas 渲染 + 语义缩放（Semantic Zoom）+ 视口裁剪** 三层策略：

| 缩放层级 | 显示内容 | 渲染方式 |
| --- | --- | --- |
| 缩小（鸟瞰） | 派系气泡聚类，不显示单个节点 | Canvas 圆形聚合 |
| 中等缩放 | 影响力 Top N（默认 Top 80）节点 + 关系 | Canvas 节点 + 边 |
| 放大（局部） | 当前视口内全量节点，视口外节点不渲染 | Canvas + 视口裁剪 |

- SVG 渲染上限约 200 节点；Canvas 方案可流畅承载 1000+ 节点
- “Top N” 阈值在图谱工具栏可调（50 / 80 / 150 / 全量）
- 节点坐标计算在 Web Worker 中进行
- 首次打开图谱：展示 Top 80 + 动画渐入；用户可点击“加载全部”切换全量

### 12.6 登录鉴权实现约束

- 使用 Next.js Middleware（`middleware.ts`）拦截全站请求，校验 `token` Cookie 中的 JWT
- JWT 签发与校验使用 `jose`（HS256）— 基于 Web Crypto API，同时兼容 Middleware 的 Edge Runtime 和 Node.js Route Handler
- Middleware 通过后注入 `x-auth-role` 请求头，Route Handler 使用 `getAuthContext(request)` 读取
- 管理员操作统一调用 `requireAdmin(auth)` 守卫
- `/admin/*` 权限校验由 Middleware + `src/app/admin/layout.tsx` 双重承担

### 12.7 安全与密码规范

- `rbac.ts` 遗留方案废弃，认证统一收敛到 `src/server/modules/auth/index.ts`
- 密码哈希标准统一采用 `Argon2id`
- 参数基线：`memoryCost=19456`、`timeCost=2`、`parallelism=1`
- 禁止回退 `bcrypt`，也禁止“密码 + secret 再 hash”这类自定义伪方案

### 12.8 AI Key 安全规范（M0）

- `api_key` 必须密文存储；前端与普通 API 仅返回脱敏值
- 加密 / 解密仅在服务端执行，日志与错误信息禁止输出明文 Key
- 预留密钥轮换方案，支持新密钥重加密历史密文
- 数据导出接口默认剔除 `api_key` 原值
- “连通性测试”仅在服务端解密后调用，不回传明文

### 12.9 静态资源安全规范（M0）

- 前端列表接口默认只返回业务所需的资源 URL 与元数据，不暴露服务器物理路径
- 删除书籍时，数据库软删除与对象删除需要有明确先后顺序；M0 可先标记删除，再由异步清理任务清理对象
- 日志、错误信息和调试输出中不得泄露本地绝对路径中的敏感信息
- `sourceFileKey` 是存储主键，`sourceFileUrl` 视为可重建派生值，后续迁移 OSS 时应优先保留 key
- 当切换到 OSS Provider 时，业务 API 契约不变，调用方无感知

### 12.10 M0 非功能基线

- `POST /api/auth/login`：`p95 < 300ms`
- 书库到图谱首屏可交互：`p95 < 2.5s`（前 20 回数据集）
- 前 20 回解析总耗时：`p95 < 10 分钟`（默认模型，单次任务）

### 12.11 数据生命周期

- 业务删除一律先软删除（写入 `deleted_at`）
- 软删除记录保留 30 天支持恢复
- 超过保留期后进入离线物理清理任务，并保留审计日志

### 12.12 现有代码参考

- AI 客户端：`src/server/providers/ai/`
- 建议新增存储 Provider：`src/server/providers/storage/`
- 章节解析 Service：`src/server/modules/analysis/services/ChapterAnalysisService.ts`
- Auth 模块：`src/server/modules/auth/index.ts`
- API 响应规范：`src/server/http/api-response.ts`

### 12.13 前端数据请求规范

> 对应 `.trellis/spec/frontend/react-guidelines.md`（强制规则），前端实现必须遵守。

- **渲染期数据读取**：统一使用 `use()` + Suspense。Server Component 直接调用服务层函数；Client Component 通过 props 接收父级创建的 Promise，在组件内用 `use(promise)` 消费。
- **禁止**在 React 组件渲染期使用 `useEffect + setState` 做首屏数据拉取（竞态、首屏闪烁、与 React 19 模型冲突）。
- **写操作**（表单提交、审核确认、删除等）统一通过 Server Actions 处理，配合 `revalidatePath` 缓存失效。
- **轮询场景**（如解析进度 `GET /api/books/:id/status`）使用 **SWR `refreshInterval`**，状态变为 `COMPLETED` / `ERROR` 后回调返回 `0` 自动停止；SWR 仅限此场景，禁止用于首屏加载。
- **全局 Store（Zustand）** 按需引入，须同时满足：① 多组件共享可写状态 ② props/context 不足 ③ 更新频率高；否则优先本地 `useState` / Context。
- 不引入 TanStack Query 等重量级服务端状态库。

## 十四、关键决策记录

> 本节记录设计阶段的架构决策（ADR-lite），已冻结，不再变更。

### 14.1 决策 1 — 认证采用 `users` 表方案

- **背景**：产品有 `viewer / admin` 权限边界与后台操作审计需求；"env-only" 与 "users 查库" 两套方案不能并存
- **决策**：`users` 表 + `Argon2id` 密码哈希 + JWT（HS256，`jose`），初始管理员通过 seed 写入
- **影响**：需补齐登录 API、seed 初始化与认证测试；后续扩展多管理员或密码轮换成本低

### 14.2 决策 2 — API Key 采用"入库 + 加密存储"方案

- **背景**：模型管理页需要支持多模型 Key 在线维护、启用禁用与连通性测试
- **决策**：`ai_models.api_key` 存储密文，由服务端用 `APP_ENCRYPTION_KEY` 加解密；前端与普通查询接口只返回脱敏值
- **影响**：需补齐加密、轮换、日志脱敏与导出限制约束

### 14.3 决策 3 — 交付策略调整为 M0 全量 + Phase 分步

- **背景**：产品同时覆盖鉴权、解析、图谱、审核、手动管理与视觉打磨，复杂度高；初版采用 M0/M1 双批策略，后经评估决定将全部需求并入一个发布批次
- **决策**：取消 M1 独立批次，所有功能（含图谱高性能渲染、路径查找、视觉沉浸、手动人物管理、`.epub` 支持）均纳入 M0；执行层按 Phase 1–5 顺序分步交付，每阶段独立可测，但产品需求定义不再按批次后置
- **影响**：研发节奏不变（Phase 分步），产品承诺更清晰（无模糊的"M1 后再做"表述）；不牺牲完整产品愿景，按优先级在单一发布周期内落地

### 14.4 决策 4 — 登录接口安全基线

- **背景**：登录接口面向公网，需防止暴力破解；Cookie 需防 CSRF
- **决策**：`POST /api/auth/login` 在应用层实现速率限制（同 IP 5 分钟内失败 10 次后锁定 15 分钟）；CSRF 防护通过 `SameSite=Strict` Cookie + 校验 `Origin` Header 实现，不引入额外 CSRF token 机制
- **影响**：需在 Route Handler 或 Middleware 层实现限流逻辑

### 14.5 决策 5 — 前端数据请求采用 `use()` + Server Actions 模式

- **背景**：前端存在三类数据场景：①页面加载时读取数据 ②用户操作触发写变更 ③客户端轮询解析进度；需统一规范避免 `useEffect` 拉数引发的竞态与多余依赖
- **决策**：
  - 渲染期数据读取：统一 `use()` + Suspense，禁止 `useEffect + setState` 首屏拉数
  - 写操作：Server Actions + `revalidatePath`
  - 客户端轮询（解析进度等）：SWR `refreshInterval`，状态终止时回调返回 `0` 自动停止
  - SWR 使用范围限定为**轮询场景**，禁止用于首屏数据加载（该场景已由 `use()` 覆盖）
  - 不引入 TanStack Query
- **依据**：`use()` 是渲染模型，SWR 是客户端缓存层，两者职责不重叠可共存；SWR `refreshInterval` 比手写 `setInterval` 更健壮（自动去重、条件停止、错误重试）
- **影响**：Client Component 数量减少，loading/error 由 Suspense/Error Boundary 统一承载；SWR 仅作为轮询专用工具，不引入数据所有权混乱
