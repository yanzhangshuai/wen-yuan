# 规范质量标准

> 目标：所有 `.trellis/spec/**/*.md` 都达到“具体、有代码、有原因”。

---

## 必须包含的结构

每份规范文档至少包含以下 4 类内容：

1. **具体规则**：必须/禁止/边界条件，不写空泛原则。
2. **代码示例**：至少 1 组“反例 / 正例”示例。
3. **原因说明**：解释为何采用该规则（稳定性、可维护性、性能、协作成本等）。
4. **落地参考**：至少一个仓库内路径或命令作为落地点。

---

## 推荐模板

~~~markdown
## 必须遵守
- 规则 A
- 规则 B

## 代码示例
反例：
```ts
// ...
```

正例：
```ts
// ...
```

## 原因
- 原因 1
- 原因 2
~~~

---

## 验收标准

- [ ] 文档中存在可执行的规则条目（不是口号）
- [ ] 至少 1 个代码块，且与规则直接相关
- [ ] 存在“原因”段落并解释 trade-off
- [ ] 读者可据此直接改代码，不需要二次猜测
