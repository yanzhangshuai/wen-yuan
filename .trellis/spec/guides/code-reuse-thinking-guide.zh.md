## 问题

**重复代码是不一致性 Bug 的第一来源。**

当你复制粘贴或重写已有逻辑时：
- Bug 修复不会自动传播
- 行为会随着时间逐步分叉
- 代码库会变得更难理解

---

## 编写新代码前

### 第 1 步：先搜索

```bash
# Search for similar function names
grep -r "functionName" .

# Search for similar logic
grep -r "keyword" .
```

### 第 2 步：问自己这些问题

| 问题 | 如果是… |
|----------|-----------|
| 是否已有相似函数？ | 复用或扩展它 |
| 这个模式是否在其他地方使用？ | 遵循现有模式 |
| 这是否可以成为共享工具？ | 在正确位置创建共享实现 |
| 我是否在从另一个文件复制代码？ | **停止**——提取为共享代码 |

---

## 常见重复模式

### 模式 1：复制粘贴函数

**Bad**：把校验函数复制到另一个文件

**Good**：提取到共享工具，在需要处导入

### 模式 2：高度相似组件

**Bad**：新建一个与现有组件 80% 相似的组件

**Good**：通过 props/variants 扩展现有组件

### 模式 3：重复常量

**Bad**：在多个文件定义同一个常量

**Good**：单一事实来源，统一导入

## 何时抽象

**应抽象当**：
- 同样代码出现 3 次以上
- 逻辑复杂到容易出 Bug
- 多个人可能都会用到

**不应抽象当**：
- 仅使用一次
- 只是简单一行
- 抽象本身比重复更复杂

---

## 批量修改后

当你在多个文件做了类似改动：

1. **复查**：是否覆盖了所有实例？
2. **搜索**：运行 grep 查找是否有漏改
3. **评估**：这是否应该被抽象？

---

## 易错点：不同机制产出同一输出

**问题**：当两个不同机制必须产出相同文件集（例如：初始化阶段用递归目录复制，更新阶段用手工 `files.set()`），目录结构变更（重命名、移动、增加子目录）只会通过自动机制传播，手工机制会悄悄漂移。

**症状**：初始化完全正常，但更新时文件路径错误或直接漏文件。

**预防清单**：
- [ ] 迁移目录结构时，搜索所有引用旧结构的代码路径
- [ ] 若一条路径自动推导（glob/copy），另一条手工枚举，手工路径必须同步更新
- [ ] 增加回归测试，比较两种机制的输出是否一致
