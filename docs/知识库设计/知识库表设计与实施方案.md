# 知识库表设计与实施方案

> 文档创建日期：2025-04-11  
> 最后更新：2026-04-10  
> 状态：设计阶段（待实施）

---

## 1. 背景与动机

### 1.1 现状分析

当前系统中有三类"知识"硬编码在代码里：

| 类别 | 文件位置 | 性质 | 示例 |
|------|---------|------|------|
| **书籍类型枚举** | `src/app/admin/books/import/page.tsx` 等多处 | 书籍类型名称的固定枚举列表 | `明清官场 / 武侠 / 宫廷家族...` |
| **NER 调谐规则** | `src/server/modules/analysis/config/pipeline.ts` → `GENRE_PRESETS` | 控制"哪些称谓不该被当作泛化标题" | 武侠书籍类型下 `exemptGenericTitles: ["掌门", "帮主", "盟主"]` |
| **字号/绰号事实** | `src/server/modules/analysis/config/classical-names.ts` → `GENRE_CLASSICAL_NAMES` | 人物多称谓的确定性映射 | `关羽 ↔ 关云长 ↔ 云长 ↔ 关公 ↔ 美髯公` |

**问题：**

1. **书籍类型不可扩展**：增加新书籍类型需改代码多处（import 页、下拉框、GENRE_PRESETS、GENRE_CLASSICAL_NAMES）然后重新部署。
2. **知识不可扩展**：增加一本新书的字号知识，同样需要修改代码、重新部署。
3. **不可审计**：谁加了什么、什么时候加的、有没有经过验证——无法追溯。
4. **不可组合**：无法为同一本书叠加多个知识包（如《三国演义》同时需要"历史演义"通用包 + "三国人物专属"包）。
5. **规模受限**：当前 ~130 条人物映射，无法覆盖数千部古典文学作品。

### 1.2 目标

- 将**书籍类型**从枚举字符串升级为独立数据表，使其可在管理后台动态增删；
- 将**NER 调谐规则**（GENRE_PRESETS）随书籍类型一并存入数据库，实现配置热更新；
- 将**字号/绰号事实知识**迁移到数据库知识包，支持 CRUD、版本管理和审核；
- 管理后台提供完整的**知识库管理工作流**：LLM 批量生成 → 审核 → 激活。

## 2. 概念模型

```
┌────────────┐        ┌────────────────────┐        ┌────────────────────────┐
│  BookType  │ 1──N   │  KnowledgePack     │ 1──N   │   KnowledgeEntry       │
│(书籍类型表) │────────│  (知识包)           │────────│   (知识条目)            │
│            │        │                    │        │                        │
│ id         │        │ id                 │        │ id                     │
│ key        │        │ bookTypeId (FK)    │        │ packId (FK)            │
│ name       │        │ name               │        │ canonicalName          │
│ desc       │   N    │ scope              │        │ aliases (String[])     │
│ preset     │───┐    │ version            │        │ entryType              │
│ Config     │   │    │ isActive           │        │ confidence             │
└────────────┘   │    └────────────────────┘        │ source                 │
                 │              │ N                 │ sourceDetail           │
                 │    ┌─────────┴────────┐          │ reviewStatus           │
                 └────│  Book            │          │ reviewNote             │
                      │  (书籍)           │          │ notes                  │
                      │                  │          │ createdAt              │
                      │ id               │          │ updatedAt              │
                      │ bookTypeId (FK)  │          └────────────────────────┘
                      │ title            │
                      └────────┬─────────┘
                             │ N
                    ┌────────┴────────────┐
                    │  BookKnowledgePack   │  (书籍 ↔ 知识包 多对多)
                    │                     │
                    │ bookId (FK)         │
                    │ packId (FK)         │
                    │ priority            │
                    └─────────────────────┘
```

### 核心设计决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 书籍类型是字符串 vs 数据表 | **数据表** | 支持动态增删；NER 配置（presetConfig）随类型一起存储，无需改代码 |
| NER presetConfig 存哪 | **BookType.presetConfig (JSON)** | 书籍类型 + 词典规则天然绑定；管理后台可在线编辑 |
| 知识按包组织 vs 平铺 | **按包组织** | 支持类型级复用和"书籍专属包"同时存在 |
| 一本书能用多个包 | **是** | 《三国演义》可同时挂载"历史演义通用包" + "三国人物专属包" |
| 别名是数组字段 vs 独立表 | **数组字段** | 单人别名通常 ≤ 15 条，数组简洁；PostgreSQL 支持 `@> ARRAY[]` 查询 |
| 条目审核状态 | **reviewStatus 枚举** | PENDING / VERIFIED / REJECTED，工作流化审核减少误入库风险 |

## 3. 数据库 Schema 设计

### 3.1 完整 Prisma Schema

```prisma
// ============================================================
// 书籍类型表：类型的标准定义 + NER 调谐配置
// ============================================================
/// @db.remark: 古典文学书籍类型定义表，存储 NER 词典规则配置，可在管理后台热更新。
model BookType {
  id           String  @id @default(uuid()) @db.Uuid
  key          String  @unique  // 代码引用键，如"历史演义" — 用于解析流水线中的配置查找
  name         String           // 展示名称（可与 key 不同，如 key="历史演义", name="历史演义小说"）

  /// NER 调谐配置（对应代码中的 BookLexiconConfig）。
  /// 字段格式（JSON 对象）：
  /// {
  ///   "exemptGenericTitles": ["丞相","军师"],       // 豁免的过滤称谓
  ///   "additionalTitlePatterns": ["太守","刺史"],   // 额外职位 pattern
  ///   "additionalPositionPatterns": ["司马"],       // 额外官职 pattern
  ///   "softRelationalSuffixes": ["哥哥","姐姐"]     // 软亲属后缀
  /// }
  presetConfig Json?   @map("preset_config")

  description  String? @db.Text  // 类型说明，如"包含三国、水浒等历史类小说"
  isActive     Boolean @default(true) @map("is_active")
  sortOrder    Int     @default(0) @map("sort_order")  // 前端展示排序

  books          Book[]
  knowledgePacks KnowledgePack[]
  surnameEntries SurnameEntry[]
  extractionRules ExtractionRule[]

  createdAt DateTime @default(now()) @map("created_at") @db.Timestamptz(6)
  updatedAt DateTime @updatedAt @map("updated_at") @db.Timestamptz(6)

  @@index([key], map: "book_types_key_idx")
  @@index([isActive, sortOrder], map: "book_types_active_sort_idx")
  @@map("book_types")
}

// ============================================================
// 知识包：按书籍类型/作品组织的一组人物别名映射
// ============================================================
/// @db.remark: 知识包表，scope=GENRE 为书籍类型通用包，scope=BOOK 为书籍专属包。
model KnowledgePack {
  id          String  @id @default(uuid()) @db.Uuid
  bookTypeId  String? @map("book_type_id") @db.Uuid  // 关联书籍类型（scope=BOOK 时可为空）
  name        String                                  // 如「三国演义核心人物知识库」
  description String? @db.Text
  version     Int     @default(1)         // 版本号，支持增量更新追踪
  isActive    Boolean @default(true) @map("is_active")
  /// GENRE: 书籍类型级通用（所有该类型书籍自动继承）
  /// BOOK:  书籍专属（仅手动挂载的书籍使用）
  scope       String  @default("GENRE")

  bookType    BookType?           @relation(fields: [bookTypeId], references: [id])
  entries     KnowledgeEntry[]
  bookPacks   BookKnowledgePack[]

  createdAt DateTime @default(now()) @map("created_at") @db.Timestamptz(6)
  updatedAt DateTime @updatedAt @map("updated_at") @db.Timestamptz(6)

  @@index([bookTypeId, isActive], map: "knowledge_packs_book_type_active_idx")
  @@index([scope], map: "knowledge_packs_scope_idx")
  @@map("knowledge_packs")
}

// ============================================================
// 知识条目：单个人物的标准名 + 别名列表
// ============================================================
/// @db.remark: 知识条目。reviewStatus 驱动审核工作流（PENDING→VERIFIED or REJECTED）。
model KnowledgeEntry {
  id            String @id @default(uuid()) @db.Uuid
  packId        String @map("pack_id") @db.Uuid

  canonicalName String   @map("canonical_name")     // 标准全名，如「关羽」
  aliases       String[] @default([])                // 所有已知别名
  /// CHARACTER | LOCATION | ORGANIZATION
  entryType     String   @default("CHARACTER") @map("entry_type")
  confidence    Float    @default(1.0)               // 置信度 0.0-1.0（LLM 生成时由模型输出）

  /// 数据来源：
  /// MANUAL         — 管理员手动录入
  /// LLM_GENERATED  — LLM 批量生成，待审核
  /// IMPORTED       — 从 JSON/CSV 导入
  source        String   @default("MANUAL")
  sourceDetail  String?  @map("source_detail") @db.Text  // 如"Qwen-Max 于 2026-04-10 批量生成"

  /// 审核状态（工作流核心字段）：
  /// PENDING  — 新建 / 待人工审核
  /// VERIFIED — 已确认，参与解析流水线
  /// REJECTED — 拒绝，不参与解析
  reviewStatus  String   @default("PENDING") @map("review_status")
  reviewNote    String?  @map("review_note") @db.Text  // 审核备注（拒绝原因等）
  reviewedAt    DateTime? @map("reviewed_at") @db.Timestamptz(6)

  notes         String?  @db.Text  // 额外备注
  pack          KnowledgePack @relation(fields: [packId], references: [id], onDelete: Cascade)

  createdAt DateTime @default(now()) @map("created_at") @db.Timestamptz(6)
  updatedAt DateTime @updatedAt @map("updated_at") @db.Timestamptz(6)

  @@index([packId, reviewStatus], map: "knowledge_entries_pack_review_idx")
  @@index([canonicalName], map: "knowledge_entries_canonical_name_idx")
  @@map("knowledge_entries")
}

// ============================================================
// 书籍 ↔ 知识包 多对多关联
// ============================================================
model BookKnowledgePack {
  id       String @id @default(uuid()) @db.Uuid
  bookId   String @map("book_id") @db.Uuid
  packId   String @map("pack_id") @db.Uuid
  priority Int    @default(0)  // 多包合并时的优先级（数值越大越优先）

  book  Book          @relation(fields: [bookId], references: [id], onDelete: Cascade)
  pack  KnowledgePack @relation(fields: [packId], references: [id], onDelete: Cascade)

  createdAt DateTime @default(now()) @map("created_at") @db.Timestamptz(6)

  @@unique([bookId, packId], map: "book_knowledge_pack_unique")
  @@index([bookId], map: "book_knowledge_packs_book_id_idx")
  @@map("book_knowledge_packs")
}
```

**Book 模型需要更新：**
```prisma
// 将 genre String? 替换为：
bookTypeId String? @map("book_type_id") @db.Uuid
bookType   BookType? @relation(fields: [bookTypeId], references: [id])
bookKnowledgePacks BookKnowledgePack[]
```

> **迁移说明**：`Book.genre` (String) 作为过渡字段保留，迁移脚本将其值与 `BookType.key` 对照后填入 `bookTypeId`，再删除旧字段。

### 3.2 表关系总结

```
BookType ─1:N─ Book ─1:N─ BookKnowledgePack ─N:1─ KnowledgePack ─1:N─ KnowledgeEntry
    │                                                    │
    └───────────────── 1:N ───────────────────────────── ┘
      (BookType 与 KnowledgePack 的书籍类型级关联)
```

**知识包查找逻辑（按优先级）：**
1. 书籍手动挂载的包（`BookKnowledgePack`，按 priority 降序）
2. 若无手动挂载 → 按 `Book.bookTypeId` 自动匹配 `scope=GENRE` 的通用包
3. 若无任何包 → `aliasLookup` 为空 Map，LLM 在分析时依赖上下文理解（仍可工作）

## 4. 与解析流水线的集成

### 4.1 书籍类型配置加载（替换 GENRE_PRESETS）

```typescript
// 迁移前（pipeline.ts 硬编码）
const config = GENRE_PRESETS[bookTypeKey] ?? {};

// 迁移后（从数据库加载，任务启动时预加载一次）
async function loadBookTypeConfig(bookTypeKey: string, prisma: PrismaClient): Promise<BookLexiconConfig> {
  const bookType = await prisma.bookType.findUnique({
    where: { key: bookTypeKey, isActive: true }
  });
  if (!bookType?.presetConfig) return {};
  return bookType.presetConfig as BookLexiconConfig;
}
```

### 4.2 知识库别名表加载（替换 buildAliasLookup）

```typescript
// 迁移后（从数据库加载）
async function buildAliasLookup(
  bookId: string,
  bookTypeKey: string | null | undefined,
  prisma: PrismaClient
): Promise<Map<string, string>> {
  // Step 1: 查找手动挂载的知识包
  let packIds: string[] = [];

  const bookPacks = await prisma.bookKnowledgePack.findMany({
    where: { bookId },
    orderBy: { priority: 'desc' },
    select: { packId: true }
  });

  if (bookPacks.length > 0) {
    packIds = bookPacks.map(bp => bp.packId);
  } else if (bookTypeKey) {
    // Step 2: 按书籍类型自动匹配
    const typePacks = await prisma.knowledgePack.findMany({
      where: { bookType: { key: bookTypeKey }, isActive: true, scope: 'GENRE' },
      select: { id: true }
    });
    packIds = typePacks.map(p => p.id);
  }

  // Step 3: 无知识包 → 返回空 Map（LLM 仍可基于上下文工作）
  if (packIds.length === 0) return new Map();

  // Step 3: 加载所有已验证条目 (reviewStatus=VERIFIED only)
  const entries = await prisma.knowledgeEntry.findMany({
    where: { packId: { in: packIds }, reviewStatus: 'VERIFIED' },
    select: { canonicalName: true, aliases: true, confidence: true }
  });

  // Step 4: 展平为 alias → canonicalName 映射（高置信度条目优先）
  const lookup = new Map<string, string>();
  const sortedEntries = entries.sort((a, b) => b.confidence - a.confidence);
  for (const entry of sortedEntries) {
    lookup.set(entry.canonicalName.toLowerCase(), entry.canonicalName);
    for (const alias of entry.aliases) {
      if (alias.trim()) lookup.set(alias.trim().toLowerCase(), entry.canonicalName);
    }
  }
  return lookup;
}
```

### 4.3 任务级缓存（避免循环查库）

Pass 2 (`resolveGlobalEntities`) 启动前，一次性加载：
- `bookTypeConfig` → 传给 `resolveBookLexiconConfig()`
- `aliasLookup` → 传给 `buildCandidateGroups()`

同一任务内不重复查询。

## 6. 管理后台 UI 设计

### 6.1 书籍类型管理页 `/admin/knowledge-base/book-types`

**功能：**
- 书籍类型列表（表格）：key / name / 关联知识包数 / 关联书籍数 / 启用状态
- **创建/编辑弹窗**：
  - key（唯一键）/ name（展示名）/ sortOrder / description
  - NER 配置编辑器（JSON 编辑器，可折叠）：显示 `presetConfig` 结构，带字段说明
- 软删除（有关联书籍时禁止删除，提示先迁移书籍）

**知识包管理入口**：
- 每行有"管理知识包"按钮 → 跳转至别名知识包管理页并预筛该类型

> **📌 推荐为超著名著作创建书籍专属包（scope=BOOK）**

## 7. 数据填充策略

### 7.1 种子数据迁移（JSON 导入方式）

> **推荐 JSON 文件导入，而非硬编码种子脚本**
> 
> 优点：数据与代码分离 → 可单独更新数据 → 可视化审查 → 可通过 API 直接重新导入无需重跑迁移。

**初始化数据文件**：`data/knowledge-base/book-types.init.json`

该 JSON 包含全部 7 个书籍类型（含 `presetConfig`）及 5 个类型级知识包（含 ~131 条已验证条目）。

**导入脚本**（需在 Phase 1 新建）：

```typescript
// scripts/init-knowledge-base.ts
// 用法：npx tsx scripts/init-knowledge-base.ts
//       npx tsx scripts/init-knowledge-base.ts --file data/knowledge-base/book-types.init.json
import fs from 'fs';
import path from 'path';
import { PrismaClient } from '@prisma/client';

const prisma = new PrismaClient();

interface InitData {
  version: string;
  bookTypes: Array<{
    key: string; name: string; description?: string;
    sortOrder: number; isActive: boolean; presetConfig: object;
    knowledgePacks: Array<{
      name: string; scope: string; description?: string;
      entries: Array<{ canonicalName: string; aliases: string[] }>;
    }>;
  }>;
}

async function run() {
  const filePath = process.argv[3] ?? 'data/knowledge-base/book-types.init.json';
  const raw = fs.readFileSync(path.resolve(filePath), 'utf-8');
  const data: InitData = JSON.parse(raw);

  for (const bt of data.bookTypes) {
    const bookType = await prisma.bookType.upsert({
      where: { key: bt.key },
      create: { key: bt.key, name: bt.name, description: bt.description,
                sortOrder: bt.sortOrder, isActive: bt.isActive, presetConfig: bt.presetConfig },
      update: { presetConfig: bt.presetConfig, name: bt.name }
    });

    for (const pack of bt.knowledgePacks) {
      const existing = await prisma.knowledgePack.findFirst({
        where: { bookTypeId: bookType.id, name: pack.name }
      });
      if (existing) { console.log(`  skip existing pack: ${pack.name}`); continue; }

      const kp = await prisma.knowledgePack.create({
        data: { bookTypeId: bookType.id, name: pack.name,
                scope: pack.scope, description: pack.description }
      });
      await prisma.knowledgeEntry.createMany({
        data: pack.entries.map(e => ({
          packId: kp.id, canonicalName: e.canonicalName, aliases: e.aliases,
          source: 'IMPORTED', sourceDetail: `book-types.init.json v${data.version}`,
          reviewStatus: 'VERIFIED', confidence: 1.0
        }))
      });
      console.log(`  created pack: ${pack.name} (${pack.entries.length} entries)`);
    }
    console.log(`✓ BookType: ${bt.key}`);
  }
  await prisma.$disconnect();
}

run().catch(e => { console.error(e); process.exit(1); });
```

**当前实现说明**：仓库中的 `prisma/seed.ts` 已串联知识库初始化，标准 `prisma db seed` 会一起写入管理员、AI 模型、书籍类型、种子知识包、姓氏/称谓/规则/提示词模板基线。

脚本仍可独立使用：
- `prisma/seed.ts`：标准初始化入口，适合新环境一次性完成基础数据准备
- `scripts/init-knowledge-base.ts`：单独重放 JSON 种子（书籍类型 + 5 个种子知识包）
- `scripts/init-knowledge-phase6.ts`：单独重放姓氏/称谓/NER 规则/提示词模板基线

## 12. 姓氏知识库（SurnameLibrary）

### 12.1 背景与问题

`lexicon.ts` 中 `CHINESE_SURNAME_LIST`（约 150 个姓）在 `PersonaResolver` 的"姓+称号"匹配中至关重要（如识别"范举人"→范进）。当前痛点：
- 罕见姓、书籍专有姓无法扩展（需改代码部署）
- **复姓优先级**（欧阳/司马/诸葛）须先于单姓匹配，顺序硬编码
- 无法标注某个姓是否来自特定书籍类型

### 12.2 Prisma Schema

```prisma
// ============================================================
// 姓氏条目：支持单姓/复姓，复姓优先匹配
// ============================================================
/// @db.remark: 姓氏库。extractSurname() 运行时从此表加载，优先匹配复姓。
model SurnameEntry {
  id          String  @id @default(uuid()) @db.Uuid
  surname     String  @unique      // 姓氏本体，如"欧阳"/"关"
  isCompound  Boolean @default(false) @map("is_compound") // 是否复姓（复姓在匹配中优先于单姓）
  priority    Int     @default(0)  // 覆盖默认优先级（越大越高，复姓建议设 10）
  description String? @db.Text    // 备注，如"出现于《三国演义》"

  /// 关联书籍类型（可选，表示此姓在某类型作品中多见，nil 表示通用）
  bookTypeId  String? @map("book_type_id") @db.Uuid
  bookType    BookType? @relation(fields: [bookTypeId], references: [id])

  isActive    Boolean @default(true) @map("is_active")
  source      String  @default("MANUAL")   // MANUAL / LLM_SUGGESTED / IMPORTED

  createdAt DateTime @default(now()) @map("created_at") @db.Timestamptz(6)
  updatedAt DateTime @updatedAt @map("updated_at") @db.Timestamptz(6)

  @@index([isCompound, priority], map: "surname_compound_priority_idx")
  @@index([isActive], map: "surname_active_idx")
  @@map("surname_entries")
}
```

> **Genre 模型需追加关联**：
> ```prisma
> surnameEntries SurnameEntry[]
> extractionRules ExtractionRule[]
> ```

### 12.3 与解析流水线的集成

```typescript
// src/server/modules/knowledge/loadSurnameList.ts
export async function loadSurnameList(prisma: PrismaClient): Promise<{
  compounds: string[];  // 复姓（优先匹配），按 priority 降序
  singles  : string[];  // 单姓
}> {
  const entries = await prisma.surnameEntry.findMany({
    where: { isActive: true },
    orderBy: [{ priority: 'desc' }, { surname: 'asc' }],
    select: { surname: true, isCompound: true }
  });
  return {
    compounds: entries.filter(e => e.isCompound).map(e => e.surname),
    singles  : entries.filter(e => !e.isCompound).map(e => e.surname),
  };
}

// extractSurname() 调用侧改为：
// 1. 任务启动时调用 loadSurnameList() 一次，缓存结果
// 2. 优先检查 compounds（2字），再检查 singles（1字）
```

### 13.1 背景与问题

**问题**：书籍级的豁免/追加配置同样硬编码于 `BookLexiconConfig`（`pipeline.ts`）。

### 13.2 Prisma Schema

```prisma
// ============================================================
// 泛化称谓条目
// ============================================================
/// @db.remark: 泛化称谓库。tier=SAFETY 绝不个体化，tier=DEFAULT 可被书籍类型/书籍覆盖。
model GenericTitleEntry {
  id    String @id @default(uuid()) @db.Uuid
  title String @unique   // 称谓本体，如"老爷"

  /// SAFETY  — 绝对泛称，任何情况下不允许个体化
  /// DEFAULT — 默认泛称，可被书籍类型 exemptGenericTitles 豁免
  tier  String @default("DEFAULT")  // SAFETY | DEFAULT

  /// 书籍类型级别覆盖（该称谓在哪些书籍类型下不视为泛称）
  /// JSON 数组，存储 Genre.key 列表，如 ["武侠","历史演义"]

  description String? @db.Text
  isActive    Boolean @default(true) @map("is_active")
  source      String  @default("MANUAL")

  createdAt DateTime @default(now()) @map("created_at") @db.Timestamptz(6)
  updatedAt DateTime @updatedAt @map("updated_at") @db.Timestamptz(6)

  @@index([tier, isActive], map: "generic_titles_tier_idx")
  @@map("generic_title_entries")
}
```

> **注意**：`GenericTitleEntry.exemptInGenres` 替代了 `GENRE_PRESETS.exemptGenericTitles`；

  updatedAt DateTime @updatedAt @map("updated_at") @db.Timestamptz(6)

// ============================================================

  @@index([templateId, createdAt(sort: Desc)], map: "prompt_versions_template_created_idx")

## 16. 知识变更审计日志（KnowledgeAuditLog）

### 16.1 背景与问题

知识库中任何一条变更（添加别名、修改规则、激活新提示词版本）都可能影响解析结果。当前完全没有变更追踪，无法排查"上周改了什么导致这本书准确率下降"。

### 16.2 Prisma Schema

```prisma
// ============================================================
// 知识变更审计日志
// ============================================================
/// @db.remark: 所有知识库对象（KnowledgeEntry/PromptTemplate/SurnameEntry 等）的变更日志。
model KnowledgeAuditLog {
  id         String @id @default(uuid()) @db.Uuid

  /// 被操作对象类型：
  /// KNOWLEDGE_ENTRY | KNOWLEDGE_PACK | GENRE | SURNAME | GENERIC_TITLE |
  /// PROMPT_TEMPLATE | EXTRACTION_RULE
  objectType String @map("object_type")
  objectId   String @map("object_id")   // 被操作对象的 UUID
  objectName String @map("object_name") // 冗余名称，防止对象被删除后无法追溯

  /// 操作类型：
  /// CREATE | UPDATE | DELETE | VERIFY | REJECT | ACTIVATE | DEACTIVATE | IMPORT | GENERATE
  action     String

  /// 变更前/后快照（JSON 格式，可 null = 首次创建 or 最终删除）
  before     Json?  @db.JsonB
  after      Json?  @db.JsonB

  /// 操作人（userId，nil=系统自动/种子迁移）
  operatorId String? @map("operator_id")
  operatorNote String? @db.Text @map("operator_note")

  /// 关联的书籍（若此次变更由书籍维度触发）
  relatedBookId String? @map("related_book_id") @db.Uuid

  createdAt DateTime @default(now()) @map("created_at") @db.Timestamptz(6)

  @@index([objectType, objectId], map: "audit_log_object_idx")
  @@index([createdAt(sort: Desc)], map: "audit_log_created_idx")
  @@index([operatorId, createdAt(sort: Desc)], map: "audit_log_operator_idx")
  @@map("knowledge_audit_logs")
}
```

### 16.3 自动记录机制

在以下操作的服务层函数中统一插入审计日志（**无需前端主动调用**）：

## 附录 D: 知识库成本模型与使用策略

> **核心结论：每本书不需要独立的知识库。绝大多数书只需设置正确的书籍类型即可立即开始分析。**

### D.1 三层成本模型

知识库采用"按需精细化"原则，分三层逐步加精度：

```
Level 1 (零成本，所有书自动享有)
  └── BookType.presetConfig
      • 定义 NER 调谐参数（豁免称谓、额外职位词等）
      • 无人工维护成本，迁移种子数据后自动生效

Level 2 (低成本，同类书共享一个知识包)
  └── scope=GENRE 的通用知识包
      • 举例：100 本"历史演义"书统一使用《三国演义》知识库中的人物别名
      • 一次构建，所有同类书受益
      • 每种类型通常只需维护 1 个通用包

Level 3 (按需，仅超著名作品值得投入)
  └── scope=BOOK 的书籍专属包
      • 举例：额外为《三国演义》单独建一份精确到"曹孟德↔曹操↔丞相↔魏王"的包
      • 适用场景：书中人物别名极多、极复杂，且用户会反复分析该书
      • 推荐范围：~10 部超著名古典作品（三国、水浒、西游、红楼梦等）
```

### D.2 新书导入的默认路径（零额外操作）

```
新书导入
  ↓ 选择书籍类型（如"历史演义"）
  ↓ 触发解析
  ↓ buildAliasLookup(bookId, "历史演义", prisma)
     → 查找 bookId 挂载的专属包  → (无)
     → 查找 bookTypeKey="历史演义" 的 scope=GENRE 包 → 命中通用包
     → 返回 "历史演义" 的别名 Map
  ↓ LLM 分析时带入通用别名上下文
```

**结论：99% 的新书无需任何知识库维护，直接继承类型通用包即可获得合理的分析质量。**

### D.3 何时需要书籍专属包？

| 场景 | 是否需要专属包 | 说明 |
|------|--------------|------|
| 普通历史小说 | ❌ 不需要 | 类型通用包已覆盖常见别名 |
| 《三国演义》 | ✅ 推荐 | 200+ 人物，字号/绰号错综复杂 |
| 《红楼梦》 | ✅ 推荐 | 人物别名系统独特（宝玉/二爷/怡红公子） |
| 《水浒传》 | ✅ 推荐 | 绰号系统需精确映射（黑旋风↔李逵） |
| 普通网文 | ❌ 不需要 | 人物名较简单，LLM 上下文理解已足够 |
| 当代文学 | ❌ 不需要 | 无字号/绰号系统，不依赖别名知识库 |

### D.4 是否存在更优方案的评估

| 备选方案 | 优点 | 缺点 | 结论 |
|---------|------|------|------|
| **按需 LLM 生成**（分析时动态生成别名） | 无需预构建 | 每次分析都消耗额外 token；生成质量不稳定 | ❌ 成本更高 |

### D.5 运营维护成本预估

- **Phase 1（种子迁移）**：将现有 5 个知识包（约 131 条已验证条目）自动迁移入库，**零人工成本**。
