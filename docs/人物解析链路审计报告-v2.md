> **审计范围**：项目导入书籍 → 角色解析 → 知识库全链路  
> **审计对象**：书籍 `402f2282-1a27-4aa4-bf5b-726f0bb1b28a`（儒林外史 · 吴敬梓）  
> **审计日期**：2025-07  
> **审计方法**：静态代码审计 + 数据库结果分析 + 架构推演

## 一、执行摘要

本次审计对「文渊」项目的人物解析链路进行了端到端的严格审查，范围覆盖：两种解析架构（Sequential / TwoPass）的设计与实现、核心服务代码（7 个关键模块、~5000+ 行）、知识库集成流程、以及《儒林外史》实际解析结果的数据库验证。

### 关键发现

| 级别 | 发现 | 影响 |
|------|------|------|
| **P0-严重** | 实际运行使用 Sequential 架构——缺乏全局实体去重，直接导致同一人物被拆分为多个 Persona | 7 组同名重复、43 组别名冲突 |
| **P0-严重** | 泛称/关系词被错误创建为独立人物（管家→尤胡子、邻居→张国重、姑老爷→鲍廷玺），别名严重污染 | 13+ 人物受影响 |
| **P1-高** | 别名注册表（AliasMapping）在本次解析中未产生任何记录（0 条），跨章节实体消歧完全失效 | 全书范围 |
| **P1-高** | 不同人物被错误合并（向鼎→董知县、匡太公→郑老爹、杜倩→杜慎卿），PersonaResolver 相似度算法存在系统性缺陷 | 5+ 人物数据错误 |
| **P1-高** | 性别字段全部为 null（227/227），性别提取功能完全缺失 | 全量人物 |
| **P2-中** | 知识库（classical-names.ts）包含儒林外史核心人物映射，但本次解析实际为 Sequential 架构——知识库映射仅在 TwoPass Pass 2 使用，Sequential 完全绕过 | 设计意图未生效 |
| **P2-中** | dynamicTitleResolutionEnabled=false、llmTitleArbitrationEnabled=false，两个关键特性开关关闭 | 灰区称谓未仲裁 |

### 结论判定

**当前链路在 Sequential 架构下的人物解析准确率估计约 65-70%**，主要错误来源为：实体分裂（~15%）、错误合并（~5%）、泛称误建（~8%）、别名污染（~5%）。切换至 TwoPass 架构并启用知识库可预期提升至 80-85%，但仍需修复 PersonaResolver 的核心缺陷才能突破 90%。

知识库→解析链路的集成从代码层面是完备的，但**仅对 TwoPass 架构生效**。Sequential 架构下知识库完全不参与实体消歧，这是本次解析结果质量低下的根本原因之一。

## 二、当前整套流程还原

### 2.1 全链路数据流

```
┌─────────────┐     ┌──────────────┐     ┌───────────────────┐     ┌──────────────┐
│  导入书籍    │────▶│  创建分析任务  │────▶│  选择架构 & 执行   │────▶│  后处理阶段   │
│  (Book+      │     │  (AnalysisJob │     │  (Pipeline)       │     │  (Post-proc)  │
│   Chapters)  │     │   PENDING)    │     │                   │     │               │
└─────────────┘     └──────────────┘     └───────────────────┘     └──────────────┘
                                                   │                        │
                                                   ▼                        ▼
                                          ┌─────────────────┐     ┌──────────────────┐
                                          │ Persona/Profile  │     │ OrphanMarking    │
                                          │ Mention/Bio/Rel  │     │ TitleResolution   │
                                          │ AliasMapping     │     │ GrayZoneArbitrat. │
                                          └─────────────────┘     │ BookValidation    │
                                                                  └──────────────────┘
```

### 2.2 逐阶段详细流程（以本次 Sequential 架构为例）

**阶段 0：任务初始化**（`runAnalysisJob.ts:runAnalysisJobById`）
- 认领任务（乐观锁 `claimJobWithOptimisticLock`）
- 加载章节列表（按 `orderIndex` 排序）
- 读取模型策略（3 级：GLOBAL → BOOK → JOB）
- 构建有效词典（`buildEffectiveLexicon`：默认泛称 + 书籍类型预设 + 豁免列表）
- 创建 `AliasRegistryService` 实例

**阶段 1：Roster Discovery**（`ChapterAnalysisService.ts:analyzeChapter` Phase 1）
- 对每章调用 `buildRosterDiscoveryPrompt`，让 LLM 识别出现人物
- 输出：`surfaceForm / entityId / isNew / generic / aliasType`
- 长章节自动分片（>20000 字分为 15000 字 chunks）
- Roster 结果与已有 Profile 合并，生成本章有效人物列表

**阶段 2：Chunk Extraction**（Phase 2）
- 章节内容按 10000 字分片（800 字重叠）
- 每个 chunk 调用 `buildChapterAnalysisPrompt`，提取 biographies / mentions / relationships
- 注入本章过滤后的 profiles（roster 命中 + 兜底层 top-15）

**阶段 3：Merge & Persist**（Phase 2 后半段）
- `mergeChunkResultsForAnalysis`：跨 chunk 去重（按 personaName+rawText+paraIndex / personaName+event / source+target+type）
- `persistResult`：在事务中 → 删除旧 draft 数据 → 逐人物调用 PersonaResolver → 创建 Mention/BiographyRecord/Relationship
- 每个新识别的别名通过 `aliasRegistry.registerAlias` 注册

**阶段 4：增量称号溯源**（每 5 个成功章节触发一次）
- `resolvePersonaTitles`：将 TITLE_ONLY 类型人物提交 LLM 做标题→真名解析

**阶段 5：后处理**（全书完成后）
- `markOrphanPersonas`：提及 <2 次的人物 confidence 降至 0.4
- `resolvePersonaTitles`：最终一轮称号溯源
- `runGrayZoneArbitration`：灰区称谓 AI 仲裁（**当前 `llmTitleArbitrationEnabled=false`，未执行**）
- `runBookValidation`：全书校验（抽样 6 条原文 × 280 字）

### 2.3 Sequential vs TwoPass 架构差异

| 维度 | Sequential | TwoPass |
|------|-----------|---------|
| **全局实体视图** | ❌ 无——每章独立发现人物 | ✅ Pass 1 全章节独立提取 → Pass 2 全局合并 |
| **知识库利用** | ❌ 不使用 classical-names 知识库 | ✅ Pass 2 通过 `buildAliasLookup(genre)` 做规则预合并 |
| **实体去重** | 仅靠 PersonaResolver 逐章匹配 | Pass 2 Union-Find + LLM 消歧 + PersonaResolver |
| **进度分配** | 0-100% 全给章节循环 | 0-35% Pass1, 36-40% Pass2, 40-100% Pass3 |
| **风险** | 同名异人拆分、泛称失控 | Pass 2 LLM 可能误合并不同人物 |

## 三、两种角色解析架构审计

### 3.1 Sequential 架构审计

**代码入口**：`src/server/modules/analysis/pipelines/sequential/SequentialPipeline.ts`

**设计意图**：章节按序处理，利用前序章节积累的 Profile 做增量式实体对齐。

**代码级问题**：

| # | 问题 | 代码位置 | 严重度 | 证据 |
|---|------|---------|--------|------|
| S-1 | **无全局实体视图**——每章 roster discovery 独立运行，前序章节的人物只通过 DB 中已有 Profile 传递。如果第 N 章创建了重复 persona，后续章节会基于这个错误继续积累。 | `SequentialPipeline.ts:runSequentialChapterLoop` → 调用 `analyzer.analyzeChapter` | P0 | DB 结果显示周进、严贡生等有 2 条同名记录 |
| S-2 | **roster 到 chunk 的人物过滤存在信息损失**——chunk analysis 只注入 roster 命中 + top-15 兜底层 persona，其余长尾人物在 chunk 阶段完全不可见，导致 LLM 无法正确归并到已有人物。 | `ChapterAnalysisService.ts` 的 roster → chunk profile 过滤逻辑 | P1 | 长尾人物在后续章节被重新创建 |
| S-3 | 增量称号溯源间隔 `incrementalResolveInterval=5` 在儒林外史（56 回）中意味着约 11 次称号解析，频率偏低，可能导致中间章节大量 TITLE_ONLY 人物无法及时归并。 | `pipeline.ts:incrementalResolveInterval=5` | P2 | — |

**架构评价**：Sequential 架构适合**角色固定、章节独立性强**的小说（如短篇集），但对于儒林外史这类**角色流动性极高**（上半部人物完全退场、下半部全新人物登场）的讽刺小说，Sequential 的逐章积累模式会导致：
1. 早期人物在后期"复活"时无法识别（名字已不在 roster 高频区）
2. 泛称（如"老爷""先生"）在不同上下文指向不同人物时，PersonaResolver 无法跨章节消歧

### 3.2 TwoPass 架构审计

**代码入口**：`src/server/modules/analysis/pipelines/twopass/TwoPassPipeline.ts`

**设计意图**：先全局独立提取 → 全局实体合并 → 再做带全局人物图谱的精细分析。

**Pass 1（独立提取）**：
```
对每章调用 extractChapterEntities()
→ buildIndependentExtractionPrompt()（不注入已有 profiles）
→ 输出 name/aliases/description/category
```

**Pass 2（全局实体解析）**（`GlobalEntityResolver.ts`）：
```
collectGlobalDictionary()     → 建立全局名字→章节映射
buildCandidateGroups()        → Union-Find 分组：
  1. 知识库别名查找（buildAliasLookup）
  2. 编辑距离 ≤ 1
  3. 同姓 + 别名重叠
resolveCandidateGroupsWithLLM() → 批量 15 组送 LLM 判定 merge/no-merge
resolveGlobalEntities()       → 创建 Persona + 构建 globalPersonaMap
```

**代码级问题**：

| # | 问题 | 代码位置 | 严重度 | 证据 |
|---|------|---------|--------|------|
| T-1 | **Pass 2 创建的 Persona 不与 DB 已有 Persona 去重**——如果此前已有一轮解析结果，Pass 2 会创建全新的 Persona 集合，导致旧数据残留。 | `GlobalEntityResolver.ts:resolveGlobalEntities` → `prisma.persona.create` | P1 | 代码审计发现无 `upsert` 或已有 persona 查询 |
| T-2 | **Union-Find 编辑距离阈值（≤1）过于激进**——中文人名通常 2-3 字，编辑距离 1 意味着任意一个字不同就可能被合并。例如"杜倩"和"杜慎"编辑距离=1，但完全是不同人。 | `GlobalEntityResolver.ts:buildCandidateGroups` | P1 | "杜倩"→"杜慎卿"的错误合并可由此路径产生 |
| T-3 | **知识库 classical-names.ts 中儒林外史映射不完整**——仅覆盖 ~15 个核心人物，未覆盖如景兰江/景本蕙、牛浦郎/牛布衣等关键别名对。 | `classical-names.ts:RULIN_NAMES` | P2 | 景兰江被拆分为 2 条记录 |
| T-4 | **LLM 批量消歧（15 组/batch）缺乏上下文**——仅提供 name+aliases 列表，不提供出现章节或原文片段，LLM 判断依据不足。 | `GlobalEntityResolver.ts:resolveCandidateGroupsWithLLM` | P2 | — |

**架构评价**：TwoPass 在理论上能解决 Sequential 的核心缺陷（无全局实体视图），但当前实现存在上述问题。对于儒林外史这类角色规模大（原著约 200+ 有名人物）、称谓复杂的小说，TwoPass 是更合适的选择，但需要修复 T-1 至 T-4 的问题。

### 3.3 知识库流程可靠性评审

**知识库数据流向**：
```
KnowledgePack → KnowledgeEntry (canonical_name + aliases[])
     ↓ (buildAliasLookupFromDb)
alias lookup Map<string, string>
     ↓ (仅 TwoPass Pass 2 使用)
GlobalEntityResolver.buildCandidateGroups() 规则预合并
```

**问题**：
1. **单向流——知识库只读，解析结果不回写**：解析产生的新人物/新别名不会自动沉淀到知识库，下次解析同一书籍时仍从零开始。
2. **Sequential 架构完全不使用知识库别名查找**：`loadTwoPassRuntimeContext()` 仅在 TwoPass 路径调用，Sequential 路径的 `aliasLookup` 为空 Map。
3. **知识库挂载依赖 BookType**：书籍 `402f2282` 的 `bookTypeId=null`，即使有知识库 Pack 也无法通过类型继承自动关联。

#### 类型 C：泛称/关系词被创建为独立人物（幻觉实体）

| 错误人物名 | 被映射到 | 别名污染情况 | 根因 |
|-----------|---------|-------------|------|
| **管家** | 尤胡子 | 13 个别名：邵管家、严管家、万管家… | "管家"在 `DEFAULT_GENERIC_TITLES` 中，但 roster 阶段仍创建了 Persona |
| **邻居** | 张国重 | 含王羽秋、施牛的人等完全不相关人名 | "邻居"不在泛称列表中，被错误识别为实体 |
| **姑老爷** | 鲍廷玺 | 别名含蘧太守（完全不同人物） | 关系称谓被创建为实体后，后续章节的不同"姑老爷"全部合并 |
| **老爹** | — | 被 8 个不同 Persona 共享 | "老爹"在 `HARD_BLOCK_SUFFIXES` 中但作为独立词未被拦截 |
| **大爷** | — | 被 3 个 Persona 共享 | 同上 |

**估计影响**：~15-20 个 Persona 为幻觉实体或受泛称污染。

#### 类型 D：别名分配错误（跨人物污染）

`老爹`（8 个 Persona 共享）、`大爷`（3 个）、`王老爹`（3 个）、`先生`（2 个）等别名被多个不相关人物同时持有。这意味着 `registerAlias` 在创建 Mention 时，同一个 surfaceForm 被注册到了多个不同的 Persona，**别名注册的唯一性约束缺失**。

从代码看，`AliasRegistryService.registerAlias` 的去重逻辑是 `bookId + alias + chapterStart + chapterEnd`，但同一章节内同一称谓可以被不同 Persona 各注册一次（只要 confidence 不同），导致 1:N 的别名映射。

### 5.2 逐模块审计

#### 5.2.1 `PersonaResolver.ts`（P0 关键路径）

**功能**：将一个 surfaceForm 解析到已有 Persona 或创建新 Persona。

**解析流程**：
```
resolve(name)
  → 空检查 / 过短(<2) / 过长(>10) → 拒绝
  → safety 泛称过滤（SAFETY_GENERIC_TITLES）
  → 动态泛称个性化判定
  → rosterMap 查找（含名字相似度校验）
  → aliasRegistry 查找
  → 相似度评分（loadCandidates → multiSignalScore → scorePair）
  → 创建新 Persona 或匹配已有
```

**发现的问题**：

| # | 问题 | 代码证据 | 影响 |
|---|------|---------|------|
| PR-1 | **`scorePair` 的 substring 匹配存在误报**：当 name A 包含 name B 时给高分，但"匡太公"包含"匡"→ 可能匹配到"匡超人"。substring 匹配缺乏长度比例检查。 | `scorePair` 函数中 substring 判断分支 | P1：错误合并 |

## 六、准确率问题全景图
