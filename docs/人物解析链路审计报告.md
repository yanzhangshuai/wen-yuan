## 一、人物解析整体架构梳理

### 1.1 系统定位

文渊（Wen Yuan）是一个面向中国古典文学的 AI 人物知识图谱平台。核心功能是从百万字非结构化文言文中自动提取人物实体、传记事件、人物关系，并构建可视化知识图谱。

### 1.2 六阶段 Pipeline 架构

```
POST /api/books/:id/analyze
  │
  ▼
┌──────────────────────────────────────────────────────────────┐
│ runAnalysisJobById()                                         │
│                                                              │
│  FOR each chapter:                                           │
│   ┌─────────────────────────────────────────────┐            │
│   │ Phase 1: Roster Discovery (全章)            │            │
│   │   → buildRosterDiscoveryPrompt()            │            │
│   │   → AI: 识别所有人物称谓表面形式             │            │
│   │   → 输出: rosterMap, titleOnlyNames,         │            │
│   │          pendingRosterAliasMappings           │            │
│   ├─────────────────────────────────────────────┤            │
│   │ Phase 2: Content Chunking                    │            │
│   │   → splitContentIntoChunks(content, 2000)    │            │
│   ├─────────────────────────────────────────────┤            │
│   │ Phase 3: Chunk Analysis (3并发)              │            │
│   │   → buildChapterAnalysisPrompt() × N chunks  │            │
│   │   → AI: 提取 mentions/biographies/relations   │            │
│   │   → mergeChunkResults(): flatMap 合并         │            │
│   ├─────────────────────────────────────────────┤            │
│   │ Phase 4: Persist (事务)                      │            │
│   │   → PersonaResolver.resolve() 去重/新建       │            │
│   │   → AliasRegistryService.registerAlias()      │            │
│   │   → DB: Mention, BiographyRecord, Relationship│            │
│   ├─────────────────────────────────────────────┤            │
│   │ Phase 5: Async Chapter Validation (非阻塞)    │            │
│   │   → ValidationAgentService.validateChapter()  │            │
│   └─────────────────────────────────────────────┘            │
│                                                              │
│  EVERY 5 chapters:                                           │
│   → resolvePersonaTitles() (增量标题解析)                      │
│                                                              │
│  POST-BOOK (FULL_BOOK only):                                  │
│   → markOrphanPersonas() (< 2 mentions → confidence 0.4)      │
│   → resolvePersonaTitles() (终态解析)                          │
│   → validateBookResult() + applyAutoFixes()                   │
└──────────────────────────────────────────────────────────────┘
```

## 三、当前实现核心流程说明

### 3.1 入口与触发

- 用户通过 `POST /api/books/:id/analyze` 触发
- 支持三种作用域：`FULL_BOOK` / `CHAPTER_RANGE` / `CHAPTER_LIST`
- 创建 `AnalysisJob`(QUEUED) → 异步 fire-and-forget 调用 `runAnalysisJobById()`

### 3.2 文本切分

- **章节级**：由 `chapterSplit.ts` 在书籍上传时完成（正则匹配 "第X章/回"、楔子等）
- **AI 分析级**：`splitContentIntoChunks(content, 2000)`
  - 按 `\n+` 分段
  - 段落 > 2000字：硬切
  - 否则逐段累积至 2000 字上限
  - 【已确认】不存在跨 chunk 上下文传递

### 3.3 人物提取机制

**Phase 1 (Roster Discovery)**：全章文本 → LLM → `ChapterRosterEntry[]`
- 识别所有称谓表面形式（surfaceForm）
- 分类：GENERIC / KNOWN / NEW / TITLE_ONLY
- 输出 aliasType (TITLE/POSITION/KINSHIP/NICKNAME/COURTESY_NAME)
- 携带 suggestedRealName + aliasConfidence

**Phase 3 (Chunk Analysis)**：每 2000 字 chunk → LLM → biographies + mentions + relationships
- 输入携带 Phase 1 已知人物档案上下文
- 以 `[1] 范进 | 别名: 范老爷, 范举人 | 小传: ...` 格式注入

### 3.4 去重与归并

**PersonaResolver.resolve()** — 五步决策：
1. 空名/短名 → hallucinated
2. GENERIC_TITLES 精确匹配 → hallucinated
3. rosterMap 快速路径 → resolved (0.97)
4. AliasRegistry 查询 → resolved (≥0.7)
5. multiSignalScore 候选评分 → ≥0.72 resolved，否则新建

**multiSignalScore 四信号算法**：
- 精确匹配 → 1.0
- 双向子串 → 0.60 + 0.37 × (短/长) ≈ 0.67~0.97
- Levenshtein (≥6字) → 1 - dist/maxLen
- 字符 Jaccard (<6字) → intersection/union

### 3.5 审查机制

**ValidationAgentService**：
- 章节级验证 (`validateChapterResult`)：非阻塞
- 全书级验证 (`validateBookResult`)：FULL_BOOK 完成后触发
- 自动修复 (`applyAutoFixes`)：confidence ≥ 0.9 的 MERGE/ADD_ALIAS/UPDATE_NAME

### 3.6 数据落库

- 所有写入在 `$transaction` 中完成（30秒超时）
- 状态：`ProcessingStatus.DRAFT`
- 去重键：
  - Mention: `[chapterId, personaId, rawText, paraIndex, summary]`
  - Biography: `[chapterId, personaId, category, event, title, location, virtualYear]`
  - Relationship: `[chapterId, sourceId, targetId, type]`

## 四、准确度分析

### 4.1 核心机制

当前实现是**两阶段 LLM 提取 + 多信号评分归并 + AI 审查**的架构。准确度主要依赖：
1. Roster Discovery prompt 的质量
2. Chunk Analysis prompt 的质量
3. PersonaResolver 的评分阈值
4. Validation Agent 的审查准确度

### 4.2 准确度风险点（按严重度排序）

#### 🔴 P0：scorePair() 子串检测后缺失 return 0（**已确认 Bug**）

**文件**：`PersonaResolver.ts`, multiSignalScore → scorePair(), lines 415-426
**问题**：当子串匹配命中 RELATIONAL_SUFFIXES（如 "蘧公孙" vs "蘧公孙父亲"），应当返回 0 阻止归并，但代码缺少 `return 0`，会穿透到 Levenshtein/Jaccard 继续评分。
**影响**：亲属关系人物被错误归并（"蘧公孙" 和 "蘧公孙父亲" 可能合并为同一人）。
**等级**：低成本高收益修复

#### 🔴 P0：分片后上下文完全断裂（**已确认**）

**文件**：`ChapterAnalysisService.ts`, mergeChunkResults() line 563-569
**问题**：
- `splitContentIntoChunks()` 产生的各 chunk 之间无上下文共享
- `mergeChunkResults()` 只是 flatMap，无跨 chunk 去重或归并
- Chunk Analysis prompt 明确要求"不要跨段推测"
**影响**：
- 同一人物在不同 chunk 可能被识别为不同名字变体
- 跨 chunk 的人物关系完全丢失
- 分片边界处的人物可能漏识别
**缓解**：Phase 1 Roster Discovery 在全章级运行，提供了全局人物名册作为上下文。这是当前架构的关键设计——通过 roster 预解析缓解分片断裂。
**残余风险**：如果 Roster 漏识别了某人物，后续 chunk 更不可能发现。【推断】

#### 🟡 P1：Roster Discovery prompt 中 `isTitleOnly` 逻辑错误（**已确认**）

**文件**：`prompts.ts`, buildRosterDiscoveryPrompt, Rule 9
**问题**：prompt 要求"如果是称号/帝号且原文无法得知真实姓名 → isNew: true + isTitleOnly: true"。但称号可能指代已知人物（如"吴王"可能是已建档的角色），不应强制 `isNew: true`。
**影响**：已知人物的称号形式被错误创建为新实体，产生重复 Persona。
**等级**：低成本高收益修复（修改 prompt 规则即可）

#### 🟡 P1：泛化称谓列表在 prompt 和代码中不一致（**已确认**）

**文件**：
- `prompts.ts` buildRosterDiscoveryPrompt: "老爷、夫人、众人、那人、先生、他"
- `prompts.ts` buildChapterAnalysisPrompt: "老爷、夫人、太太、小姐、公子、掌柜、那人、众人、旁人"
- `PersonaResolver.ts` GENERIC_TITLES: 40+ 项
**问题**：三处列表覆盖范围不同。"先生"在 Roster prompt 中是泛化，在 Analysis prompt 中不是；"太太、小姐、公子"在 Roster prompt 中未提及。
**影响**：LLM 在不同阶段对同一称谓的处理方式不一致。
**等级**：低成本高收益修复

#### 🟡 P1：Profile upsert 空 update 导致 localName 不更新（**已确认 Bug**）

**文件**：`PersonaResolver.ts`, Step 3, lines 284-297
**问题**：`profile.upsert({ update: {} })` — 如果 profile 已存在，localName 不会被更新为新提取的名字。
**影响**：同一人物的不同地方性名称（如"范进"vs"范老爷"）只保留最先创建的版本。
**等级**：低成本高收益修复

#### 🟡 P1：Alias Registry 置信度门槛过低（0.7）（**已确认**）

**文件**：`PersonaResolver.ts`, Step 2.5, line 254
**问题**：AliasRegistry 在 0.7 confidence 就接受映射，而 multiSignalScore 需要 0.72。两个阈值接近但不同，且 0.7 偏低容易导致错误归并。
**影响**：低置信度别名映射导致误归并。
**等级**：低成本中收益修复

#### 🟡 P1：Chunk Analysis prompt 中英文混用（**已确认**）

**文件**：`prompts.ts`, buildChapterAnalysisPrompt, line 183
**问题**：当无已知人物时，fallback 文本为英文 "No existing entities found in this book yet." 而整个 prompt 是中文。
**影响**：语言切换可能导致 LLM 输出不稳定。
**等级**：低成本低风险修复

#### 🟢 P2：multiSignalScore 对 2-5 字短名使用 Jaccard 而非 Levenshtein（**已确认**）

**文件**：`PersonaResolver.ts`, scorePair(), lines 429-445
**问题**：Levenshtein 仅对 ≥6 字名字生效。中文人名通常 2-4 字，全部走 Jaccard。
**影响**：Jaccard 对"范进"vs"范举"给出 1/3≈0.33（正确不归并），但对"娄三公子"vs"娄四公子"给出 3/5=0.6（正确不归并，但接近阈值）。整体安全，但缺乏对单字差异的精细区分。
**等级**：中成本中收益

#### 🟢 P2：Validation Agent 无法看到原文（全书级）（**已确认**）

**文件**：`ValidationAgentService.ts`, validateBookResult()
**问题**：Book-level validation 只接收聚合数据（人物列表+关系统计），没有原文。无法验证关系是否有原文支撑。
**影响**：全书验证只能做结构性检查，无法做内容真实性验证。
**等级**：高成本但必要（长期）

### 4.3 准确度优化建议

| 优先级 | 建议 | 成本/收益 |
|--------|------|-----------|
| **P0** | 修复 scorePair() 子串匹配后缺失 return 0 | 低成本高收益 |
| **P0** | 在 chunk analysis prompt 中注入 Phase 1 roster 全量结果作为上下文 | 低成本高收益（已部分实现，需确认 roster 是否完整传入） |
| **P1** | 统一三处泛化称谓列表（prompt + GENERIC_TITLES），使用单一源定义 | 低成本高收益 |
| **P1** | 修复 isTitleOnly 规则，允许称号映射到已知人物 | 低成本高收益 |
| **P1** | 修复 profile upsert 空 update | 低成本高收益 |
| **P1** | 提高 AliasRegistry 置信度门槛至 0.75 或与 multiSignalScore 对齐 | 低成本中收益 |
| **P2** | 引入 chunk 重叠（overlap 200-300字）以缓解分片断裂 | 中成本高收益 |
| **P2** | mergeChunkResults 增加跨 chunk 去重逻辑 | 中成本高收益 |
| **P3** | 全书验证引入抽样原文片段 | 高成本但必要 |

## 五、性能分析

### 5.1 耗时分布（推断）

| 阶段 | 估计占比 | 说明 |
|------|---------|------|
| Phase 1: Roster Discovery | ~20% | 全章文本单次 AI 调用 |
| Phase 3: Chunk Analysis | ~55% | 多 chunk × AI 调用（3并发） |
| Phase 4: Persist | ~5% | 数据库事务写入 |
| Phase 5: Chapter Validation | ~10% | AI 验证（非阻塞但消耗资源） |
| Title Resolution | ~5% | 每5章一次 |
| Book Validation | ~5% | 全书完成后 |

### 5.2 性能问题

#### 重复解析问题（已确认）

- **Phase 1 + Phase 3 重复**：Phase 1 对全章文本做 roster discovery，Phase 3 对同一文本分 chunk 再做 entity extraction。实际是两次完整文本处理。
- **影响**：全章文本被 AI 处理 2 次（roster + N 个 chunk analysis）

#### 章节串行处理（已确认）

- `runAnalysisJob.ts` 中章节处理完全串行（for 循环）
- chunk 有 3 并发，但章节间无并行
- 【推断】对 50+ 章的长篇小说，总耗时 = Σ(章节处理时间)，可能数小时

#### 每5章 Title Resolution 带来额外 AI 调用（已确认）

- `INCREMENTAL_RESOLVE_INTERVAL = 5`
- 50 章书 = 10 次增量 + 1 次终态 = 11 次 title resolution AI 调用
- 如果 TITLE_ONLY 人物少，大多数调用可能返回空结果

### 5.3 性能优化建议

| 优先级 | 建议 | 说明 |
|--------|------|------|
| **P1** | 考虑章节间有限并行（如 2-3 章并行） | 中成本高收益。需要处理 PersonaCache 和 AliasRegistry 的并发安全 |
| **P2** | Title Resolution 增量调用改为"仅当新增 TITLE_ONLY 人物时触发" | 低成本中收益 |
| **P2** | Chapter Validation 设为可选或仅对高风险章节执行 | 低成本中收益（减少 AI 调用） |
| **P3** | 引入 chunk 分析结果缓存，支持局部重试不重跑整章 | 高成本中收益 |

## 六、Token 消耗分析

### 6.1 Token 消耗分布（推断）

**每章消耗模型**（假设章节 8000 字、4 个 chunk）：

| 阶段 | 输入 tokens (估) | 输出 tokens (估) | 占比 |
|------|-----------------|-----------------|------|
| Roster Discovery | ~10K (prompt ~1.5K + 全章 ~8K + 已知人物 ~0.5K) | ~2K | ~30% |
| Chunk Analysis × 4 | ~4 × (prompt ~1.8K + chunk ~2.5K + profiles ~0.5K) ≈ 19.2K | ~4 × 1.5K = 6K | ~50% |
| Chapter Validation | ~5K (prompt + entities + mentions + relationships) | ~1.5K | ~15% |
| Title Resolution (每5章) | ~2K | ~0.5K | ~5% |
| **单章总计** | ~36K input | ~10K output | |

**50章书全书**：≈ 50 × 46K + 11 × 2.5K + 1 × 8K ≈ **2.34M tokens**

### 6.2 Token 浪费点

#### 🔴 Roster Discovery 重复传入全章文本

**问题**：Phase 1 把完整章节文本传入做 roster discovery，Phase 3 又把同样的文本切片后再次传入。
**影响**：章节文本被计费 2 次。
**可优化**：【推断】如果 Roster Discovery 足够准确，可以考虑在 roster 中直接标注段落位置，减少 chunk analysis 的 prompt 模板冗余。但这会牺牲 chunk 级分析的独立性。
**判断**：**Roster 全文输入是必要的**（需要全局视角），但 prompt 模板可以精简。

#### 🟡 Chunk Analysis prompt 模板重复

**问题**：每个 chunk 都携带相同的 prompt 模板（~1800 tokens）和相同的 profiles 上下文（~500 tokens）。4 个 chunk = 重复 3 次 ≈ 6900 tokens 浪费。
**可优化**：无法避免（每次 AI 调用需完整 prompt），但可以精简 prompt 模板、压缩 profiles 格式。

#### 🟡 Chapter Validation 重复传入已有上下文

**问题**：Validation prompt 再次列出所有已知人物 + 新建人物 + mentions + relationships。这些数据刚从 chunk analysis 中提取。
**可优化**：仅对高风险章节（如新建人物多、hallucination 多）执行验证。
**判断**：对准确度有帮助但 token 成本高。

#### 🟢 已知人物档案注入格式冗长

**问题**：profiles 格式为 `[N] 范进 | 别名: 范老爷, 范举人 | 小传: ...`。后期人物多时，profiles 上下文可能占 2000+ tokens。
**可优化**：对 chunk analysis 只注入该 chunk 中出现的人物（基于 roster 结果），而非全书人物。
**判断**：**中成本高收益** — 可能显著减少后期章节的 token 消耗。

### 6.3 Token 优化建议

| 优先级 | 建议 | 预估节省 |
|--------|------|---------|
| **P1** | Chunk Analysis 只注入 roster 中本 chunk 涉及的人物 profiles | ~20-30% profiles token |
| **P1** | Chapter Validation 改为条件触发（高风险章节才执行） | ~15% 总 token |
| **P2** | 精简 prompt 模板：去除重复说明、合并相似规则 | ~10% 模板 token |
| **P2** | profiles 格式压缩：去除 `小传` 字段或限制 50 字 | ~10% profiles token |
| **P3** | 探索 batch API（如 Gemini batch）减少 prompt 重复开销 | 取决于 API 支持 |

## 七、审查机制分析

### 7.1 当前审查架构

```
Chapter Analysis → Chapter Validation（非阻塞）→ 报告落库
  ↓ (所有章节完成后)
Book Validation → Auto-Fix（confidence ≥ 0.9）→ 报告落库
```

### 7.2 审查内容

| 检查维度 | 章节级 | 全书级 |
|---------|--------|--------|
| ALIAS_AS_NEW_PERSONA | ✅ | ✅ |
| WRONG_MERGE | ✅ | ✅ |
| MISSING_NAME_MAPPING | ✅ | ✅ |
| INVALID_RELATIONSHIP | ✅ | ✅ |
| SAME_NAME_DIFFERENT_PERSON | ✅ | ✅ |
| DUPLICATE_PERSONA | ✅ | ✅ |
| LOW_CONFIDENCE_ENTITY | ✅ | ✅ |
| ORPHAN_MENTION | ✅ | ✅ |

### 7.3 审查机制问题

#### 🔴 Chapter Validation 非阻塞但无法影响后续章节（**已确认**）

**问题**：Chapter Validation 是 fire-and-forget，验证结果不会反馈到后续章节的解析中。即使发现了错误（如别名被误创建为新人物），后续章节仍然基于错误数据继续处理。
**影响**：错误在全书范围内累积和放大。
**建议**：高严重度问题（ERROR 级、confidence ≥ 0.9）应阻塞后续章节并自动修复后继续。

#### 🟡 Auto-Fix 阈值单一且偏严（**已确认**）

**文件**：`ValidationAgentService.ts`, line 522
**问题**：所有自动修复统一使用 confidence ≥ 0.9。MERGE 操作风险较高合理要求 0.9，但 ADD_ALIAS 操作风险低，0.8 就可以自动执行。
**建议**：按 action 类型设置分层阈值：MERGE ≥ 0.9, ADD_ALIAS ≥ 0.8, UPDATE_NAME ≥ 0.85。

#### 🟡 Validation 置信度过滤硬编码（**已确认**）

**文件**：`ValidationAgentService.ts`, lines 367, 485
**问题**：`confidence >= 0.6` 硬编码过滤。低于 0.6 的问题被静默丢弃。
**影响**：潜在问题丢失，无法追溯。
**建议**：低置信度问题降级为 INFO 级保留，不直接丢弃。

#### 🟡 Book Validation 无原文上下文（**已确认**）

**问题**：全书验证只有聚合数据，无法验证关系证据的真实性。
**影响**：只能做结构性检查（重复人物、自环关系），无法做内容真实性验证。

#### 🟢 审查成本 vs 收益分析

**每章增加 ~5K tokens 输入 + ~1.5K 输出 = ~6.5K tokens per chapter。**
50 章 = ~325K tokens 仅用于章节验证。
加上全书验证 ~10K tokens。
**总审查 token ≈ 335K，占总消耗 ~14%。**

**收益评估**：
- Chapter Validation 因非阻塞，实际修复效果有限（仅落报告）
- Auto-Fix 在全书完成后执行，但此时错误已扩散
- 【推断】当前架构下，审查的**实际质量提升有限，主要价值在问题发现而非自动修复**

### 7.4 审查优化建议

| 优先级 | 建议 | 成本/收益 |
|--------|------|-----------|
| **P1** | Chapter Validation 中高置信度错误应阻塞并自动修复后继续 | 中成本高收益 |
| **P1** | 按 action 类型设置分层 auto-fix 阈值 | 低成本中收益 |
| **P2** | 低置信度验证结果降级保留而非丢弃 | 低成本低收益 |
| **P2** | 仅对"新建人物 ≥ 3 或 hallucination ≥ 2"的章节执行验证 | 中成本高收益（节省 token） |
| **P3** | Book Validation 引入抽样原文 | 高成本中收益 |

## 八、代码漏洞与工程风险分析

### 8.1 已确认 Bug

#### Bug 1：scorePair() 子串匹配 RELATIONAL_SUFFIXES 后缺失 return 0

**文件**：`PersonaResolver.ts`, lines 415-426
**问题**：当 `a.includes(b)` 且尾部在 RELATIONAL_SUFFIXES 中时，应返回 0 阻止归并，但缺少 `return 0`，穿透到 Levenshtein/Jaccard。
**触发条件**：任何名字 + 亲属后缀的匹配场景
**修复**：在 `if (!RELATIONAL_SUFFIXES.has(tail))` 的 else 分支加 `return 0`

#### Bug 2：Profile upsert 空 update

**文件**：`PersonaResolver.ts`, Step 3, lines 284-297
**问题**：`profile.upsert({ update: {} })` 不更新 localName
**修复**：`update: { localName: input.extractedName }`

#### Bug 3：ValidationAgentService 连续 UPDATE_NAME 导致别名链

**文件**：`ValidationAgentService.ts`, lines 596-601
**问题**：多个 UPDATE_NAME fix 连续应用时，第二次 update 会把第一次的 newName 推入 aliases。
**修复**：跟踪原始名称，或每个 validation report 只允许一次 UPDATE_NAME

#### Bug 4：ValidationReport issues JSON 双重序列化

**文件**：`ValidationAgentService.ts`, line 518
**问题**：`parseValidationResponse(JSON.stringify(report.issues))` — issues 已是 JSON 字段，再 stringify 再 parse 效率低且可能出错。
**修复**：直接使用 `report.issues as ValidationIssue[]` 并做类型校验

### 8.2 边界问题

#### 空值风险

- **PersonaResolver Step 1 vs Step 2 归一化不一致**：Step 1 用 raw `extractedName.trim()`，后续步骤用 `normalizeName()`。对包含空格或特殊字符的名字行为不同。
- **BookPersonaCache aliasIndex 碰撞**：两个不同人物拥有相同别名时，后注册的覆盖前者，无警告。
- **Validation sanitizeIssuesByPersona**：过滤后 `affectedPersonaIds` 可能为空数组，留下"无主"问题。

#### 并发风险

- **BookPersonaCache 无锁**：多个 chunk 并行处理时（AI_CONCURRENCY=3），同时调用 `addPersona()` 和 `addAlias()` 可能产生竞态（JS 单线程内安全，但异步操作间的 DB 写入可能导致缓存与 DB 不一致）。
- **AliasRegistry 内存缓存**：`upsertInMemoryCache()` 在 `registerAlias()` 的 CONFIRMED 分支更新缓存，但 DB 写入和缓存更新非原子。

#### 数据一致性

- **PersonaResolver 创建 Persona 后 Profile 创建失败**：非事务内调用可能导致孤立 Persona。
- **mergeChunkResults flatMap 无去重**：跨 chunk 的同一关系可能重复写入（去重仅在 persist 阶段通过 unique key 实现）。

### 8.3 测试覆盖不足

**未覆盖的关键路径**：
1. 事务回滚场景（Persona 创建成功但 Profile 失败）
2. 并发章节分析（当前测试都是单章）
3. Chunk 部分失败（3 个 chunk 中 2 个成功 1 个失败）
4. 超大章节处理（>100K 字）
5. Unicode 归一化（简繁体、异体字）
6. Prompt 注入（恶意 personaName 中包含 JSON 特殊字符）
7. repairJson 对深层嵌套 JSON 的处理

### 8.4 安全风险

- **API Key 解密**：在 `createRuntimeAiClient()` 中解密 API Key，【推断】日志中未泄露（需确认 error log 是否包含敏感信息）
- **Prompt 注入**：personaName 直接拼入 prompt，理论上可被利用（低风险，因为数据来自用户上传的书籍文本）

### 8.5 配置风险

- **MAX_CHUNK_LENGTH = 2000**：对包含大量对话的章节可能切分过细，增加 AI 调用次数
- **AI_CONCURRENCY = 3**：对免费 tier API（如 DeepSeek）可能触发 rate limit
- **Transaction timeout = 30s**：对大章节（50+ 人物、200+ mentions）可能不够

## 九、Prompt 专项分析

### 9.1 Prompt 关键问题

#### 问题 1：Roster Discovery 和 Chunk Analysis 对泛化称谓的指导不一致

Roster prompt 列举的泛化称谓示例少于 Chunk Analysis prompt，且两者与 GENERIC_TITLES 代码列表不一致。LLM 在两个阶段可能对同一称谓做出矛盾判断。

#### 问题 2：ChapterAnalysis Rule 10 的循环依赖

prompt 要求"若原文使用官衔或亲属称谓指代已知人物，personaName 必须填写标准名"。但这依赖 Phase 1 已经正确识别了所有别名映射。如果 Phase 1 遗漏，Phase 3 无法自行解决。

#### 问题 3：TitleResolution 置信度区间有空白

prompt 定义：0.85+（有据可查）、0.5-0.7（推断）、<0.5（不确定）。0.7-0.85 区间无定义，LLM 输出可能在此区间产生不一致行为。

#### 问题 4：Evidence 长度限制不一致

- ChapterAnalysis: evidence ≤ 120字
- TitleResolution: historicalNote ≤ 30字
- 无统一标准

### 9.2 Prompt 优化建议

| 建议 | 成本 |
|------|------|
| 统一泛化称谓列表，从代码 GENERIC_TITLES 动态生成 prompt 片段 | 低 |
| 修复 isTitleOnly 规则，区分"已知人物的称号"和"未知人物的称号" | 低 |
| 补全 TitleResolution 置信度 0.7-0.85 区间说明 | 低 |
| 统一 evidence 长度限制 | 低 |
| 修复英文 fallback 为中文 | 低 |

## 十二、结论

当前"书籍人物解析"系统的架构设计是**合理且相对完整**的。六阶段 pipeline + roster discovery + 多信号评分 + AI 审查的组合在同类系统中属于较高水平。

**最关键的准确度问题**是：
1. `scorePair()` 子串匹配的 bug（已确认，需立即修复）
2. prompt 中泛化称谓列表不一致（已确认，需尽快统一）
3. isTitleOnly 强制 isNew 的逻辑错误（已确认，需修复 prompt）
4. 分片断裂的缓解手段可进一步加强（chunk overlap）

**最大的效率瓶颈**是：
1. 章节串行处理（可引入有限并行）
2. Chapter Validation 无条件执行（可改为条件触发）

**最值得的 token 优化**是：
1. Chunk Analysis 只注入相关人物 profiles
2. Validation 条件触发

建议按上述优先级表执行，优先处理 🔴 级别的 6 项修改，预计 1-2 天即可完成，对准确度提升最为显著。
