# OpenSpec + Trellis 流程使用手册（统一口径）

> 原则：OpenSpec 负责规范资产；Trellis 负责执行门禁。

## 1. 默认触发词

- `ff-fast+n: <需求>`：速度策略 + 新分支
- `ff-fast=c: <需求>`：速度策略 + 当前分支
- `ff-full+n: <需求>`：严格策略 + 新分支
- `ff-full=c: <需求>`：严格策略 + 当前分支

## 2. 核心命令

```bash
# 速度策略
bash .trellis/scripts/flow_feature_init_openspec.sh --strategy fast "<需求>" [task-dir] [--stack "<技术栈>"] [--req-doc <需求文档>] [--stack-doc <技术栈文档>]

# 严格策略
bash .trellis/scripts/flow_feature_init_openspec.sh --strategy strict "<需求>" [task-dir] [--stack "<技术栈>"] [--req-doc <需求文档>] [--stack-doc <技术栈文档>]

# 新分支快捷初始化（自动创建 task + 分支并设为当前任务）
bash .trellis/scripts/flow_feature_create.sh "<需求>" [short-name]

# 升级（fast -> strict）
bash .trellis/scripts/flow_feature_upgrade_docs_openspec.sh [task-dir]

# 门禁
python3 ./.trellis/scripts/task.py flow-confirm [task-dir]
python3 ./.trellis/scripts/task.py flow-confirm --approve --notes "执行" [task-dir]
python3 ./.trellis/scripts/task.py flow-guard --verify [task-dir]
```

## 3. OpenSpec 目录约定

每个 flow-feature 任务对应：

```text
openspec/changes/<change-id>/
├── proposal.md
├── design.md
├── tasks.md
└── spec-delta.md
```

## 4. 执行步骤（严格策略）

1. `/trellis:start`
2. 运行 `flow_feature_init_openspec.sh --strategy strict`
3. 补齐 `proposal/design/tasks/spec-delta`
4. 运行 `flow-confirm`
5. 用户确认后实现
6. 验证 success/failure/boundary
7. 运行 `flow-guard --verify`
8. `$finish-work` 后询问是否执行 `$record-session`

## 5. 执行步骤（速度策略）

1. `/trellis:start`
2. 运行 `flow_feature_init_openspec.sh --strategy fast`
3. 补齐最小必要 OpenSpec 文档
4. 运行 `flow-confirm --compact`
5. 用户确认后实现
6. 最少验证：success/failure/boundary
7. `flow-guard --verify`

## 6. 何时必须升级到严格策略

满足任一项：

- 需求歧义
- 跨层契约变化（API/Action/DB/ENV）
- 风险升高（高影响、难回滚、未知依赖）

## 7. 规范与职责

- OpenSpec 规范资产：`openspec/changes/*`、`openspec/specs/*`
- Trellis 执行门禁：`.trellis/scripts/*`、`.trellis/tasks/*`
