# Trellis + Spec-Kit 流程使用手册（中文完整版）

> 目标：提供一份可直接执行的流程手册。  
> 原则：默认速度优先；风险上升时切到严格策略。  
> 范围：包含预定义触发词与手动命令两种使用方式。

---

## 1. 流程总览（先看这个）

- `ff-fast+n: <需求>`：速度策略 + 新分支（推荐默认）
- `ff-fast=c: <需求>`：速度策略 + 当前分支
- `ff-full+n: <需求>`：严格策略 + 新分支
- `ff-full=c: <需求>`：严格策略 + 当前分支
- `ff+n / ff=c / ff`：通用 flow-feature（未指定 fast/full 时由助手判断）
- `fl+n / fl=c / fl`：flow-lite（小改动快走）
- `fb+n / fb=c / fb`：flow-bug（问题修复）

分支后缀规则：
- `+n`：新建分支
- `=c`：当前分支
- 无后缀：先确认分支选择

---

## 2. 什么时候用哪个流程

### A) 速度策略（Speed Strategy）
适用：需求明确、低风险、可快速回滚。  
检查要求：只做必要检查（success / failure / boundary）。

### B) 严格策略（Strict Strategy）
适用：需求不清晰、跨层契约变化、高风险。  
检查要求：完整文档、完整门禁（含 flow-guard）。

### C) flow-lite
适用：单点小改动（低风险、短时间可完成）。

### D) flow-bug
适用：已知问题修复，重点在复现、修复与回归。

---

## 3. 使用预定义触发词（推荐）

### 3.1 速度策略

```text
ff-fast+n: <需求>
ff-fast=c: <需求>
```

### 3.2 严格策略

```text
ff-full+n: <需求>
ff-full=c: <需求>
```

### 3.3 通用触发

```text
ff+n: <需求>
ff=c: <需求>
ff: <需求>
```

### 3.4 轻量与修复

```text
fl+n: <需求>
fl=c: <需求>
fb+n: <需求>
fb=c: <需求>
```

## 4. 各流程详细步骤（怎么使用）

### 4.1 速度策略（ff-fast）

1) 启动：
```text
/trellis:start
```

2) 初始化骨架：
```bash
bash .trellis/scripts/flow_feature_fast_init.sh "<需求>" [task-dir] [--stack "<技术栈>"] [--req-doc <需求文档>] [--stack-doc <技术栈文档>]
```
说明：默认两次输入（先需求、再技术栈）；兼容一条输入 `"<需求> || <技术栈>"`。

3) 任务确认（精简输出）：
```bash
python3 ./.trellis/scripts/task.py flow-confirm --compact --preview 8 [task-dir]
```

4) 你确认后执行：
```bash
python3 ./.trellis/scripts/task.py flow-confirm --approve --notes "执行" [task-dir]
/speckit.implement
```

5) 必要检查（必须有）：
- success：主路径可跑通
- failure：至少 1 条失败路径，错误可读且稳定
- boundary：至少 1 条边界验证

6) 收尾：
```text
$finish-work
```

---

### 4.2 严格策略（ff-full）

1) 启动：
```text
/trellis:start
```

2) 初始化完整骨架（推荐）：
```bash
bash .trellis/scripts/flow_feature_full_init.sh "<需求>" [task-dir] [--stack "<技术栈>"] [--req-doc <需求文档>] [--stack-doc <技术栈文档>]
```
说明：严格策略也建议默认两次输入；双段一条仅作兼容输入。

3) 执行完整 Spec-Kit：
```text
/speckit.specify
/speckit.clarify
/speckit.plan
/speckit.tasks
```

4) 任务确认：
```bash
python3 ./.trellis/scripts/task.py flow-confirm [task-dir]
python3 ./.trellis/scripts/task.py flow-confirm --approve --notes "执行" [task-dir]
```

5) 实施与门禁：
```text
/speckit.implement
python3 ./.trellis/scripts/task.py flow-guard [task-dir]
$finish-work
```

---

### 4.3 flow-lite（fl）

1) 启动：`/trellis:start`  
2) 实施：`/speckit.implement`  
3) 快速验证：至少验证主路径  
4) 收尾：`$finish-work`

---

### 4.4 flow-bug（fb）

1) 启动：`/trellis:start`  
2) 澄清问题：建议 `specify -> clarify -> tasks`  
3) 任务确认：建议 `flow-confirm`  
4) 修复：`/speckit.implement`  
5) 回归：至少覆盖问题复现路径 + 修复路径  
6) 收尾：`$finish-work`

---

## 5. 中途修改任务（怎么改）

你可以在 `flow-confirm` 后输入：

- 自然语言：`修改：把 T3 拆成两步，并补失败路径验证`
- 结构化：`+` 新增 / `-` 删除 / `~` 改写 / `>` 重排 / `!` reopen

结构化命令：

```bash
python3 ./.trellis/scripts/task.py flow-edit-tasks "<ops>" [task-dir]
```

---

## 6. 升级规则（速度策略 -> 严格策略）

命中任一项必须升级：

1. 需求或验收标准有歧义
2. 跨层契约变化（API/Action/DB 签名、payload、env）
3. 风险升高（高影响、回滚困难、依赖不确定）

升级命令：

```bash
bash .trellis/scripts/flow_feature_upgrade_docs.sh [task-dir]
/speckit.specify
/speckit.clarify
/speckit.plan
/speckit.tasks
python3 ./.trellis/scripts/task.py flow-confirm [task-dir]
/speckit.implement
```

---

## 7. 手动模式（不使用预定义触发词）

你可以直接手动执行：

- 手动速度策略：`fast_init -> flow-confirm --compact -> implement -> 必要检查 -> finish`
- 手动严格策略：`full_init -> specify/clarify/plan/tasks -> flow-confirm -> implement -> flow-guard -> finish`

---

## 8. 常用命令速查

```bash
# 分支与初始化
bash .trellis/scripts/flow_feature_create.sh "<需求>" [short-name]
bash .trellis/scripts/flow_feature_fast_init.sh "<需求>" [task-dir] [--stack "<技术栈>"] [--req-doc <需求文档>] [--stack-doc <技术栈文档>]
bash .trellis/scripts/flow_feature_full_init.sh "<需求>" [task-dir] [--stack "<技术栈>"] [--req-doc <需求文档>] [--stack-doc <技术栈文档>]
bash .trellis/scripts/flow_feature_upgrade_docs.sh [task-dir]

# 任务门禁
python3 ./.trellis/scripts/task.py flow-confirm [task-dir]
python3 ./.trellis/scripts/task.py flow-confirm --compact --preview 8 [task-dir]
python3 ./.trellis/scripts/task.py flow-confirm --approve --notes "执行" [task-dir]
python3 ./.trellis/scripts/task.py flow-guard [task-dir]

# 任务编辑
python3 ./.trellis/scripts/task.py flow-edit-tasks "<ops>" [task-dir]
```
