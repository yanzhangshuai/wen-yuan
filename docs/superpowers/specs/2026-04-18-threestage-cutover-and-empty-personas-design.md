# threestage 全书解析收口与空角色问题设计

## 1. 结论

本任务采用方案 1：

1. `threestage` 继续作为全书解析唯一主架构。
2. 取消旧架构兼容在产品层面的长期保留，不再把 `Profile`/旧链路作为主读取真相。
3. 本次不修改按章分析 `sequential`，因为当前用户反馈按章分析准确率反而更稳定，该路径保持独立。

这不是“再做一套并行方案”，而是对当前全书解析链路做单线收口。

## 2. 现状与根因

### 2.1 当前并不存在“threestage 没跑出来任何角色”

针对书籍 `61c34092-5295-4438-af28-46b5b0e9d558`：

- 最新作业 `1d45726b-b91a-4738-a613-231b74e21653`
- `architecture = threestage`
- `scope = FULL_BOOK`
- `status = SUCCEEDED`

数据库事实不是空：

- `profiles = 0`
- `persona_mentions = 10`
- `biography_records = 12`
- `merge_suggestions = 2`

因此“没有一个角色”并非真正没有抽取，而是写入层与读取层不一致。

### 2.2 “0 角色”直接根因

`threestage` Stage B 当前会写：

- `persona`
- `personaMention.promotedPersonaId`
- `mergeSuggestion`

但书籍角色列表与详情聚合仍大量依赖：

- `profiles`

结果是：

1. 三阶段成功晋级了实体。
2. 旧读取口径没有消费这些实体。
3. 前台最终显示为 0 角色。

这不是 Prompt 偶发失误，而是读写模型设计错位。

### 2.3 第二个更严重的问题：全书结果异常稀疏

该书共有 56 章，章节正文完整存在，但当前三阶段结果只在第 31 章产出：

- `persona_mentions` 仅出现在 chapter 31
- `biography_records` 仅出现在 chapter 31

这说明当前问题不只是一层读取错位，还包括：

- 三阶段阶段执行缺少可观测性
- 无法判断 Stage A 是否真正遍历全部章节
- 无法判断 Stage B / Stage C 是否因过滤、批处理、异常吞掉了绝大多数结果

当前单条 `analysis_phase_logs` 仅可见 `BOOK_VALIDATION|SUCCESS`，远不足以支撑生产诊断。

## 3. 设计目标

### 3.1 主目标

- 把 `threestage` 收口为全书解析单一路径。
- 统一角色主真相源，修复“作业成功但角色为 0”。
- 为全书解析建立阶段级可观测性，能解释异常结果。

### 3.2 非目标

- 不重构 `sequential`。
- 不把按章分析改造成三阶段。
- 不在本次设计中承诺立即达到 85%+ 业务准确率。

## 4. 架构决策

### 4.1 全书解析只认 `threestage`

决策：

- `FULL_BOOK` 解析的唯一目标架构是 `threestage`。
- 旧 `twopass` 兼容只允许存在于历史数据迁移或一次性脚本，不再作为运行时产品语义。

原因：

- 当前运行时已经基本切到 `sequential | threestage` 二选一，继续保留旧兼容只会扩大认知成本。
- 用户要求是新项目，不需要为旧概念持续付出复杂度。

### 4.2 角色主真相源迁移到 `persona`

决策：

- 对于 `threestage/FULL_BOOK` 结果，书内人物主真相源定义为 `persona`。
- `personaMention` 是候选观察层，不直接面向用户。
- `Profile` 不再作为三阶段书籍角色列表的读取依据。

推荐落地方式：

1. 先改读路径，使 `listBookPersonas`、`getBookById` 等聚合直接读取 `persona` 及其统计字段。
2. 若仍需兼容旧页面结构，可在查询层做投影适配，而不是继续以 `profiles` 为真相。
3. `Profile` 后续只保留二选一角色：
   - 历史兼容投影
   - 待清理遗留表

不推荐继续方案：

- Stage B 反向补写 `Profile` 只为迎合旧页面。

原因：

- 这会把旧模型错误固化为长期协议。
- 以后任何 persona 语义增强都要双写，成本持续增加。

### 4.3 历史兼容的处理原则

本次收口不要求“一次删库式清理”，但要求明确分层：

- 运行时主路径：不再依赖旧兼容语义。
- 读模型兼容：允许短期存在，但只能是投影，不得反向定义写模型。
- 历史数据兼容：通过迁移脚本、回填脚本、一次性修复处理，不进入长期核心逻辑。

## 5. 需要清理或弱化的残留点

### 5.1 `Profile` 仍是主读取口径

问题：

- 这是当前“0 角色”最直接根因。

要求：

- 所有三阶段书籍详情、人物列表、统计卡片必须改为以 `persona` 为源。

### 5.2 `Persona.aliases` 与 `AliasEntry` / `AliasMapping` / `suspectedResolvesTo` 并存

问题：

- 别名语义分散，容易出现“一个地方认为已归并，另一个地方仍视作不同实体”。

要求：

- 在设计上明确别名主来源。
- 推荐把 `AliasEntry` 作为人工审核后的知识入口，把 `suspectedResolvesTo` 作为运行时候选证据。
- `Persona.aliases` 若保留，只能是派生展示字段，而非仲裁依据。

### 5.3 三阶段没有像 sequential 那样显式维护章节进度

问题：

- 无法直接回答“为什么 56 章只有 1 章有结果”。

要求：

- 为 `threestage` 增加章节处理状态或等价阶段日志。
- 至少能记录每章在 Stage A / Stage C 的开始、成功、失败、跳过。

## 6. 可观测性设计

### 6.1 阶段级日志

新增或补齐以下可观测信息：

- Stage A
  - 总章节数
  - 已处理章节数
  - 每章 mention 数
  - 失败章节数与失败原因
- Stage B
  - 输入 mention 总数
  - 晋级 persona 数
  - merge suggestion 数
  - 被过滤 mention 数及原因分布
- Stage C
  - 输入 persona 数
  - 每章归因 biography 数
  - 跳过章节数及原因

### 6.2 作业完成前的异常护栏

对 `FULL_BOOK/threestage` 增加最小合理性检查，避免“形式成功、业务空结果”：

- 若处理章节数远低于全书章节数，不能直接标记为普通成功。
- 若 `persona_mentions > 0` 且 `profiles = 0` 但页面仍读 `profiles`，这是设计缺陷，不应被成功掩盖。
- 若全书只单章出结果，应记录告警信号。

状态建议：

- 保留 `SUCCEEDED`
- 增加 `completed_with_warnings` 类告警字段或 phase warning 机制

重点不是改状态名，而是让异常可见、可追。

## 7. 针对书籍 61c34092-5295-4438-af28-46b5b0e9d558 的验证方案

### 7.1 修复后必须满足

- 书籍角色列表不再为 0。
- 可以从日志中看出 Stage A 实际处理了哪些章节。
- 可以从日志中看出 Stage B 输入多少 mention、产出多少 persona。
- 可以从日志中看出 Stage C 是否仅处理了 chapter 31，若是，原因是什么。

### 7.2 最低回归检查

- 回放该书最新或重跑作业后：
  - `persona` 数 > 0
  - 页面展示角色数 = `persona` 聚合结果
  - `analysis_phase_logs` 或替代日志包含 Stage A/B/C 关键指标
  - 不再出现“只有 BOOK_VALIDATION 日志”的不可诊断状态

## 8. 实施边界

### 8.1 本次纳入

- 三阶段全书解析的主真相源统一
- 旧读路径兼容收口
- 异常作业可观测性补齐
- 特定异常书籍回归验证

### 8.2 本次不纳入

- `sequential` 路径的算法调整
- 按章分析 UI/接口重做
- 全量知识库重建
- 对外承诺 85%+ 业务准确率

## 9. 风险与取舍

### 9.1 风险

- 如果直接移除旧 `Profile` 读取而不提供投影兼容，部分旧页面可能断裂。
- 如果只做展示层映射、不收口底层真相，后续仍会持续出现双模型漂移。
- 如果不补可观测性，即便修掉“0 角色”，下次仍无法解释稀疏输出。

### 9.2 取舍

推荐采用“先统一真相源，再保守兼容展示”的渐进方案：

1. 查询层先切到 `persona`。
2. 旧页面需要的结构通过查询投影补齐。
3. 再逐步移除 `Profile` 主业务职责。

这样风险最小，也最符合“这是新项目，不保留无意义兼容”的要求。

## 10. 结论与下一步

下一步实施应围绕四件事展开：

1. 改造角色读取口径，停止以 `profiles` 作为三阶段书籍人物真相。
2. 梳理并弱化旧兼容字段与旧别名语义。
3. 为 `threestage` 增加阶段级日志、章节级进度与异常警告。
4. 以 `61c34092-5295-4438-af28-46b5b0e9d558` 为回归样本验证修复闭环。

本设计故意不动按章分析，因为当前任务目标不是替换准确率更好的按章路径，而是把全书解析链路收口到一个业务上可解释、可验证、不会再出现“成功但 0 角色”的状态。
