# threestage 架构业务适配性评估

## 0. 结论先行

本次评估的结论很明确：

1. `threestage` 在架构方向上是正确的，长期比 `sequential` 更接近“角色数量满足 + 事迹满足”的业务目标。
2. 但以当前实现状态，`threestage` 还不能单独承担唯一主方案，原因不是理念错误，而是召回保底、失败隔离、事迹沉淀、诊断能力都还不够成熟。
3. 因此当前最合理的业务判断是：
   - `sequential` 继续作为结果保底路径保留，不动当前实现；
   - `threestage` 继续作为未来主方案重点优化；
   - 在补齐关键短板前，**不能严肃承诺“已经稳定达到角色数量、事迹都满足，且准确率稳定在 85% 以上”。**

更直白地说：

- `threestage` 不是不该做，而是还没有工程化到足以稳定兑现价值。
- 当前它的风险已经不只是“识别偶尔不准”，而是会出现“整本书直接空结果”或“跑完了但角色/事迹偏薄”的业务失败。

---

## 1. 本次评估范围

本次不是重新讨论两套架构谁“更优雅”，而是围绕用户明确目标做业务可用性判断：

1. 角色数量要足够接近真实书中有效人物规模。
2. 角色事迹要有足够覆盖，不能只有名字没有事。
3. 同一角色的跨章称谓、别名、身份变化不能大量拆散或误合并。
4. 失败后要能快速定位是模型、余额、模板、知识库还是阶段逻辑问题。

因此，本次的核心问题不是“`threestage` 是否理论成立”，而是：

`threestage` 是否已经足以稳定满足“角色数量 + 事迹质量”的主方案标准。`

---

## 2. 评估框架

本次按五个维度判断 `threestage` 的业务适配性：

1. `角色数量召回`
   - 是否能把核心角色、次核心角色、稳定反复出现的称谓人物抓进候选池。
   - 是否容易出现“该有人却没抓到”的结构性漏召回。
2. `事迹覆盖`
   - 是否能把主要行为、关键经历、跨章事件沉淀到角色身上。
   - 是否容易出现“有人但没有足够 biography/event”的空壳角色。
3. `跨章一致性`
   - 是否能正确处理同名异人、同人多称、冒名、借称、官称、亲属称等复杂关系。
4. `失败韧性`
   - 模型失败、余额不足、提示词异常、章节异常时，是否会放大成整本书失效。
5. `可诊断性`
   - 出问题后，是否能快速知道到底丢在 Stage A、Stage B、Stage C，还是模型/配置层。

这套框架下，`sequential` 和 `threestage` 的优劣不是抽象比较，而是直接映射到“角色数量够不够、事迹够不够、能不能稳定跑完”。

---

## 3. 支持 threestage 作为长期主方案的论点

### 3.1 它天然更适合“先召回、再归并、再挂事”

从 [`ThreeStagePipeline.ts`](/home/mwjz/code/wen-yuan/src/server/modules/analysis/pipelines/threestage/ThreeStagePipeline.ts)、[`StageAExtractor.ts`](/home/mwjz/code/wen-yuan/src/server/modules/analysis/pipelines/threestage/stageA/StageAExtractor.ts)、[`StageBResolver.ts`](/home/mwjz/code/wen-yuan/src/server/modules/analysis/pipelines/threestage/stageB/StageBResolver.ts)、[`StageCAttributor.ts`](/home/mwjz/code/wen-yuan/src/server/modules/analysis/pipelines/threestage/stageC/StageCAttributor.ts) 可见，当前三阶段已经把问题拆成：

1. Stage A：逐章提取 mention 候选
2. Stage B.5：时序一致性检查
3. Stage B：全书实体仲裁、晋级 persona
4. Stage C：基于已晋级 persona 再做事迹归属

这比逐章边读边定案更符合人物识别本质，因为：

- “原文中出现了谁”与“这个称谓最终归到谁”是两件事；
- “这个人是谁”与“这件事归谁”也是两件事；
- 把这些问题拆开，才有机会同时提升角色数量与事迹质量。

### 3.2 它对复杂古典文本更有上限

对《儒林外史》这类古典文本，复杂点不在简单 NER，而在：

- 同姓多人
- 官称/泛称/亲属称频繁出现
- 同一角色称谓多变
- 借名、冒名、误认、代称混杂
- 叙述、对白、评点、诗词区段交织

在这种书型里，若过早把章节内称谓直接定成人物，通常会带来两个后果：

1. 角色数量过量，出现大量临时 persona；
2. 事迹过早挂错人，后续难以回收。

三阶段至少在架构上允许：

- 先保留 mention 候选，不急着建人；
- 等全书上下文更充分后再做归并；
- 先把人边界做稳，再把事挂上去。

这正是它比 `sequential` 更值得继续投资的原因。

### 3.3 它更适合作为长期演进底座

如果未来要持续提升准确率，`threestage` 比 `sequential` 更适合插入新能力：

- Stage A 前后增加规则层或轻量补召回层
- Stage B 增加更强的别名证据、时序约束、人工审核入口
- Stage C 增加更强的 biography 聚合和事件去噪
- 各阶段分别做指标与回归，而不是所有问题混在一次章节调用里

从研发演进角度看，`threestage` 的可塑性明显更强。

## 4. 反对 threestage 立即成为唯一主方案的论点

### 4.1 Stage A 仍然是过强单点

当前 `threestage` 的事实运行方式是：

- 先由 Stage A 产出 `persona_mentions`
- Stage B / C 都建立在 Stage A 已有有效候选的前提上

这意味着：

- Stage A 抓漏，后面很难补救；
- Stage A 调用失败，后面直接空跑；
- Stage A 召回不足，再聪明的 Stage B / C 也无从发挥。

对你的业务目标，这个问题尤其致命，因为你最在意的是：

- 角色数量必须够
- 角色还要有足够事迹

而这两件事都建立在入口召回足够全的前提上。

### 4.2 它当前缺少“召回保底”能力

现在的 Stage A 更像一个“主提取器”，还不是“保底召回器”。

在古典小说里，LLM 仅靠一次 prompt 提取容易漏掉以下对象：

- 反复出现但表达不稳定的官称/称谓
- 同族同辈人物
- 一度冒名、暂时伪装、被误认的人物
- 非主角但稳定参与情节的次级角色

这些漏掉的对象一旦没进入 mention 池，后面不会自动补回来。  
因此当前 `threestage` 的最大结构性风险不是“多抓错几个”，而是“漏掉很多该进池的人物”，这会直接伤害角色数量。

### 4.3 它当前缺少“失败隔离”能力

本次真实排查已经证明这一点。

对书籍 `fc80efc0-6add-4adb-838a-35ba3504a144`（《儒林外史》）：

- `books.error_log = 所有章节解析失败，共 56 章`
- 最新作业 `fbfb017b-d03e-4627-9109-b2b1c9aa8b52`
- `architecture = threestage`
- `Stage A` 统计为 `completedChapters = 0`, `failedChapters = 56`
- 所有章节仍停留在 `PENDING`

进一步只读复现第一章的实际运行时错误后，确认真实 provider 报错是：

- `Insufficient Balance`

这说明当前问题不是《儒林外史》“没有角色”，而是：

1. Stage A 调统一 AI client；
2. 该模型运行时因余额不足不可用；
3. 56 章一起失败；
4. 后面的 Stage B / C 只是在空输入上收尾；
5. 外层只看到“所有章节解析失败”。

这类故障属于基础设施失败，不应被包装成普通业务解析失败，更不应放大成整本书 0 结果。

### 4.4 它当前缺少“结果真实性”与“显示一致性”保障

另一组反例来自书籍 `61c34092-5295-4438-af28-46b5b0e9d558`。

在该书上，之前已经确认过一次不同类型的问题：

- `threestage` 作业成功
- `persona_mentions`、`biography_records`、`merge_suggestions` 都不为 0
- 但前台显示“没有一个角色”

其本质不是 Stage A 没抽到，而是：

- 写入层已走 `persona`/`personaMention`
- 读取层仍大量依赖旧 `profiles`
- 最终出现“数据库有结果，页面显示 0 角色”

这说明当前 `threestage` 的问题不只是算法层，还包括读写模型尚未完全收口。  
对业务来说，这种“实际有结果但显示为空”与“真的没解析出结果”同样不可接受。

### 4.5 它当前的可诊断性不足

当前链路中，存在两个明显诊断断点：

1. 章节实际上失败了，但章节状态未真实写成 `FAILED`，仍可能停在 `PENDING`
2. 基础设施错误、模型错误、模板错误、业务规则错误没有被明确分层展示

这会带来一个后果：

- 当你看到“0 角色”或“全失败”时，无法快速知道究竟是：
  - 模型余额不足
  - provider 不可用
  - prompt/template 缺失
  - few-shot 为空
  - Stage A JSON 解析失败
  - Stage B 过滤过猛
  - Stage C 没挂上 biography
  - 还是读模型错位

对于后续优化“角色数量”和“事迹质量”，这种不可诊断性会严重拖慢迭代效率。

## 5. 为什么当前体感上按章分析更准

用户反馈“按章分析准确率反而不错”是合理现象，不是错觉。

### 5.1 按章分析的失败是分散的

`sequential` 的一个现实优势是：  
它通常坏一章是一章，不容易因为一个统一模型或统一入口失效导致整本书一起归零。

在工程成熟度不够的阶段，这种“局部坏”会给人更稳定的体感。

### 5.2 按章分析更容易抓到章节内显性人物和显性事件

对很多章节而言，短上下文下：

- 当前章谁出现了
- 当前章谁说了什么
- 当前章谁做了什么

这些问题反而比较直接。  
所以按章分析即使全书统一能力弱，也可能在“每章都抓出一些角色与行为”上看起来更扎实。

### 5.3 threestage 的理论优势尚未完全兑现成工程收益

`threestage` 的优势本应体现在：

- 更好的全书一致性
- 更少的误合并/漏合并
- 更强的事迹回填质量

但在当前阶段，它的入口保底、失败隔离、读写收口还没完全做好，于是会出现：

- 理论上限更高
- 实际稳定性反而不如按章分析

这就是为什么你会觉得“按章分析反而更准”。

---

## 6. 对“85%+ 准确率是否已可稳定承诺”的判断

如果业务口径是：

- 角色数量接近真实规模
- 主要角色事迹能挂到正确的人
- 结果稳定，不因单点故障整书归零

那么当前我不认为 `threestage` 已经可以稳定承诺 85%+。

理由不是“它一定做不到”，而是：

1. `角色数量` 仍受 Stage A 单点召回强约束
2. `事迹质量` 仍明显依赖 Stage C 的后续增强
3. `失败韧性` 还不足以支撑生产级稳定性
4. `可诊断性` 还不足以支撑高效迭代校准

也就是说，当前问题已经不只是模型能力问题，而是系统工程成熟度不足。

---

## 7. 必须补齐的能力

下面这些不是“锦上添花”，而是 `threestage` 想成为唯一主方案前必须补齐的最小能力。

### 7.1 Stage A 保底召回

目标：

- 让 Stage A 从“单次硬提取器”升级成“高召回入口层”

要求：

- 保留当前 LLM Stage A 主提取
- 增加规则/轻量补召回层，只负责补明显漏掉的人物候选
- 允许“先高召回，再由 Stage B 控制精度”，而不是入口过度保守

原因：

- 对你的目标，入口少抓比入口多抓更伤
- 角色数量不够时，后续再精确也没有意义

### 7.2 失败隔离与降级

目标：

- 不允许模型/余额/provider 级问题直接放大成整本书 0 结果

要求：

- 基础设施错误与业务解析错误分层
- 主模型失败时允许降级到备用模型或备用通道
- 整书任务允许“部分成功 + 警告”，而不是只有成功/失败两极
- 章节失败必须真实落状态，不得长期停留 `PENDING`

### 7.3 事迹沉淀增强

目标：

- 让 Stage C 从“给角色挂少量动作”升级成“形成足够支撑角色画像的事件沉淀层”

要求：

- 不只抓动作词，还要沉淀关键事件片段
- 支持跨章累计，而不是仅有零碎局部 biography
- 区分“被提到”和“发生了有价值的事”
- 输出的 biography/event 要能支撑角色画像，而不是只形成空壳角色

### 7.4 读写主真相收口

目标：

- 避免再次出现“作业成功但页面 0 角色”的错位

要求：

- 对 `threestage/FULL_BOOK`，统一以 `persona` 为角色主真相源
- `personaMention` 作为观察候选层
- 旧 `Profile` 若保留，只能作为兼容投影，不再定义主读取语义

### 7.5 阶段级指标与诊断

目标：

- 让后续优化有证据，不再靠猜

至少应长期追踪：

- 每章 mention 数
- 每章成角数
- 全书 persona 总数
- 每角平均有效事迹数
- 空产出章节数
- 低置信章节数
- 失败章节数及失败原因分布

没有这些指标，后续每次“角色数量不够”或“事迹太薄”都很难精确定位。

---

## 8. 优先级排序

如果不动 `sequential`，只优化 `threestage`，建议严格按以下顺序推进：

1. `Stage A 保底召回`
2. `失败隔离与降级`
3. `读写主真相收口`
4. `事迹沉淀增强`
5. `阶段级指标与诊断`

排序原因：

- 先保角色数量不塌，再谈全书质量
- 先保整书不会归零，再谈精细优化
- 先保证结果真实可见，再谈业务体验
- 最后再持续提高事迹密度与准确率

---

## 9. 最终判断

站在当前业务目标上，结论如下：

1. `threestage` 的方向合理，且仍然是未来主方案的正确候选。
2. `sequential` 不是长期理想架构，但在当前阶段仍承担着结果保底作用，因此不应在此轮随意动它。
3. 以当前实现成熟度，`threestage` 还不具备独占主方案资格。
4. 真正阻碍它的已经不是“有没有三阶段思路”，而是：
   - 入口召回不够保底
   - 单点失败会放大成整本书失败
   - 事迹沉淀不够强
   - 诊断与读写收口还不够完整

因此，当前最合理的业务策略不是回退 `threestage`，而是：

- 不动按章分析
- 继续把 `threestage` 作为重点主攻方向
- 先补齐最小可用能力，再讨论让它替代按章分析成为唯一主方案

这也是本次评估的最终结论。
