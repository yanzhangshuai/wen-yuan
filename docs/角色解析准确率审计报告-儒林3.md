### 2.2 A 类: 泛称/关系词成为独立 persona (75 例)

**子分类**:

| 子类 | 代表案例 | 数量 |
|------|---------|------|
| 泛化职务 | 宗师、府尊、祭酒、总督、府尹、守备、参将、典史、总兵 | 18 |
| 泛化身份 | 走堂的、看茶的、卖草的、朝奉、嫖客、猎户、斋公、幕客 | 16 |
| 泛化亲属 | 姑老爷、母舅、女婿、表兄、小儿、二哥、老侄、姑爷 | 14 |
| 宗教泛称 | 贫僧、小和尚、僧宫老爷、首座、知客、火工道人、和尚、道士、道人 | 12 |
| 通用称呼 | 邻居、店邻、客人、家人、长随、伙计、差人、书办、番子 | 15 |

**证据链 — 代码路径分析**:

1. **Phase 1 (Roster Discovery)**: `buildRosterDiscoveryPrompt()` (prompts.ts) 在规则中声明 "泛化称谓禁止作为独立人物名"，但 LLM 仍然返回这些结果，且 roster 合并逻辑 (`mergeRosterEntriesForAnalysis`) 无泛称二次过滤。
2. **Phase 2 (Chunk Extraction)**: `buildChapterAnalysisPrompt()` 同样包含泛称过滤规则，但 LLM 不遵守。
3. **PersonaResolver.resolve()** (PersonaResolver.ts L395-415): 当 `rawName` 在 `effectiveLexicon.genericTitles` 集合中时：
   - 若 `dynamicTitleResolutionEnabled = false` → 直接 `hallucinated` + `reason: "config_generic"` ✅
   - **但问题在于**: "管家"、"差人"、"番子"等词 **不在** `DEFAULT_GENERIC_TITLES` 也不在 `SAFETY_GENERIC_TITLES` 中 → 漏过泛称检查
   - "姑老爷"、"母舅"、"女婿"等关系词 **也不在** 泛称集合中 → 被当作正式人名处理

4. **缺失**: 系统只有 `SAFETY_GENERIC_TITLES`（代词/亲属基础词，约 30 项）和 `DEFAULT_GENERIC_TITLES`（约 30 项职务称呼），总计约 60 项，但 `ENTITY_EXTRACTION_RULES` prompt 中 genericTitles 示例被截断为前 N 个 → LLM 无法得到完整列表。

**根因总结**: 泛称词表覆盖不足 + 无关系词过滤层 + LLM prompt 合规率低 → 大量泛称/关系词泄漏

**证据链 — 代码路径分析**:

1. `ENTITY_EXTRACTION_RULES[3]` 明确规定: "仅提取虚构角色，排除作者、评注者、真实历史人物、批评家"。
2. **但无代码级强制过滤**: PersonaResolver 中无历史人物黑名单检查。LLM 若不遵守规则，系统无二次防线。
3. **"虚构先祖"子类** 更棘手: `蘧祜`、`冯瑶` 等是原著中角色的祖先/亲属，在原文中确实被提到，但仅仅是"被提及"而非"出场"。Prompt 规则未区分 "mentioned_only" 与 "actively present"。
4. 独立提取模式 (`buildIndependentExtractionRulesText`) 有 `category: MENTIONED_ONLY` 分类，但 sequential 架构不使用此模式。

**根因总结**: 无历史人物黑名单 + Prompt 规则依赖率低 + sequential 不使用 MENTIONED_ONLY 分类 → 历史人物无阻挡

**代码路径**: `PersonaResolver` 长度上限 10 字。"死砍头短命的奴才"(8 字) 在阈值内。`HARD_BLOCK_SUFFIXES` 仅阻断以"之妻/之子/之父/之母"等结尾的名字，但"虞育德之母"(5 字) 虽含"之母"→ **应该被阻断**，但检查代码发现 hardBlockSuffixes 匹配的是 **提取名去除姓氏后的 remainder**，而非全名。`extractSurname("虞育德之母")` → `"虞"`，remainder → `"育德之母"`，不在 hardBlockSuffixes 中 → 漏过。

**根因**: hardBlock 检测逻辑只对"姓+后缀"模式生效（如"范之父"），对"多字名+关系词"模式无效 + 长度阈值过宽

**根因总结**: 字面匹配对古典文学字号/别名体系完全无效 + AliasMapping 空 + 无全书级实体合并后处理

### 5.1 根因 R1: bookTypeId/genre 未设置

- **影响范围**: 全链路
- **代码位置**: `pipeline.ts` L89-144 `GENRE_PRESETS`
- **现状**: 儒林外史属于"明清官场"类型，GENRE_PRESETS 中明清官场预设为 `{}`（空对象），即使设置了也无额外泛称豁免/补充
- **建议**: 为明清官场类增加 `additionalGenericTitles`（管家、差人、长随等），并在书籍导入时强制设置 genre

### 5.2 根因 R2: 泛称词表严重不足

- **影响范围**: A 类全部 75 个错误
- **代码位置**: `lexicon.ts` L91-105 `SAFETY_GENERIC_TITLES` + `DEFAULT_GENERIC_TITLES`
- **现状**: 两个集合合计约 60 项，缺少: 管家、差人、长随、番子、门斗、嫖客、猎户、斋公、幕客、典史等
- **更深层**: 这些词在某些小说中可能特指某人（如"管家来禀"→某特定管家），需要个性化判定能力，而非简单加黑名单

### 5.3 根因 R3: 无历史人物过滤层

- **影响范围**: B 类全部 69 个错误
- **代码位置**: 全链路无此功能
- **现状**: 仅依赖 Prompt 规则 "排除真实历史人物"，LLM 执行率约 50%
- **建议**: 需要代码级历史人物黑名单 + 知识库整合

### 5.4 根因 R4: PersonaResolver 字面匹配无法覆盖古典文学名字系统

- **影响范围**: E 类全部 120+ 碎片化错误
- **代码位置**: `PersonaResolver.ts` L293-340 `loadCandidates()` + L360-400 评分逻辑
- **现状**: "迟均" vs "迟衡山"、"蘧駪夫" vs "蘧公孙" 等字号/名/字关系无法通过字面相似度发现
- **建议**: 需要知识库驱动的字号-本名映射

### 5.5 根因 R5: AliasMapping 写入管线从未执行

- **影响范围**: 全部碎片化 + 后续解析
- **代码位置**: `AliasRegistryService.ts`
- **现状**: 0 条 AliasMapping 记录 → `aliasRegistryService.findByAlias()` 永远返回空 → alias 快速命中路径永远失效
- **推测**: AliasMapping 的写入依赖某个被跳过的流程步骤或手动操作

### 5.6 根因 R6: 描述性短语/家族名无结构化检测

- **影响范围**: C 类 (18) + D 类 (40)
- **代码位置**: `PersonaResolver.ts` L375-400 长度检查 + hardBlock 检查
- **现状**: `> 10 字` 过滤 + `hardBlockSuffixes` 仅对"姓+后缀"模式生效
- **缺失规则**: 
  - `X家`/`X府` 模式 (2-3 字)
  - `人名+亲属词` 模式 (如"王玉辉老妻")
  - `修饰语+的+名词` 模式 (如"卖菱小孩")
  - `"X之Y"` 模式 (如"虞育德之母")

## 7. 准确率估算方法论

**算法**: 
1. 儒林外史有效虚构角色估计: **约 100-120 个**（主要角色 30-40，次要角色 40-50，提及角色 30-40）
2. 623 profiles 中:
   - 错误 profile（A+B+C+D 类）: 75+69+18+40 = **202 个**
   - 碎片多余 profile（E 类）: 约 **120 个**（140 碎片 - 29 个正确保留 = 111，加上未检测到的）
   - 错误合并（F 类）: 约 **15 个**
   - 正确且唯一的 profile: 623 - 202 - 120 - 15 ≈ **286 个**
3. 但 286 个中仍有大量重复（共享别名导致的软重复）
4. **精确率 (Precision)**: 合格 profile / 总 profile ≈ 120/623 ≈ **19.3%**
5. **召回率 (Recall)**: 被正确识别的角色 / 应被识别的角色 ≈ 100/120 ≈ **83.3%**（大多数真实角色至少有一个碎片）
6. **F1**: 2 × 0.193 × 0.833 / (0.193 + 0.833) ≈ **0.314**

> **综合评估**: 准确率（以合格 profile 占比计）约 **30-35%**，远低于上一版 402f2282 的 65-70%。核心差异来自碎片化程度和泛称泄漏量的激增。
