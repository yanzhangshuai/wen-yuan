## 1. 优化项逐项分析

### 1.1 条件触发章节验证

#### 1.1.1 背景与现状

**现有行为**：`workerLoop()`（`runAnalysisJob.ts` 约 L555-557）在每章 `analyzeChapter` 成功后无条件调用 `runChapterValidationBlocking`。

**代码路径**：
- `src/server/modules/analysis/jobs/runAnalysisJob.ts` → `workerLoop()` → `runChapterValidationBlocking()`
- 每次调用产生 7 次 DB 查询 + 1 次 AI generateJson 调用

**当前造成的问题**：
1. 一本 50 章的书会产生 50 次 AI 验证调用，即使大部分章节只是重复确认已有人物
2. 每次验证消耗 ~5800-7000 输入 token（3000 字原文 ≈ 4500 token + 静态指令 ~522 token + 人物/提及/关系数据 ~800-2000 token）
3. 每次验证增加 ~3-8 秒串行延迟（DB 查询 + AI 调用）

#### 1.1.2 必要性分析

**为什么值得做**：
- 验证的核心价值在于发现"新建人物的质量问题"。当某章无新建人物（只是已有人物的再次出现），验证几乎不会发现有价值的 issue
- 审计报告 🟡#12 明确建议"条件触发"
- 这是当前实现中性价比最高的 token 优化点

**不做的代价**：
- 50 章书籍中约 60-70% 的章节验证是低价值调用（大多数章节新建人物 < 3）
- 全书额外消耗 ~120,000-200,000 token 用于低价值验证
- 全书额外耗时 ~90-240 秒（假设每次验证 3-8 秒）

**主要影响**：性能 ✅ token ✅ 正确性（可控）

#### 1.1.3 收益评估

**估算模型**：
- 假设 50 章书籍，每章验证 token ≈ 6500 input + 500 output = 7000 token
- 假设 60-70% 的章节可安全跳过验证（新建人物 < 3）

| 指标 | 保守（跳过 50%） | 中性（跳过 65%） | 乐观（跳过 75%） |
|------|-----------------|-----------------|-----------------|
| 节省验证 token（50 章） | 175,000 | 227,500 | 262,500 |
| 节省 AI 调用次数 | 25 次 | 32 次 | 37 次 |
| 节省 DB 查询 | 175 次 | 224 次 | 259 次 |
| 节省串行耗时 | 75-200s | 96-256s | 111-296s |
| 占全书验证 token 比例 | 50% | 65% | 75% |

**估算依据**：
- 典型古典文学（如《儒林外史》55 回），前 10-15 回人物密集建档，后续章节多为已有人物复现
- `created.personas` 是已有字段（`ChapterAnalysisResult` 中），零成本获取

#### 1.1.4 设计方案

**修改文件**：
1. `src/server/modules/analysis/config/pipeline.ts` — 新增配置项
2. `src/server/modules/analysis/jobs/runAnalysisJob.ts` — `workerLoop()` 中添加条件判断

**新增配置项**（`pipeline.ts`）：

```typescript
/** 触发章节验证的最低新建人物数。低于此值跳过验证以节省 token。 */
chapterValidationMinNewPersonas: 3,
/** 触发章节验证的最低幻觉数。即使新建人物少，幻觉多也应验证。 */
chapterValidationMinHallucinations: 2,
```

**条件判断逻辑**（`runAnalysisJob.ts` `workerLoop()` 内，L555-557 之间）：

```typescript
// 原：
const validationResult = await runChapterValidationBlocking(job.id, chapter, job.bookId);

// 改为：
const shouldValidate =
  result.created.personas >= ANALYSIS_PIPELINE_CONFIG.chapterValidationMinNewPersonas ||
  result.hallucinationCount >= ANALYSIS_PIPELINE_CONFIG.chapterValidationMinHallucinations;

let chapterNeedsReview = false;

if (shouldValidate) {
  const validationResult = await runChapterValidationBlocking(job.id, chapter, job.bookId);
  if (validationResult.needsReview) {
    chapterNeedsReview = true;
    console.warn(
      "[analysis.runner] chapter.validation.needs_review",
      JSON.stringify({
        jobId, chapterId: chapter.id, chapterNo: chapter.no,
        reportId: validationResult.reportId,
        errorCount: validationResult.errorCount
      })
    );
  }
} else {
  console.info(
    "[analysis.runner] chapter.validation.skipped",
    JSON.stringify({
      jobId: job.id, chapterId: chapter.id, chapterNo: chapter.no,
      reason: "below_threshold",
      newPersonas: result.created.personas,
      hallucinations: result.hallucinationCount
    })
  );
}
```

**状态流转**：
- `shouldValidate = true` → 走现有完整验证流程
- `shouldValidate = false` → 跳过验证，标记 SUCCEEDED（无 NEEDS_REVIEW 风险），记录跳过原因日志

**为什么选择多条件触发而非单条件**：
- 单看 `created.personas >= 3` 会遗漏高幻觉率章节（幻觉多意味着 AI 提取质量差，值得复审）
- 两个条件覆盖"新建人物多"和"AI 质量差"两种场景
- 条件之间是 OR 关系，任一命中即触发

**Schema 变更**：无
**迁移脚本**：无
**前端兼容**：无影响

#### 1.1.5 风险与副作用

**精度影响分析**：
- 跳过验证的章节特征是"新建人物少、幻觉少"——这类章节本身质量已较高
- 验证的核心价值在于发现新建人物的 DUPLICATE/HALLUCINATION/WRONG_TYPE 问题，当新建人物 < 3 时，这些问题的发生概率极低
- **已有人物的关系/提及仍然由 PersonaResolver 保障质量**，不依赖 validation

**可能遗漏的场景**：
- 已有人物的关系被错误提取（但 validation 对此的召回率本身也不高——推断）
- 一个章节只新建 1 个人物，但该人物恰好是重复项（概率极低，且会在后续章节被全书验证覆盖）

**监控指标**：
- 日志 `chapter.validation.skipped` 的频率与分布
- 全书验证（Phase 6）发现的 issue 数量是否因跳过章节验证而增加

**回滚策略**：将 `chapterValidationMinNewPersonas` 设为 0 即可恢复全量验证

#### 1.1.6 测试与验收

**单元测试**（`runAnalysisJob.test.ts`）：

2. 新测试：`"triggers chapter validation when hallucinations exceed threshold"`
   - Mock `analyzeChapter` 返回 `created.personas: 0, hallucinationCount: 3`
   - 断言 `validateChapterResult` 被调用
3. 现有测试：`"does not abort whole job when chapter validation reports errors"` — 不受影响（该测试的 mock 默认 created.personas 足够触发验证）

**验收标准**：
- 全部现有 530 测试通过
- 新增 ≥ 2 条针对条件触发的测试
- `pnpm lint` 0 错误

### 1.2 Chunk Profiles 按 Roster 过滤

#### 1.2.1 背景与现状

**现有行为**：`analyzeChapter()`（`ChapterAnalysisService.ts` L282）将全书所有 profiles 注入每个 chunk 的 prompt。

**代码路径**：
```
analyzeChapter() → 加载 chapter.book.profiles（无 LIMIT）
  → Phase 1: buildRosterDiscoveryPrompt(profiles) — 全量
  → Phase 2: 对每个 chunk 调用 buildChapterAnalysisPrompt({ profiles }) — 全量
```

**当前造成的问题**：
- 50 人物的书，每个 profile 约 80-150 token（`[1] 范进 | 别名: 范老爷, 范举人 | 小传: ...`）
- 50 profiles × 125 avg token = 6250 token 注入每个 chunk
- 一个 6000 字章节有 ~3 chunks → 3 × 6250 = 18,750 token 仅用于重复 profiles
- 书的后半部分可能积累 100+ profiles，进一步放大浪费

#### 1.2.2 必要性分析

**为什么值得做**：
- Profiles 上下文是 chunk analysis prompt 中仅次于原文内容的第二大 token 消耗来源
- 大部分 chunk 只涉及 5-15 个人物，注入 50-100 个无关 profile 是浪费
- Phase 1 Roster 已经识别了本章涉及的所有人物称谓，可作为过滤依据

**不做的代价**：
- 50 章 × 3 chunks/章 × 6250 token/chunk = 937,500 token（profiles 部分）
- 其中约 50-70% 是无关人物的 profiles

**主要影响**：token ✅ 性能（间接，减少 prompt 长度加速 AI 响应）

#### 1.2.3 收益评估

**估算模型**：
- 假设 50 章书籍，平均每章 3 chunks，书籍共 60 profiles
- 每 chunk 平均只涉及 15 个人物（基于 roster 结果）
- 过滤后注入 20 个 profiles（15 + 5 兜底 buffer）

| 指标 | 保守（过滤 40%） | 中性（过滤 60%） | 乐观（过滤 75%） |
|------|-----------------|-----------------|-----------------|
| 单 chunk profiles token 节省 | 2500 | 3750 | 4688 |
| 单章节 token 节省（3 chunks） | 7500 | 11,250 | 14,064 |
| 50 章全书 token 节省 | 375,000 | 562,500 | 703,200 |
| 占全书 chunk analysis token 比例 | ~15% | ~22% | ~28% |

**注意**：Phase 1 Roster Discovery 仍然需要全量 profiles（因为它需要将称谓映射到已有人物档案），不纳入过滤范围。过滤仅影响 Phase 2 Chunk Analysis。

**估算依据**：
- 参考典型古典文学，每章核心出场人物约 5-15 人
- Roster 结果中 `entry.entityId` 已映射到 profiles 的序号索引，可直接用于关联

#### 1.2.4 设计方案

**方案对比**：

| 方案 | 优点 | 缺点 | 推荐 |
|------|------|------|------|
| A: roster 硬过滤 | 简单，直接用 Phase 1 结果 | 可能遗漏 roster 未识别但 chunk 中出现的人物 | ❌ |
| B: roster + 全量兜底 TOP-K | 保底性好 | 仍然注入较多无关 profiles | ❌ |
| C: roster 关联 + 最近活跃 + 高频人物（推荐） | 覆盖率高，token 控制好 | 逻辑稍复杂 | ✅ |

**推荐方案 C 详细设计**：

**修改文件**：
1. `src/server/modules/analysis/config/pipeline.ts` — 新增配置
2. `src/server/modules/analysis/services/ChapterAnalysisService.ts` — 新增过滤函数 + 修改 chunk 循环

**新增配置项**（`pipeline.ts`）：

```typescript
/** Chunk 分析注入 profiles 的最大数量。超过时按相关性排序截断。 */
chunkProfileMaxCount: 25,
```

**新增函数**（`ChapterAnalysisService.ts`）：

```typescript
/**
 * 功能：基于 roster 结果过滤 profiles，只保留与本章高度相关的人物上下文。
 * 策略：roster 映射命中 + isNew 的 suggestedRealName 匹配 + 高频人物兜底。
 * 输入：全量 profiles、roster 结果、rosterMap、配置上限。
 * 输出：过滤后的 profiles 子集。
 */
function filterProfilesForChunk(
  allProfiles: AnalysisProfileContext[],
  roster: ChapterRosterEntry[],
  rosterMap: Map<string, string>,
  maxCount: number
): AnalysisProfileContext[] {
  // Step 1: 收集 roster 命中的 personaId 集合
  const relevantIds = new Set<string>();
  for (const [, personaId] of rosterMap) {
    if (personaId !== "GENERIC") {
      relevantIds.add(personaId);
    }
  }

  // Step 2: roster 中 suggestedRealName 对应的 profiles 也纳入
  for (const entry of roster) {
    if (entry.suggestedRealName) {
      const matched = allProfiles.find(
        p => p.canonicalName === entry.suggestedRealName ||
             p.aliases.includes(entry.suggestedRealName)
      );
      if (matched) {
        relevantIds.add(matched.personaId);
      }
    }
  }

  // Step 3: 按 relevantIds 优先 + 其余按序号兜底
  const primary = allProfiles.filter(p => relevantIds.has(p.personaId));
  if (primary.length >= maxCount) {
    return primary.slice(0, maxCount);
  }

  // 兜底：补充未命中但靠前的 profiles（早期章节创建的通常是核心人物）
  const remaining = allProfiles.filter(p => !relevantIds.has(p.personaId));
  return [...primary, ...remaining.slice(0, maxCount - primary.length)];
}
```

**修改 chunk 循环**（`ChapterAnalysisService.ts` L270-290）：

```typescript
// 原：
const batchPromises = batch.map((chunk, idx) =>
  analyzeChunkWithRetry(runtimeAiClient, {
    ...,
    profiles,  // ← 全量
    ...
  })
);

// 改为：
const chunkProfiles = filterProfilesForChunk(
  profiles,
  roster,
  rosterMap,
  ANALYSIS_PIPELINE_CONFIG.chunkProfileMaxCount
);

const batchPromises = batch.map((chunk, idx) =>
  analyzeChunkWithRetry(runtimeAiClient, {
    ...,
    profiles: chunkProfiles,  // ← 过滤后
    ...
  })
);
```

**重要**：`buildRosterDiscoveryPrompt` 仍使用全量 `profiles`，不受影响。过滤仅应用于 Phase 2 chunk analysis。

**Schema 变更**：无
**迁移脚本**：无
**前端兼容**：无影响

#### 1.2.5 风险与副作用

**精度影响分析**：
- **低风险**：Phase 1 Roster 已覆盖本章所有出现的人物称谓。chunk 中出现的人物必然在 roster 中被识别过（除非 roster AI 漏识别）
- **兜底机制**：即使 roster 漏了某人，`remaining.slice()` 会补充早期核心人物 profiles
- **唯一风险场景**：书中后半部分出现大量新人物（profiles > 100），且 roster 恰好漏了某个罕见称谓。此时该人物不在 chunk profiles 中，可能被 AI 作为新人物处理而非匹配已有档案。但 `PersonaResolver.resolve()` 仍会通过 `multiSignalScore` 在持久化阶段兜底

**监控指标**：
- 比较优化前后的 `created.personas` 平均值（如果明显上升说明匹配率下降）
- Phase 6 全书验证的 DUPLICATE_PERSONA issue 数量

**回滚策略**：将 `chunkProfileMaxCount` 设为 `Infinity` 即可恢复全量注入

#### 1.2.6 测试与验收

**单元测试**（`ChapterAnalysisService.test.ts`）：
1. 新测试：`"filterProfilesForChunk returns roster-matched profiles first"`
2. 新测试：`"filterProfilesForChunk pads with remaining profiles up to maxCount"`
3. 新测试：`"filterProfilesForChunk handles empty roster gracefully"`

**验收标准**：
- 全部现有测试通过
- 新增 ≥ 3 条 filterProfilesForChunk 单测
- `pnpm lint` 0 错误

### 1.3 PENDING → NEEDS_REVIEW 状态修正

#### 1.3.1 背景与现状

**现有行为**：`workerLoop()`（`runAnalysisJob.ts` 约 L622）在 `chapterNeedsReview = true` 时将 `parseStatus` 写回 `"PENDING"`。

```typescript
parseStatus: chapterSucceeded ? (chapterNeedsReview ? "PENDING" : "SUCCEEDED") : "FAILED"
```

**当前造成的问题**：
- `PENDING` 在 schema 中是默认初始值（`@default("PENDING")`），表示"尚未开始解析"
- 复用 PENDING 表示"需人工复审"造成语义歧义
- 前端 `parse-progress-panel.tsx` 中 `ChapterStatusBadge` 对 PENDING 显示灰色"等待中"——实际上该章节已完成解析，只是验证发现问题
- 前端 `canReanalyze` 逻辑仅允许 `FAILED` 或 `SUCCEEDED` 状态重新分析，PENDING 状态下无法触发重分析

#### 1.3.2 必要性分析

**为什么值得做**：
- 用户看到"等待中"会误以为该章节未处理，实际上已经处理完毕只是有质量疑点
- 无法通过 UI 对 NEEDS_REVIEW 的章节发起重分析
- 后续如果需要统计"待复审章节"，无法区分

**不做的代价**：
- 用户体验混乱
- 运维无法区分"真正未开始"和"需复审"的章节
- 潜在的流程阻塞（canReanalyze 不识别 PENDING）

**主要影响**：可维护性 ✅ 前端语义 ✅ 正确性（流程完整性）

#### 1.3.3 收益评估

| 指标 | 评估 |
|------|------|
| 性能影响 | 无 |
| Token 影响 | 无 |
| 代码改动量 | 极小（~20 行） |
| 用户体验提升 | 显著——消除状态歧义 |
| 运维效率提升 | 中等——可通过 SQL 查询待复审章节 |

#### 1.3.4 设计方案

**方案对比**：

| 方案 | 优点 | 缺点 | 推荐 |
|------|------|------|------|
| A: 新增 `NEEDS_REVIEW` 为 parseStatus 合法值 | 语义最清晰，改动最小 | parseStatus 是 String 非 Enum，无需 schema migration | ✅ |
| B: 新增 `reviewFlag` 布尔字段 | 不影响现有 parseStatus 流转 | 需 schema migration + 新字段 | ❌ |
| C: 新增 `reviewStatus` 独立字段 | 分离关注点 | 过度设计，当前只有一种复审来源 | ❌ |

**推荐方案 A**：直接使用 `"NEEDS_REVIEW"` 字符串。

**理由**：
- `parseStatus` 是 `String` 类型（非 Enum），可以直接写入任意值，无需 Prisma migration
- 前端已有 `ChapterStatusBadge` 的 fallback 分支（default case 显示灰色），只需新增一个 `if` 分支
- 改动量最小（3 个文件，约 20 行）

**修改文件**：
1. `src/server/modules/analysis/jobs/runAnalysisJob.ts` — 状态写入
2. `src/app/admin/books/[id]/_components/parse-progress-panel.tsx` — 前端 badge + canReanalyze
3. `prisma/schema.prisma` — 更新注释

**代码变更**：

**文件 1：`runAnalysisJob.ts`（L622）**

#### 1.3.5 风险与副作用

**风险**：极低
- parseStatus 是 String 类型，新增值不会导致 DB 异常
- 已有代码中对 parseStatus 的判断均为精确匹配（`=== "SUCCEEDED"` 等），不会误匹配 `"NEEDS_REVIEW"`

### 1.4 aliasRegistryMinConfidence 监控

#### 1.4.1 背景与现状

**现有行为**：`aliasRegistryMinConfidence` 从 `0.7` 提高到 `0.75`（`pipeline.ts` L4）。

**代码路径**：
- `PersonaResolver.ts` L254：`aliasResult.confidence >= ANALYSIS_PIPELINE_CONFIG.aliasRegistryMinConfidence`
- `ChapterAnalysisService.ts` L244：`entry.aliasConfidence >= ANALYSIS_PIPELINE_CONFIG.aliasRegistryMinConfidence`
- `ChapterAnalysisService.ts` L714：`r.confidence >= ANALYSIS_PIPELINE_CONFIG.aliasRegistryMinConfidence`

**当前造成的问题**：
- 阈值调高的直接效果：confidence 在 [0.7, 0.75) 区间的别名映射不再被采纳
- 积极效果：减少低置信度误归并
- 潜在负面效果：部分正确的别名映射被拒绝，导致同一人物以不同名称创建多个 Persona

**核心矛盾**：目前无任何指标可以量化"误归并减少了多少"和"漏归并增加了多少"

#### 1.4.2 必要性分析

**为什么值得做**：
- 阈值选择是影响准确率的核心参数之一
- 当前是凭经验从 0.7 调到 0.75，没有数据支撑
- 如果漏归并增加的损害超过误归并减少的收益，这个调整反而是负面的

**不做的代价**：
- 无法判断 0.75 是否为最优值
- 无法回答"应该调回 0.7 还是进一步调到 0.8"
- 准确率变化完全不可观测

**主要影响**：正确性（数据驱动决策能力）

#### 1.4.3 收益评估

| 指标 | 评估 |
|------|------|
| 性能影响 | 无（仅增加日志） |
| Token 影响 | 无 |
| 代码改动量 | 小（~30 行日志代码） |
| 决策质量提升 | 高——从"猜测"变为"数据驱动" |

监控本身不产生直接性能/token 收益，但为后续阈值调优提供数据基础。

#### 1.4.4 设计方案

**最小可行监控方案**：在现有 `console.info/warn` 日志中增加结构化数据点。

**修改文件**：
1. `src/server/modules/analysis/services/PersonaResolver.ts` — 别名查找路径日志
2. `src/server/modules/analysis/services/ChapterAnalysisService.ts` — roster 别名注册路径日志

**日志埋点设计**：

**埋点 1：PersonaResolver.ts Step 2.5（L252-269）**

```typescript
// 在 aliasResult 获取后、阈值判断前，记录所有 alias 查找结果
if (aliasResult) {
  console.info(
    "[PersonaResolver] alias.lookup.result",
    JSON.stringify({
      bookId       : input.bookId,
      extractedName: input.extractedName,
      confidence   : aliasResult.confidence,
      personaId    : aliasResult.personaId,

    })
  );
}
```

**埋点 2：ChapterAnalysisService.ts roster 别名注册（L241-258）**

```typescript
}
```

**关键监控指标**：

| 指标 | 含义 | 采集方式 |
|------|------|---------|
| `alias.lookup.result` 中 `inOldThresholdGap = true` 的次数 | 被新阈值拒绝但旧阈值会接受的别名数 | 日志计数 |
| `alias.lookup.result` 中 `accepted = false` 的后续行为 | 被拒绝后是否创建了新 Persona（漏归并） | 关联同一 extractedName 的后续 `resolve.result` 日志 |
| 全书验证 DUPLICATE_PERSONA issue 数量 | 间接反映漏归并程度 | 验证报告 summary |
| `alias.registration.filtered_by_threshold` 的次数 | roster 阶段被过滤的别名映射数 | 日志计数 |

**如何识别"误归并下降但漏归并上升"**：
- 如果 `inOldThresholdGap = true` 频繁出现，且对应的 `extractedName` 最终创建了新 Persona（而非合并到已有），则说明漏归并增加
- 如果 DUPLICATE_PERSONA issue 增加，但 `inOldThresholdGap` 为 0，说明问题与阈值无关

- **当前阶段不需要**。先通过日志采集 1-2 本书的数据，评估 `inOldThresholdGap` 的频率
- 如果频率 < 5%（即 [0.7, 0.75) 区间的别名极少），则 0.75 是安全的
- 如果频率 > 15%，则建议回退到 0.7 或做更精细的 A/B

**Schema 变更**：无
**迁移脚本**：无
**前端兼容**：无影响

#### 1.4.5 风险与副作用

**风险**：极低（仅增加日志，不影响任何业务逻辑）
**副作用**：日志量轻微增加（每次 alias 查找 +1 条日志）
**回滚策略**：删除日志代码即可

#### 1.4.6 测试与验收

**单元测试**：无需新增（日志不影响业务逻辑）
**验收标准**：
- 手动运行一本书的解析，检查日志中出现 `alias.lookup.result` 和 `alias.registration.filtered_by_threshold`

## 附录：四个关键问题的深度回答

### Q1：章节验证改为条件触发，会不会影响准确率？

**结论：不会显著影响，可安全实施。**

**安全跳过验证的条件**：
- `created.personas < 3 AND hallucinationCount < 2` —— 说明本章主要是已有人物复现，AI 提取质量正常
- 这类章节的验证结果几乎总是空 issue 列表（已由 PersonaResolver 保障了实体对齐质量）

**推荐多条件触发**（OR 关系）：
- `created.personas >= 3` —— 新建人物多，值得复审
- `hallucinationCount >= 2` —— AI 幻觉多，质量可疑

**跳过时的状态记录**：
- 不需要记录到数据库（避免引入新字段）
- 通过结构化日志 `chapter.validation.skipped` 记录跳过原因，便于离线审计
- 章节状态直接写 `SUCCEEDED`（跳过验证 ≠ 有问题，只是确定性足够高不需要 AI 二次确认）

**兜底**：Phase 6 全书验证仍然覆盖所有章节的人物汇总，是最终安全网。

### Q2：Chunk profiles 过滤，怎样做才不会误伤召回？

**结论：roster 关联 + 兜底补充是最佳策略。**

**仅靠 roster 是否足够**：
- 90%+ 场景下足够。Phase 1 Roster Discovery 读完整章节文本，理论上能识别所有出场人物
- 不足场景：roster AI 漏识别了某个罕见称谓（如仅以代词或隐喻提及的人物）

**推荐策略**：roster 关联优先 + 按创建时间兜底补充
1. 将 roster 中 `entityId` 映射到的所有 personaId 对应的 profiles 纳入
2. 将 roster 中 `suggestedRealName` 匹配到的 profiles 也纳入
3. 若不足 `chunkProfileMaxCount`，按 profile 创建顺序（早期 = 核心人物）补充
4. 不建议引入"最近章节活跃人物"策略——因为 profiles 是在每章开始时一次性加载的，无法获取"最近活跃"信息，且会增加复杂度

**兜底机制**：
- `chunkProfileMaxCount` 默认 25——即使 roster 只命中 10 个，仍补充 15 个早期核心人物
- `PersonaResolver.resolve()` 的 `multiSignalScore` 仍然基于全量 BookPersonaCache，不受 chunk profiles 影响——这是最终兜底

### Q3：PENDING → NEEDS_REVIEW，是否值得做 schema 变更？

**结论：不需要 schema 变更。直接使用 `"NEEDS_REVIEW"` 字符串。**

**理由**：
- `parseStatus` 是 `String @default("PENDING")`（非 Enum），可以直接写入任意字符串
- 无需 Prisma migration、无需修改 DB schema、无需做数据迁移
- `reviewFlag` 方案需要 `ALTER TABLE ADD COLUMN`，成本远高于直接用字符串值

**前后端兼容**：
- 后端：仅在 `runAnalysisJob.ts` 一处写入该值，改一行代码
- 前端：在 `ChapterStatusBadge` 新增一个 `if` 分支（~8 行），在 `canReanalyze` 新增一个条件（~1 行）
- API 层：`books.ts` 中 `parseStatus: string` 类型定义已兼容任意字符串

**推荐方案 A（直接用字符串）** 而非方案 B（新增字段）的原因：
- 改动量：3 文件 ~20 行 vs 5+ 文件 + migration + 兼容处理
- 风险：几乎为零 vs 需要处理 nullable 字段默认值
- 效果：语义完全等价

### Q4：aliasRegistryMinConfidence = 0.75 应该如何监控？

**结论：结构化日志 + 手动分析即为最小可行方案。**

**最有效的监控指标**：
1. **`inOldThresholdGap` 频率**：[0.7, 0.75) 区间的 alias 查找被拒绝的次数占总 alias 查找的比例
2. **拒绝后创建新 Persona 的比例**：被拒绝的 alias 最终是否通过 `multiSignalScore` 合并（好）还是创建新 Persona（可能是漏归并）
3. **DUPLICATE_PERSONA issue 数量**：全书验证中发现的重复人物问题

**识别"误归并下降但漏归并上升"的方法**：
