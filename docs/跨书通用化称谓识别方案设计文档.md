# 跨书通用化称谓识别与关系阻断方案设计文档

> 基于 wen-yuan 当前人物解析链路的工程级通用化方案。面向 Codex 可执行。
>
> **v2 修订说明**：根据人工审查反馈，对以下 5 处做了实质性修正：
> 1. Phase 1 不做 schema 变更，先走运行时配置注入
> 2. Evidence scoring 从加权评分模型简化为分档规则门控
> 3. Soft block suffix 从固定 0.15 改为 `baseScore × penalty` 相对降权
> 4. AI 仲裁结果从直接写入 `CONFIRMED` 改为写入 `LLM_INFERRED`，需二次验证才升级
> 5. AI 仲裁从"可能每章触发"统一为"全书末尾批量一次"，不进入主链路

---

## 0. 执行摘要

- **核心问题**：当前 `GENERIC_TITLES`（40+ 词）、`RELATIONAL_SUFFIXES`（21 词）、`TITLE_PATTERN`（9 匹配项）、`POSITION_PATTERN`（12 匹配项）全部写死在 `PersonaResolver.ts` 中，对明清官场小说（如《儒林外史》）高度适配，但对武侠、仙侠、现代、家族、宗教等类型作品存在明确的跨书误判风险
- **最严重的风险**：`GENERIC_TITLES` 误杀——某个词在 A 书中是泛化称谓，在 B 书中是稳定指代某一具体人物的唯一称呼（如"掌门"在武侠小说中常指代特定角色）
- **次严重的风险**：`TITLE_PATTERN` / `POSITION_PATTERN` 不适配——硬编码了明清官制，对唐宋、春秋、武侠门派、宗教体系无覆盖
- **推荐方向**：不一次性去规则化，而是采用"**收缩硬编码 → 运行时配置化 → 分档 evidence 门控 → AI 全书末仲裁**"四层渐进方案
- **保留硬编码的部分**：人称代词（他/她/众人等）、极高置信泛指词（此人/旁人/某人等）——这些在任何书籍类型中都不应建实体
- **引入 AI 的部分**：仅在全书分析完成后，对灰区称谓做一次批量后处理仲裁，不进入章节主链路
- **Phase 1 不改 schema**：书级配置先通过运行时注入（`pipeline.ts` 内置预设 + 按 bookId 可选覆盖），schema 变更推迟到方案稳定后
- **推荐分 4 期落地**：Phase 1 收缩 + 配置化 → Phase 2 分档 evidence 门控 → Phase 3 AI 全书末仲裁 → Phase 4 数据驱动迭代
- **Phase 1 即可独立上线**，风险极低，**无 schema 变更**，预计 2-3 天工作量
- **总预期收益**：跨书误杀率从 estimated ~5-15% 降至 ~1-3%（中性估计），新书适配成本从"修改代码"降至"填写配置"

## 1. 问题定义：为什么当前写死规则存在跨书风险

### 1.1 `GENERIC_TITLES`

**定义位置**：`PersonaResolver.ts` L14-30，`export const`，40+ 词

**当前在《儒林外史》中的作用**：
- 阻止"老爷""夫人""掌柜""众人""丫鬟"等无定指称谓创建伪实体
- Step 1 精确匹配 → 直接标记 `hallucinated`，confidence = 1.0，不触发任何 DB 查询
- 同时通过 `GENERIC_TITLES_EXAMPLE` 注入 Roster Discovery 和 Chapter Analysis 的 prompt，指导 AI 标记 `generic: true`

**跨书风险分析**：

| 当前词条 | 在《儒林外史》中 | 在其他书中可能 | 风险类型 | 严重度 |
|---------|---------------|-------------|---------|-------|
| "先生" | 泛指教书先生 | 武侠小说中可能是某角色唯一称呼（如"先生"指诸葛亮） | **误杀真实人物** | 🔴 高 |
| "掌柜" | 泛指店铺老板 | 在某些故事中可能是核心角色的固定称呼 | **误杀真实人物** | 🔴 高 |
| "员外" | 泛称富人 | 水浒传中"员外"可能指特定角色（如卢俊义常被称为"员外"） | **误杀真实人物** | 🔴 高 |
| "举人" | 泛称科举身份 | 在《儒林外史》本身，"范举人"中的"举人"是后缀而非独立实体 | 当前正确处理 | ✅ 安全 |
| "众人" "旁人" "某人" | 泛指 | 任何书中都是泛指 | 当前正确处理 | ✅ 安全 |
| "父亲" "母亲" "兄弟" | 泛指亲属 | 任何书中都不应作为独立实体 | 当前正确处理 | ✅ 安全 |
| "将军" | 泛指 | 某些书中"将军"可能稳定指代一个特定角色 | **误杀真实人物** | 🟡 中 |
| "秀才" "进士" "状元" | 泛指科举身份 | 某些作品中可能是角色代称 | **误杀真实人物** | 🟡 中 |
| "公子" "少爷" | 泛指 | 在某些作品中"公子"可能是主角的固定称呼 | **误杀真实人物** | 🟡 中 |

**最严重的问题**：**误杀率随书籍类型变化而不可预测**。一旦误杀，该人物在全书所有章节中都会被标记为幻觉，完全从分析结果中消失，且无法通过后续验证恢复（因为验证也看不到这个人物的任何数据）。

### 1.2 `RELATIONAL_SUFFIXES`

**定义位置**：`PersonaResolver.ts` L36-41，private，21 词

**当前作用**：
- 在 `scorePair()` 的子串匹配阶段，若较长名 = 较短名 + 关系后缀（如"蘧公孙父亲" = "蘧公孙" + "父亲"），则返回 score = 0，阻止合并
- 保护了"A" 和 "A的亲属" 不被错误归并

**跨书风险分析**：

| 当前后缀 | 作为阻断信号时正确 | 可能不正确的场景 | 风险类型 |
|---------|---------------|---------------|---------|
| "大人" | "张大人" ≠ "张" → 正确阻断 | "李大人"可能就是"李四"的稳定别名 | **漏召回**（应归并但被阻断） |
| "先生" | "何先生" ≠ "何" → 正确阻断 | 但 "何先生" 和 "何美之" 可能是同一人 | **漏召回** |
| "将军" | "赵将军" ≠ "赵" → 正确阻断 | "赵将军" 和 "赵云" 是同一人 | **漏召回** |
| "夫人" | "王夫人" ≠ "王" → 正确阻断 | "王夫人" 是《红楼梦》中的独立角色 | **错误归并**或**漏召回** |
| "娘子" | "杨娘子" ≠ "杨" → 正确阻断 | "杨娘子"可能就是"杨氏"的稳定别名 | **漏召回** |
| "老爷" | "范老爷" ≠ "范" → 正确 | 但"范老爷"和"范进"应归并 | 当前由 AliasRegistry 处理 |

**核心矛盾**：当前是一刀切 `return 0`（硬阻断），没有区分"强阻断"（父亲/母亲/儿子/女儿）和"弱阻断"（大人/先生/将军/夫人——这些既可能是关系词，也可能是稳定别名后缀）。

**注意**：RELATIONAL_SUFFIXES 和 GENERIC_TITLES 有交叉——"老爷""先生""夫人""娘子"同时出现在两个集合中，语义不同：
- 在 GENERIC_TITLES 中：单独出现时标记为幻觉
- 在 RELATIONAL_SUFFIXES 中：作为已有人名的后缀时阻止合并

### 1.3 `TITLE_PATTERN`

**定义位置**：`PersonaResolver.ts` L43
**正则**：`/(皇帝|太后|太祖|太宗|吴王|国公|侯|伯|王)$/`

**当前作用**：
- 在 `inferAliasType()` 中，若人名以这些词结尾 → `aliasType = "TITLE"`
- 在 Step 5 创建新 Persona 时，若匹配 → 触发 `aliasRegistry.registerAlias()` 注册为别名

**跨书风险**：

| 问题 | 说明 |
|------|------|
| **硬编码"吴王"** | "吴王"是《儒林外史》时代背景，换到唐代应为"秦王""齐王"等，换到武侠无此概念 |
| **单字"王""侯""伯"** | 过于宽泛。"王"可能是姓氏（"王进"的"王"），"侯"可能是"侯方域"的姓氏 |
| **遗漏大量头衔体系** | 无佛教（方丈/住持/法师）、无道教（真人/天师）、无武侠（掌门/盟主/帮主）、无春秋官制（司徒/司空/司马）、无近代（总统/主席/总理） |
| **结尾匹配 `$`** | "王大人"会匹配"王"→ 误分类为 TITLE（实际是姓 + 尊称） |

**最严重**：单字"王""侯""伯"的误匹配风险，可能将姓氏误判为头衔。

### 1.4 `POSITION_PATTERN`

**定义位置**：`PersonaResolver.ts` L44
**正则**：`/(丞相|太守|知府|知县|将军|尚书|侍郎|巡抚|总督|学道|老爷|先生)$/`

**跨书风险**：与 TITLE_PATTERN 类似，但更严重：

| 问题 | 说明 |
|------|------|
| **"老爷"和"先生"在此列** | 几乎所有 "X老爷" "X先生" 都会匹配，触发 alias 注册——但这些可能只是尊称，并非真正的"官职称号" |
| **明清专属** | "丞相"是秦汉制，"巡抚""总督"是明清制，"学道"是特定时期称呼 |
| **遗漏** | 无"县令""刺史""节度使""都御史"等常见官职 |

## 2. 规则分类：哪些应该"保留硬编码"，哪些应该"通用化"

### 2.1 应长期保留的"底层安全规则"（Safety Tier）

**原则**：只保留在**任何中文文学类型中都不应创建实体**的词条。

**建议保留的最小稳定集合**：

```typescript
// ===== 人称代词 / 泛指代词（跨书籍类型绝对安全） =====
"此人", "那人", "来人", "众人", "旁人", "大家", "诸人", "某人", "一人",
"他", "她", "他们", "她们", "吾", "汝", "彼", "尔",
// ===== 亲属泛称（无专名前缀时绝对安全） =====
"父亲", "母亲", "老父", "老母", "老娘", "娘亲",
"兄长", "兄弟", "姐姐", "弟弟", "妹妹", "妻子",
// ===== 仆役泛称（跨书籍类型极高安全度） =====
"丫鬟", "丫头", "奴婢", "仆人", "仆役", "家丁", "下人", "小厮", "书童"
```

**从 Safety Tier 中移出的词条**（共 ~17 个需迁移到配置层）：

```
"老爷", "夫人", "太太", "老太太", "小姐", "少爷", "公子", "相公", "娘子", "先生",
"掌柜", "掌柜的", "账房", "管家", "老管家", "门房", "门子",
"书办", "掌舵", "按察司", "布政司", "都司", "参将", "千总", "把总",
"员外", "举人", "秀才", "进士", "状元", "老学究"
```

**理由**：这些词在特定书籍类型（武侠/仙侠/家族/宫廷）中可能稳定指代具体人物。

### 2.2 应改成"书级配置化"的规则（Config Tier）

**应迁移的词条**：上述从 Safety Tier 移出的 ~17 个称谓词 + 全部 TITLE_PATTERN / POSITION_PATTERN 中的词条

**配置结构设计**：

```typescript
interface BookLexiconConfig {
  /** 书级泛化称谓追加集（与全局 Safety 合并后用于幻觉过滤） */
  additionalGenericTitles: string[];

  /** 书级泛化称谓移除集（从全局默认列表中豁免，允许作为人物名） */
  exemptGenericTitles: string[];

  /** 书级关系后缀追加集 */
  additionalRelationalSuffixes: string[];

  /** 书级关系后缀降级为弱阻断（仅扣分不硬阻断） */
  softRelationalSuffixes: string[];

  /** 书级头衔正则追加（如"掌门|盟主|帮主"） */
  additionalTitlePatterns: string[];

  /** 书级官职正则追加（如"节度使|都御史|刺史"） */
  additionalPositionPatterns: string[];
}
```

**加载位置**（Phase 1 采用运行时注入，不改 schema）：
- 在 `pipeline.ts` 中内置书籍类型预设（`GENRE_PRESETS: Record<string, BookLexiconConfig>`），默认预设 = "明清官场"
- 在 `analyzeChapter()` 启动时，根据 bookId 查找预设（或使用默认），合并为 effective 集合
- 通过参数传入 `PersonaResolver` 的 `resolve()` 方法
- **后续稳定后**：可选迁移到 Book 模型新增 `lexiconConfig Json?` 字段持久化

**合并策略**：

```
effectiveGenericTitles = (SAFETY_GENERIC_TITLES ∪ book.additionalGenericTitles) - book.exemptGenericTitles
effectiveRelationalSuffixes = DEFAULT_RELATIONAL ∪ book.additionalRelationalSuffixes
effectiveTitlePattern = buildRegex(DEFAULT_TITLES ∪ book.additionalTitlePatterns)
```

### 2.3 应改成"动态判定"的规则（Dynamic Tier）

**需要动态判定的场景**：

| 场景 | 当前行为 | 应改为 | 所需信号 |
|------|---------|-------|---------|
| "老爷"在全书中是否总是指代同一人 | 一刀切标记为幻觉 | 如果 AliasMapping 中"老爷"高置信绑定了某 Persona → 应视为该人的稳定别名 | AliasMapping 表中的 confidence + 绑定稳定性 |
| "范先生" 中的"先生"是否应阻断合并 | RELATIONAL_SUFFIXES 硬阻断 → score=0 | 如果 AliasRegistry 中"范先生"已注册为"范进"的别名 → 应允许合并 | AliasRegistry 查找结果 |
| "太子"是否是 TITLE_ONLY | 不匹配 TITLE_PATTERN → 不注册 alias | 应检查是否符合头衔模式 | 上下文 + 书级词典 |

**动态判定所需信号**：
1. **AliasMapping 绑定稳定性**：该称谓在 AliasMapping 表中是否高置信（≥ 0.75）绑定到某个 Persona
2. **跨章节共现一致性**：该称谓在多个章节中是否始终与同一 Persona 共现
3. **独立出现时的指向稳定性**：当该称谓单独出现（无姓氏前缀）时，前后文中最近的具名人物是否总是同一人

## 3. 推荐的通用化方案：四层分层机制

### 3.1 底层兜底规则层（Hard Safety Rules）

```
输入: extractedName (String)
输出: "hallucinated" | "pass"

规则: SAFETY_GENERIC_TITLES.has(name.trim())
  → true: return "hallucinated", confidence=1.0, reason="safety_generic"
  → false: pass to next layer
```

**保留范围**：仅 ~26 个跨书籍类型绝对安全的词条（代词 + 亲属泛称 + 仆役泛称）
**禁止扩张的边界**：不得在此层添加任何可能在某个书籍类型中作为人物名的词条
**硬编码位置**：继续保留在 `PersonaResolver.ts` 中，但重命名为 `SAFETY_GENERIC_TITLES`

### 3.2 书级配置层（Book-level Lexicon / Policy）

```
输入: extractedName, bookLexicon (BookLexiconConfig)
输出: "hallucinated" | "title_only" | "pass"

规则:
  1. effectiveGenericTitles = merge(SAFETY, bookLexicon)
  2. if effectiveGenericTitles.has(name) → "hallucinated"
  3. if effectiveTitlePattern.test(name) → flag as "title_only"
  4. else → "pass"
```

**默认配置**：提供一套"明清官场小说"默认配置（即当前硬编码的词条），新书不填时使用默认
**覆盖机制**：书级 `additionalGenericTitles` 追加 + `exemptGenericTitles` 豁免
**避免配置失控**：
- 配置项有最大长度限制（如每个数组最多 50 项）
- 后台 UI 提供配置编辑页面，而非直接改 JSON
- 配置变更记日志，支持回溯

### 3.3 统计与上下文判定层（Heuristic / Evidence Layer）

**核心设计**：`MentionPersonalizationTier`——判断一个称谓是否在本书中已经"人格化"为稳定指代。

> **v2 修订**：原方案使用加权评分（0.4/0.2/0.1 权重），但权重未经数据校准，存在"拍脑袋"风险。现改为**分档规则门控**，更易解释、更易调试。

**分档判定模型**：

```typescript
interface MentionPersonalizationEvidence {
  /** 称谓文本 */
  surfaceForm: string;
  /** 是否存在高置信 AliasMapping (≥ 0.75) 绑定到唯一 Persona */
  hasStableAliasBinding: boolean;
  /** 在已处理章节中出现次数 */
  chapterAppearanceCount: number;
  /** 在已处理章节中是否始终指向同一 Persona */
  singlePersonaConsistency: boolean;
  /** Roster Discovery 中被标记为 generic 的次数 vs 非 generic 次数 */
  genericRatio: number;
}

type PersonalizationTier = "personalized" | "generic" | "gray_zone";

function classifyPersonalization(e: MentionPersonalizationEvidence): PersonalizationTier {
  // ===== Tier 1: 已人格化 =====
  // 有高置信别名绑定 且 跨章出现一致
  if (e.hasStableAliasBinding && e.singlePersonaConsistency) {
    return "personalized";
  }

  // ===== Tier 2: 确认泛化 =====
  // 完全无绑定 且 大部分章节 AI 标记为 generic
  if (!e.hasStableAliasBinding && e.genericRatio >= 0.7) {
    return "generic";
  }

  // ===== Tier 3: 灰区 =====
  // 其余所有情况 → 触发全书末 AI 仲裁（Phase 3）
  return "gray_zone";
}
```

**判定规则说明**：

| 档位 | 条件 | 行为 |
|------|------|------|
| **personalized** | 有高置信 alias binding 且跨章指向同一 Persona | 豁免 generic 过滤，走正常 resolve |
| **generic** | 无 alias binding 且 genericRatio ≥ 0.7 | 按幻觉处理 |
| **gray_zone** | 其他所有情况 | 先按 hallucinated(confidence=0.5) 暂处理，全书末批量 AI 仲裁 |

**优势**（相比原加权评分）：
- 不需要校准权重，每个档位的条件都可解释
- 调试时只需看"命中了哪个档位的哪个条件"，而非追踪分数累加
- 新增信号只需调整档位条件，不影响其他档位
- 冷启动期间（前几章无绑定数据时）自然落入 gray_zone，不会误判

**数据来源**：
- `hasStableAliasBinding`：查询 `AliasMapping` 表，`WHERE alias = surfaceForm AND bookId = X AND confidence >= 0.75 AND status IN ('CONFIRMED', 'LLM_INFERRED')`
- `chapterAppearanceCount`：查询 `Mention` 表 `WHERE rawText LIKE '%surfaceForm%' AND bookId = X`，按 chapterId 去重
- `singlePersonaConsistency`：查询 `Mention` 表，检查绑定的 personaId 是否唯一
- `genericRatio`：在 Roster 处理日志中累计（需新增运行时统计）

**执行时机**：不在每个 mention 上运行。仅在以下场景触发：
1. Step 0.5 中命中 effectiveGenericTitles 时（被命中后检查是否应豁免）
2. `RELATIONAL_SUFFIXES` 命中后检查是否应降级为弱阻断

### 3.4 AI 判定层（LLM Arbitration Layer）

**触发条件**：仅当 Evidence Layer 返回 `gray_zone`

> **v2 修订**：
> - 调用时机从"每章结束后"统一为"**全书分析完成后，一次性批量仲裁**"
> - 结果写入状态从 `CONFIRMED` 改为 `LLM_INFERRED`（需二次验证才可升级为 CONFIRMED）
> - AI 仅作为后处理建议层，不影响当前章节主链路

**调用方式**：全书分析完成后，在 `resolvePersonaTitles()` 和 `markOrphanPersonas()` 之后，收集全书所有 `gray_zone` 称谓一次性提交。

**Prompt 设计**（约 200 token 指令 + 动态数据）：

```
## 任务
判断以下称谓在本书中是否已"人格化"为特定人物的稳定称呼。

## 书籍: ${bookTitle}
## 待判定称谓:
${grayZoneTerms.map(t => `- "${t.surfaceForm}" (出现${t.chapterAppearanceCount}章, AliasMapping: ${t.hasStableAliasBinding ? '有' : '无'})`)}

## 输出格式 (JSON):
[{ "surfaceForm": "...", "isPersonalized": true/false, "confidence": 0.0-1.0, "reason": "..." }]
```

**成本控制**：
- **全书仅一次调用**（不再每章触发）
- 每次最多 20 个称谓（超出则取 evidence 最不确定的前 20 个）
- 预计每次调用 ~800-1500 token
- 同一称谓的判定结果缓存到 AliasMapping 表

**安全约束**：AI 判定结果不得覆盖 Safety Tier 规则。即使 AI 说"众人"是人格化称谓，Safety Tier 仍硬阻断。

**结果写回**：

> **v2 修订**：AI 判定不直接写入 CONFIRMED，避免单次 AI 误判被放大为"稳定事实"。

- `isPersonalized = true && confidence >= 0.7` → 写入 AliasMapping 表，status = **`LLM_INFERRED`**
- `isPersonalized = true && confidence < 0.7` → 写入 AliasMapping 表，status = `PENDING`
- `isPersonalized = false` → 不写入（维持 generic 处理）

**LLM_INFERRED → CONFIRMED 升级条件**（满足任一即可）：
1. 后续章节 evidence 重复支持（再次跑到 personalized 档位）
2. 验证阶段（validateBookResult）通过
3. AliasRegistry 再次命中（不同章节独立生成相同映射）
4. 人工 review 通过

## 4. 针对当前 4 类写死规则的替代策略

### 4.1 `GENERIC_TITLES` 的替代方案

**保留极小全局停用集（Safety Tier）**：26 个词（代词 + 亲属 + 仆役）

**迁出到 Config Tier**：~17 个词（称谓 + 官职 + 科举身份）

**人格化判定机制**：
1. 当 Step 0.5 命中 effective generic list 时，先检查 evidence 分档
2. `personalized` 档 → **豁免**，不标记为幻觉，走正常 resolve 流程
3. `generic` 档 → 按幻觉处理
4. `gray_zone` 档 → 先按 hallucinated(confidence=0.5) 暂处理，全书末 AI 仲裁

**代码变更**：

```typescript
// PersonaResolver.ts resolve() Step 1 改为：
if (SAFETY_GENERIC_TITLES.has(name)) {
  return { status: "hallucinated", confidence: 1.0, reason: "safety_generic" };
}

if (effectiveGenericTitles.has(name)) {
  // 新增：分档判定是否已人格化
  const evidence = await collectPersonalizationEvidence(name, input.bookId);
  const tier = classifyPersonalization(evidence);
  if (tier === "personalized") {
    // 已人格化 → 豁免，继续走 resolve 流程
  } else if (tier === "generic") {
    return { status: "hallucinated", confidence: 0.9, reason: "config_generic" };
  } else {
    // gray_zone → 标记为待仲裁，先按 hallucinated 处理，全书末批量 AI 仲裁
    pendingArbitrations.push({ surfaceForm: name, evidence });
    return { status: "hallucinated", confidence: 0.5, reason: "pending_arbitration" };
  }
}
```

### 4.2 `RELATIONAL_SUFFIXES` 的替代方案

**三级机制设计**：

```typescript
/** 强阻断：任何情况下都不应合并（亲子关系词） */
const HARD_BLOCK_SUFFIXES = new Set([
  "父亲", "母亲", "儿子", "女儿", "之妻", "之子", "之父", "之母",
  "老爹", "老娘"
]);

/** 弱阻断：默认阻断但可被 AliasMapping 覆盖 */
const SOFT_BLOCK_SUFFIXES = new Set([
  "大人", "将军", "老爷", "先生", "娘子",
  "太太", "夫人", "兄弟", "兄长", "弟弟", "姐姐", "妹妹"
]);
```

**scorePair 改造**：

> **v2 修订**：原方案 soft block 返回固定 0.15，在 threshold=0.72 下实际等于"准硬阻断"。现改为 `normalSubstringScore × softBlockPenalty`，实现真正的"降权而不是判死刑"。

```typescript
// scorePair() 子串匹配阶段改为：
if (a.includes(b)) {
  const tail = a.slice(a.indexOf(b) + b.length);
  if (tail && HARD_BLOCK_SUFFIXES.has(tail)) return 0;           // 硬阻断
  const normalScore = 0.60 + 0.37 * (b.length / a.length);       // 正常子串分
  if (tail && SOFT_BLOCK_SUFFIXES.has(tail)) {
    return normalScore * SOFT_BLOCK_PENALTY;                      // 相对降权
  }
  if (tail && bookSoftSuffixes?.has(tail)) {
    return normalScore * SOFT_BLOCK_PENALTY;                      // 书级弱阻断
  }
  return normalScore;
}
```

**`SOFT_BLOCK_PENALTY` 建议值**：`0.4`（配置化，定义在 `pipeline.ts`）

**效果示例**：

| 名字对 | 正常子串分 | × 0.4 penalty | 是否超过 0.72 阈值 | 行为 |
|--------|----------|---------------|-------------------|------|
| "赵将军" vs "赵" (1/3) | 0.72 | 0.29 | ❌ 不合并 | 正确：阻断了单姓匹配 |
| "范老爷" vs "范" (1/3) | 0.72 | 0.29 | ❌ 不合并 | 正确：但 AliasRegistry 可覆盖 |
| "王夫人" vs "王" (1/3) | 0.72 | 0.29 | ❌ 不合并 | 正确：AliasRegistry 可覆盖 |

**核心改进**：
- 不是一刀切写死分数，而是基于实际子串匹配质量进行相对降权
- AliasRegistry 若有高置信映射（Step 2.5），仍可在 scorePair 之前拦截并正确归并
- 降权系数可调参（0.3~0.5 区间），比固定数值更灵活

### 4.3 `TITLE_PATTERN` / `POSITION_PATTERN` 的替代方案

**拆为三层**：

/** 通用头衔层：跨书籍类型基础头衔词 */

## 6. 是否值得引入 AI？

### 6.1 AI 适合放在哪些环节

| 环节 | 理由 | 预计 token |
|------|------|-----------|
| 称谓人格化判定（灰区） | Evidence score 在 [0.3, 0.6) 时，规则无法确定，需要理解书籍上下文 | ~500-800/次 |
| 关系后缀语义仲裁 | "X夫人"到底是"X的夫人"还是一个独立角色 | ~300-500/次 |
| 新书开分析前的"书籍类型识别" | 自动识别书的时代/书籍类型，推荐默认词典配置 | ~200-400/次（仅一次） |

### 6.2 AI 不适合放在哪些环节

- ❌ 每个 mention 都调用（成本不可控）
- ❌ Safety Tier 代词/泛指过滤（100% 确定，无需 AI）
- ❌ HARD_BLOCK_SUFFIXES 阻断（亲子关系词误判率 < 0.1%，AI 不会更好）

### 6.3 成本控制策略

1. **灰区触发**：仅 `gray_zone` 分档结果才进入 AI 仲裁
2. **全书末批量判定**：全书分析完成后一次性提交所有灰区称谓（而非每章调用）
3. **缓存复用**：判定结果写入 AliasMapping（status=`LLM_INFERRED`），后续重跑可直接读取
4. **数量上限**：每次最多 20 个称谓
5. **预计成本**：1 次 × ~1200 token ≈ 1,200 token/本书 ≈ 全书总 token 的 < 0.1%

**为什么全书末批量优于每章调用**：
- 不打断主流程
- 不引入每章额外延迟
- 不污染当前解析结果
- 容易 A/B 比较
- 数据更充分（全书 evidence 比前几章更可靠）

### 6.4 推荐的最小可行 AI 方案（MVP）

**Phase 3 MVP**：

> **v2 修订**：统一为全书末尾一次性批量调用。AI 仅作为后处理建议层，结果写入 `LLM_INFERRED` 状态，不直接影响当前分析结果。

在 `markOrphanPersonas()` + `resolvePersonaTitles()` 完成后（全书后处理阶段），收集全书所有仍处于 `hallucinated(reason="gray_zone")` 的称谓，一次性提交 AI 判定：

```
"以下称谓在《{bookTitle}》中是否指代特定人物？请基于上下文判断。"
+ 称谓列表 + 每个称谓的 evidence 摘要
→ AI 返回 [{surfaceForm, isPersonalized, confidence}]
→ isPersonalized + confidence >= 0.7 → 写入 AliasMapping (status = LLM_INFERRED)
→ 结果仅影响：后处理建议 / review 报告 / 下一轮优化
→ 不回溯修改已写入的 Persona / Mention 数据
```

**仅一次调用**（全书级别），token ~800-1500，覆盖全部灰区称谓。
**不在每章循环中调用**，极低成本。
**不影响主链路**——结果仅用于后处理和下轮优化。

## 11. 最终结论

### 1. 当前写死规则是否确实有跨书风险？

**是的，风险明确且可量化。**

- GENERIC_TITLES 中约 17/40 个词条（43%）在不同书籍类型中可能误杀真实人物
- TITLE_PATTERN / POSITION_PATTERN 仅覆盖明清官制，对其他朝代/书籍类型覆盖率不足 30%
- RELATIONAL_SUFFIXES 中约 11/21 个词条（52%）存在"既是关系词也是稳定别名后缀"的歧义
- 风险级别：**中-高**——不至于让系统完全不可用，但会导致每本新书需要大量人工修正

### 2. 是否应该"一次性去规则化"？

**绝对不应该。** 原因：
- Safety Tier（代词/亲属/仆役泛称）在任何书籍类型中都是正确的硬过滤，去掉会引入大量伪实体
- Evidence scoring 和 AI 仲裁需要逐步验证效果，贸然全面切换风险不可控
- 当前系统已在《儒林外史》上验证过，保持向后兼容是基本要求

### 3. 最推荐的路线？

**"收缩 → 配置 → 证据 → 仲裁"四层渐进方案。**
- Phase 1 是零风险改动（feature flag off 时行为完全等价），立即实施
- Phase 2 引入 evidence scoring，可通过 feature flag 控制
- Phase 3 仅在全书末尾对灰区做一次批量 AI 仲裁，成本极低（~1,200 token/书）
- 每期可独立上线、独立回滚

### 4. 哪些硬编码必须保留？

**SAFETY_GENERIC_TITLES**（~26 词）和 **HARD_BLOCK_SUFFIXES**（~10 词）必须保留为底层安全兜底。
这些词在任何中文文学类型中都不应创建实体（代词）或允许合并（亲子关系词）。

### 5. AI 应该扮演什么角色？

**仅用于全书末灰区仲裁（Post-processing Arbitration），不做主判定，不进入章节主链路。**
- Safety Tier 和 Config Tier 覆盖 ~80% 的判定
- Evidence 分档门控覆盖 ~15% 的判定
- AI 仅处理剩余 ~5% 的灰区边界案例（全书末一次性批量）
- AI 的判定结果写入 `LLM_INFERRED` 状态，需二次验证才升级为 `CONFIRMED`
- AI 永远不能覆盖 Safety Tier 规则
- AI 结果仅影响后处理建议、review 报告、下轮优化，不回溯修改已写入的数据
