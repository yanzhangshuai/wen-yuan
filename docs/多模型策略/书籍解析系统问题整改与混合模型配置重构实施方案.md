## 2. 总体目标与改造范围

### 2.1 改造目标

1. **混合模型按阶段可配置**：支持 Pipeline 每个阶段独立绑定 AI 模型，支持全局/Book/Job 三级配置覆盖
2. **全量问题修复**：修复选型报告中列出的全部 17 个问题
3. **Provider 抽象升级**：统一 system/user message 分离、参数透传、usage 返回
4. **可观测性建设**：token 计数、成本追踪、阶段级日志
5. **性能优化**：增大分片、提升并发、长章节保护

### 2.2 改造范围

#### 在范围内

| 模块 | 改造内容 |
|------|---------|
| 数据库 | 新增 model_strategy_configs 表、analysis_phase_logs 表；重构 Book/AnalysisJob 模型策略字段 |
| Provider 层 | 重构 AiProviderClient 接口（system/user 分离、参数化、usage 返回）；修复 DeepSeek max_tokens |
| Analysis 服务层 | 新增 ModelStrategyResolver；重构 ChapterAnalysisService（分片、合并、fallback）；重构 aiClient 接口 |
| Prompt 层 | 所有 Prompt 函数重构为返回 { system, user } 结构 |
| 前端 | 新增阶段模型配置 UI（Book 详情页 + 导入向导 Step3）|
| API 层 | 新增/修改模型策略相关 API |
| 测试 | 新增单元测试、集成测试覆盖所有变更 |

#### 明确不在范围内

- 旧数据迁移兼容逻辑
- 旧代码分支保留
- OCR 相关功能
- 新 Provider 接入（如 GLM-4，留作后续迭代）
- 前端主题系统改造
- Neo4j 图数据库相关改动

| 收益项 | 量化预期 |

| Phase2 调用成本 | 降低 60-70%（DeepSeek V3 替代高价模型 + 分片增大减少调用次数）|

## 3. 混合模型策略设计

### 3.1 阶段划分与职责

系统定义以下 Pipeline 阶段（枚举 `PipelineStage`）：

| 阶段标识 | 阶段名称 | 职责描述 | 调用频率 | 能力要求 | 推荐模型 |
|---------|---------|---------|---------|---------|---------|
| `ROSTER_DISCOVERY` | Phase1 名册发现 | 从完整章节中识别所有人物实体及其称谓类型 | 每章1次 | 长上下文 + 强NER + 中文古典文学理解 | 通义千问 Max |
| `CHUNK_EXTRACTION` | Phase2 分片提取 | 从文本分片中提取人物提及、生平事件、人物关系 | 每章多次（主要调用量） | 结构化抽取 + JSON格式遵循 | DeepSeek V3 |
| `CHAPTER_VALIDATION` | 章节验证 | 校验单章节提取结果的一致性和准确性 | 每章1次 | 逻辑推理 + 交叉校验 | 通义千问 Plus |
| `TITLE_RESOLUTION` | Phase5 称号溯源 | 推断 TITLE_ONLY 人物的历史真名 | 全书1-3次 | 历史知识 + 推理能力 | 通义千问 Max |
| `GRAY_ZONE_ARBITRATION` | Phase3 灰区仲裁 | 判断灰区称谓是否为人格化称呼 | 全书0-1次 | 分类判断 | 通义千问 Plus |
| `BOOK_VALIDATION` | 全书验证 | 全书级别的人物一致性和关系自洽性检查 | 全书1次 | 全局理解 + 逻辑推理 | 通义千问 Max |
| `FALLBACK` | 降级兜底 | 任何阶段主模型失败后的兜底模型 | 按需 | 综合能力 | 通义千问 Plus |

> **FALLBACK 语义澄清（v1.1）**
> 
> `FALLBACK` **不是业务阶段**，而是「全阶段共享的兜底模型配置槽位」。它的语义是：
> - 当任何业务阶段（6 个真实阶段）的主模型在重试耗尽后仍然失败时，系统使用 FALLBACK 槽位配置的模型进行最后一次尝试
> - FALLBACK **不参与正常解析统计**：`analysis_phase_logs` 中 `stage` 字段仍记录原始业务阶段（如 `CHUNK_EXTRACTION`），通过 `is_fallback=true` 标记区分
> - 前端展示时，FALLBACK 配置区域应与 6 个业务阶段视觉分隔，标注为「降级兜底模型（所有阶段共享）」
> - 在 `PipelineStage` 枚举中保留 `FALLBACK` 值仅用于配置层面的 key，不用于业务逻辑的 stage 判断
> - 未来如需"每阶段独立 fallback"，可在 `StageModelConfig` 中新增 `fallbackModelId` 字段，FALLBACK 槽位降级为"全局兜底"

### 3.3 配置优先级规则

四层配置，优先级从高到低：

```
优先级 1（最高）：AnalysisJob 级配置
  └─ 优先级 2：Book 级配置
       └─ 优先级 3：全局默认配置（model_strategy_configs 表 scope=GLOBAL）
            └─ 优先级 4：系统硬编码默认值（代码中的 DEFAULT_STRATEGY）
```

**合并策略**：按阶段逐字段合并（Merge Per Stage）

```typescript
// 伪代码：resolveModelForStage
function resolveModelForStage(stage: PipelineStage, context: { jobId?, bookId? }): ResolvedModel {
  // 1. 查 Job 级配置
  const jobConfig = context.jobId 
    ? loadStrategy({ scope: 'JOB', jobId: context.jobId }) 
    : null;
  if (jobConfig?.stages[stage]?.modelId) {
    const model = validateAndLoadModel(jobConfig.stages[stage].modelId);
    if (model) return { model, source: 'JOB', params: jobConfig.stages[stage].params };
  }

  // 2. 查 Book 级配置
  const bookConfig = context.bookId 
    ? loadStrategy({ scope: 'BOOK', bookId: context.bookId }) 
    : null;
  if (bookConfig?.stages[stage]?.modelId) {
    const model = validateAndLoadModel(bookConfig.stages[stage].modelId);
    if (model) return { model, source: 'BOOK', params: bookConfig.stages[stage].params };
  }

  // 3. 查全局配置
  const globalConfig = loadStrategy({ scope: 'GLOBAL' });
  if (globalConfig?.stages[stage]?.modelId) {
    const model = validateAndLoadModel(globalConfig.stages[stage].modelId);
    if (model) return { model, source: 'GLOBAL', params: globalConfig.stages[stage].params };
  }

  // 4. 系统硬编码默认：使用 isDefault=true 的模型
  const defaultModel = loadDefaultModel();
  if (defaultModel) return { model: defaultModel, source: 'SYSTEM_DEFAULT', params: DEFAULT_PARAMS };

  throw new Error(`阶段 ${stage} 无可用模型`);
}

// validateAndLoadModel：校验模型存在且 isEnabled=true
function validateAndLoadModel(modelId: string): AiModel | null {
  const model = db.aiModel.findUnique({ where: { id: modelId } });
  if (!model || !model.isEnabled) return null; // 跳过，降级到下一优先级
  return model;
}
```

**冲突处理**：
- 高优先级配置指定的模型如果已被禁用/删除，自动降级到下一优先级层
- 如果所有层都无有效模型，使用 FALLBACK 阶段配置的模型
- 如果 FALLBACK 也无效，使用全局 isDefault 模型
- 如果仍无可用模型，抛出明确错误，任务标记 FAILED

### 3.4 降级与 fallback 规则

```
AI 调用失败处理流程：
┌─────────────────┐
│ 调用主模型       │
└────────┬────────┘
         │ 失败
         ▼
┌─────────────────┐
│ 同模型重试       │ 最多 2 次，指数退避（600ms → 1200ms）
│ （可重试错误）    │ 可重试：429/rate_limit/timeout/econnreset/network
└────────┬────────┘
         │ 仍然失败
         ▼
┌─────────────────┐
│ 切换 fallback   │ 使用当前配置中 FALLBACK 阶段指定的模型
│ 模型重试         │ 重试 1 次
└────────┬────────┘
         │ 仍然失败
         ▼
┌─────────────────┐
│ 记录错误日志     │ 标记该 chunk/chapter 为 ERROR
│ 继续下一任务     │ 不中断整书解析
└─────────────────┘
```

### 3.5 阶段级参数覆盖

每个阶段可独立配置以下参数：

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| temperature | number | 0.2 | 提取任务 0.1-0.2，推理任务 0.3-0.5 |
| maxOutputTokens | number | 8192 | 按模型能力和输出需求调整 |
| topP | number | 1.0 | 核采样参数 |
| maxRetries | number | 2 | 同模型最大重试次数 |
| retryBaseMs | number | 600 | 重试基础间隔（毫秒）|

> **Prisma relation 显式命名说明（v1.2）**
>
> `ModelStrategyConfig` 同时持有 `bookId` 和 `jobId` 两个可空 FK，且 scope 决定哪个 FK 有值。为避免 Prisma 关系推断歧义，必须在双方都使用显式 relation name：
> - `ModelStrategyConfig` 侧：`@relation("BookStrategies", ...)` / `@relation("JobStrategies", ...)`
> - `Book` 侧：`modelStrategies ModelStrategyConfig[] @relation("BookStrategies")`
> - `AnalysisJob` 侧：`modelStrategies ModelStrategyConfig[] @relation("JobStrategies")`

> **GLOBAL 唯一性约束补充说明（v1.1）**
>
> Prisma 的 `@@unique([scope, bookId])` 对 `bookId=null` 的行为因数据库而异：PostgreSQL 中 `UNIQUE(scope, book_id)` 允许多条 `(GLOBAL, NULL)` 记录（NULL != NULL）。因此 **GLOBAL 最多一条的约束不能仅靠 Prisma schema 表达**，需要在 migration SQL 中手动添加 partial unique index：
>
> ```sql
> CREATE UNIQUE INDEX uq_strategy_global
>   ON model_strategy_configs (scope)
>   WHERE scope = 'GLOBAL';
> ```
>
> 同时在 Service 层使用 `findFirst + update/create` 模式保证幂等写入（v1.2 修正：Prisma Client 无法直接将 partial unique index 作为 `upsert.where` 的类型安全输入，因此不能使用 `upsert`）：
>
> ```typescript
> // service 层保证 GLOBAL 唯一（v1.2 修正，v1.2.1 补充并发保护）
> async function upsertGlobalStrategy(stages: JsonValue) {
>   const existing = await prisma.modelStrategyConfig.findFirst({
>     where: { scope: 'GLOBAL' },
>   });
>   if (existing) {
>     return prisma.modelStrategyConfig.update({
>       where: { id: existing.id },
>       data: { stages },
>     });
>   }
>   try {
>     return await prisma.modelStrategyConfig.create({
>       data: { scope: 'GLOBAL', stages },
>     });
>   } catch (error) {
>     // v1.2.1：并发写入时 partial unique index 触发唯一约束冲突，
>     // 降级为读取已存在记录并 update
>     if (isUniqueConstraintError(error)) {
>       const latest = await prisma.modelStrategyConfig.findFirst({
>         where: { scope: 'GLOBAL' },
>       });
>       if (!latest) throw error; // 理论不可达，防御性处理
>       return prisma.modelStrategyConfig.update({
>         where: { id: latest.id },
>         data: { stages },
>       });
>     }
>     throw error;
>   }
> }
>
> // isUniqueConstraintError 辅助函数（判断 Prisma P2002 错误码）
> function isUniqueConstraintError(error: unknown): boolean {
>   return (
>     error instanceof Prisma.PrismaClientKnownRequestError &&
>     error.code === 'P2002'
>   );
> }
> ```

export const BUSINESS_PIPELINE_STAGES: PipelineStage[] = [

//
// v1.2.1 命名消歧约定：
//   modelId     → ai_models.id（数据库 UUID 主键，用于 DB 查询/日志记录/前端表单值）

export interface ResolvedStageModel {
  modelId: string;        // ai_models.id — DB UUID
  provider: AiProviderName;
  modelName: string;      // ai_models.modelId — Provider API model identifier
  displayName: string;    // ai_models.name — 用户友好展示名

  apiKey: string;         // 解密后明文，仅内存

// Resolver 接口
export interface ModelStrategyResolver {
  /**
   * 解析指定阶段应使用的模型
   * 按 JOB → BOOK → GLOBAL → SYSTEM_DEFAULT 优先级查找
   * 只返回 isEnabled=true 的模型
   */
  resolveForStage(
    stage: PipelineStage,
    context: { jobId?: string; bookId: string }
  ): Promise<ResolvedStageModel>;

  /**
   * 解析 fallback 模型（当主模型调用失败时使用）
   */
  resolveFallback(
    context: { jobId?: string; bookId: string }
  ): Promise<ResolvedStageModel>;

  /**
   * 预加载所有阶段配置（避免每个阶段单独查库）
   * 在 job 开始时调用一次，结果缓存到内存
   */
  preloadStrategy(
    context: { jobId?: string; bookId: string }
  ): Promise<Map<PipelineStage, ResolvedStageModel>>;
}

// 工厂函数
export function createModelStrategyResolver(
  prismaClient: PrismaClient
): ModelStrategyResolver;

**实现要点**：
1. `preloadStrategy` 在 job 启动时调用一次，按 JOB → BOOK → GLOBAL → SYSTEM_DEFAULT 顺序加载，结果缓存到内存 Map
2. 每个模型加载后必须校验 `isEnabled === true`，不满足则跳到下一层
3. apiKey 通过 `readEncryptedApiKey()` 解密，仅在内存中持有
4. 参数合并：阶段配置中的 params 与 DEFAULT_STAGE_PARAMS[stage] 做 Object.assign 合并

> **缓存边界与配置变更策略（v1.1 补充）**
>
> **缓存生命周期**：preload 快照绑定单个 AnalysisJob 的运行周期。Job 启动时快照，Job 结束时释放。不跨 job 共享。
>
> **多 Worker 场景**：当前项目为单 Worker（Next.js server-side），无多进程竞争问题。每个 `runAnalysisJobById` 调用在自己的上下文中持有独立的 resolver 实例和缓存。
>
> **运行中配置变更**：Job 运行期间以启动时快照为准。如果管理员在 Job 运行中禁用了某个模型：
> - 已运行的阶段不受影响
> - 后续阶段仍使用快照中的模型（Provider API 层面，已获取的 apiKey 仍有效）
> - **唯一例外**：fallback 调用时，AiCallExecutor 可选择重新查库而非使用快照，以感知最新模型状态。此行为通过配置项 `fallbackRefreshOnRetry: boolean`（默认 false）控制
> - 模型禁用对**新启动的任务**立即生效
>
> **设计理由**：避免运行中配置漂移导致半完成 job 出现不可预期行为。一致性优先于实时性。

**废弃函数**：
- `ChapterAnalysisService.resolveAnalysisModelConfig()` → 删除
- `ValidationAgentService.resolveValidationModelConfig()` → 删除

### 5.2 模型路由与 fallback 设计

**新建文件**：`src/server/modules/analysis/services/AiCallExecutor.ts`

**职责**：封装「调用主模型 → 重试 → fallback → 记录日志」的完整流程。

```typescript
export interface AiCallOptions {
  stage: PipelineStage;
  jobId: string;
  chapterId?: string;
  chunkIndex?: number;
}

export interface AiCallResult<T> {
  data: T;
  usage: { promptTokens: number; completionTokens: number };
  durationMs: number;
  modelUsed: string;       // ai_models.id
  isFallback: boolean;
}

// v1.2 修正：callFn 必须返回 { data, usage }，executor 从中提取 usage 写入日志
export interface AiCallFnResult<T> {
  data: T;
  usage: AiUsage;
}

export interface AiCallExecutor {
  /**
   * 执行 AI 调用，自动处理重试和 fallback
   * callFn 接收 AiProviderClient 和阶段参数，返回 { data, usage }
   */
  execute<T>(
    options: AiCallOptions,
    callFn: (client: AiProviderClient, params: StageParams) => Promise<AiCallFnResult<T>>
  ): Promise<AiCallResult<T>>;
}

export function createAiCallExecutor(
  strategyResolver: ModelStrategyResolver,
  prismaClient: PrismaClient
): AiCallExecutor;
```

**execute 方法实现逻辑**：

```
1. 从 strategyResolver 获取 stage 对应的 ResolvedStageModel
2. 创建 AiProviderClient 实例
3. 尝试调用 callFn，计时
4. 成功 → 写入 analysis_phase_logs（status=SUCCESS）→ 返回结果
5. 失败且为可重试错误 → 指数退避重试（最多 params.maxRetries 次）
6. 重试全部失败 → 写入日志（status=RETRIED）
7. 从 strategyResolver 获取 FALLBACK 模型
8. **反自递归守卫（v1.2.1）**：若 FALLBACK 解析出的 modelId 与当前主模型 modelId 相同，视为无独立 fallback，跳过 fallback 阶段，直接写入日志（status=ERROR, error_message="fallback model same as primary, skipped"）→ 抛出 AiCallExhaustedError
9. 创建 fallback AiProviderClient 实例
10. 尝试调用 callFn（重试 1 次）
11. 成功 → 写入日志（status=SUCCESS, isFallback=true）→ 返回结果
12. 失败 → 写入日志（status=ERROR）→ 抛出 AiCallExhaustedError
```

function mergeChunkResults(results: ChapterAnalysisResponse[]): ChapterAnalysisResponse {
  const allMentions = results.flatMap(r => r.mentions);
  const allBiographies = results.flatMap(r => r.biographies);
  const allRelationships = results.flatMap(r => r.relationships);

  // mention 去重（v1.2 修正：统一为 3 元组，避免误删同人物在不同段落的合法提及）
  // 优先键：personaName + rawText + paraIndex
  // 若 paraIndex 不可用（某些模型未输出），则退化为 personaName + rawText + chunkIndex
  const dedupedMentions = deduplicateBy(
    allMentions,
    m => `${m.personaName}||${m.rawText}||${m.paraIndex ?? m.chunkIndex ?? ''}`
  );

  // relationship 去重（v1.2 增强：合并 evidence + 聚合 chapterNos）
  // 主键：sourceName + targetName + type
  // weight = max(existing, incoming)
  // evidences = union(existing, incoming).slice(0, 5)
  // chapterNos = union(existing, incoming) 去重
  const dedupedRelationships = mergeRelationships(allRelationships);

  // biography 去重：按 personaName + event 去重
  const dedupedBiographies = deduplicateBy(allBiographies, b => `${b.personaName}||${b.event}`);

  return {
    mentions: dedupedMentions,
    biographies: dedupedBiographies,
    relationships: dedupedRelationships,
  };
}

>

>
> **实现方式**：进程内 async mutex（推荐使用 `async-mutex` npm 包的 `Mutex` 类）
>
> ```typescript
> import { Mutex } from 'async-mutex';
>
> class AliasRegistryServiceImpl {
>   private mutex = new Mutex();
>   private cacheByBook = new Map<string, Map<string, AliasMappingResult[]>>();
>
>   async registerAlias(input: RegisterAliasInput, tx?: PrismaTransaction) {
>     const release = await this.mutex.acquire();
>     try {
>       // DB upsert + 刷新内存缓存
>     } finally {
>       release();
>     }
>   }
>
>   async lookupAlias(bookId: string, alias: string, chapterNo: number) {
>     // 读操作不加锁（内存 Map 读并发安全）
>     // 如果缓存未命中再加锁查 DB
>   }
> }
> ```
>
> **设计理由**：
> - Node.js 单线程事件循环中，async mutex 足以保证内存缓存一致性
> - 读操作不加锁（Map.get 是同步原子操作），只有写操作加锁
> - DB 层面的并发安全由 Prisma upsert 的 WHERE 条件保证
> - 如果未来扩展为多 Worker，需要将 alias 合并收敛到 DB 事务级锁或任务队列串行阶段

> 两种"默认"必须在 UI 上明确区分：
> - **推荐默认值**：前端产品预设（`RECOMMENDED_MODELS` 常量），用于帮助用户快速填表
> - **系统生效默认值**：后端 GLOBAL / SYSTEM_DEFAULT 配置
>
> 当 `initialStrategy = null`（未保存过策略）时，表单行为：
> - 所有 Select **保持空值**（placeholder "使用上级默认"），**不自动选中推荐模型**
> - 每个阶段右侧显示推荐标签（如"推荐：通义千问 Max"），仅作参考提示
> - 用户可通过"恢复推荐配置"按钮一键填充推荐模型到各 Select
> - 填充后表单进入"已修改未保存"状态，需点击"保存策略"才生效
>
> **设计理由**：避免"用户没保存但以为已生效"的错觉。空值 + 推荐标签 + 一键填充比自动预选更安全。

#### 交互规则

1. **模型选择 Select**：使用 shadcn/ui 的 `<Select>` 组件（Radix UI Select）
   - 下拉选项仅展示 `isEnabled=true` 的模型
   - 每个选项格式：`{model.name} ({model.provider} / {model.modelId})`
   - 未配置时默认显示 placeholder "使用上级默认"
   - 选中的模型如果与该阶段推荐模型匹配，显示 ⭐推荐 标记

2. **推荐模型标记**：每个阶段旁边显示推荐模型名称。推荐信息来自前端硬编码的 `RECOMMENDED_MODELS` 常量，**按 `provider + modelId` 二元组匹配**（例如 `{ provider: 'qwen', modelId: 'qwen-max' }`），不依赖数据库 UUID。匹配逻辑：遍历已启用模型列表，找到 `provider` 和 `modelId` 均相等的记录即为推荐命中。

```typescript
// RECOMMENDED_MODELS 常量示例
const RECOMMENDED_MODELS: Record<PipelineStage, { provider: string; modelId: string; label: string } | null> = {
  ROSTER_DISCOVERY:       { provider: 'qwen',    modelId: 'qwen-max',         label: '通义千问 Max' },
  CHUNK_EXTRACTION:       { provider: 'deepseek', modelId: 'deepseek-chat',   label: 'DeepSeek V3' },
  CHAPTER_VALIDATION:     { provider: 'qwen',    modelId: 'qwen-plus',        label: '通义千问 Plus' },
  TITLE_RESOLUTION:       { provider: 'qwen',    modelId: 'qwen-max',         label: '通义千问 Max' },
  GRAY_ZONE_ARBITRATION:  { provider: 'qwen',    modelId: 'qwen-plus',        label: '通义千问 Plus' },
  BOOK_VALIDATION:        { provider: 'qwen',    modelId: 'qwen-max',         label: '通义千问 Max' },
  FALLBACK:               { provider: 'qwen',    modelId: 'qwen-plus',        label: '通义千问 Plus' },
};

// 匹配函数
function findRecommendedModel(stage: PipelineStage, enabledModels: AiModel[]): AiModel | undefined {
  const rec = RECOMMENDED_MODELS[stage];
  if (!rec) return undefined;
  return enabledModels.find(m => m.provider === rec.provider && m.modelId === rec.modelId);
}
```

3. **"恢复推荐配置"按钮**：点击后将所有阶段设置为推荐模型（通过 `provider + modelId` 在可用列表中查找匹配且已启用的记录），不存在的阶段留空

4. **高级参数区域**：默认收起，点击展开显示 temperature / maxTokens 输入框
   - temperature: number input, step=0.05, min=0, max=2, placeholder="默认 0.2"
   - maxOutputTokens: number input, step=1024, min=256, max=65536, placeholder="默认 8192"
   - 不填则不覆盖，使用系统默认值

5. **提交 payload**：只提交有值的阶段（未选择模型的阶段不包含在提交数据中，表示"使用上级默认"）

```typescript
// 提交结构示例

    // 未配置的阶段不包含，表示继承上级
  }
}
```

## 附录 A：实现红线（Must Not）

> 本附录面向 Codex Agent 和人类开发者，列出在实现本方案时**绝对不可违反的硬约束**。违反任何一条应视为实现缺陷，必须修复后才能合并。

| # | 红线规则 | 理由 |
|---|---------|------|
| R-01 | **禁止对 `PipelineStage` 枚举做 `Object.values()` 遍历**，展示/遍历/统计场景必须使用 `BUSINESS_PIPELINE_STAGES` 数组 | `Object.values()` 会混入 `FALLBACK`，导致前端多渲染一个阶段、统计数据错乱 |
| R-02 | **禁止在 `AiCallExecutor` 的 fallback 路径中递归调用自身** | fallback 模型与主模型相同时会造成无限循环；必须先执行反自递归守卫检查 |
| R-03 | **禁止将 `ResolvedStageModel.modelId`（DB UUID）与 `ai_models.modelId`（Provider 模型标识符）混用** | 两者含义完全不同，混用会导致查库失败或调用错误模型；代码中必须遵守 `modelId / modelName / displayName` 三名约定 |
| R-04 | **禁止在 Prisma Client 中对 partial unique index 使用 `upsert`** | Prisma Client 不支持将 partial unique index 作为类型安全 `where` 输入；必须使用 `findFirst + update/create` 模式 |
| R-05 | **禁止在 `generateJson` 中将 system 和 user 内容合并为单条 message 发送** | system/user 分离是提升模型规则遵循能力的关键手段，且影响 prompt cache 命中率 |
| R-06 | **禁止在 Provider Client 中硬编码 `temperature` / `maxOutputTokens` 等参数** | 这些参数必须通过 `AiGenerateOptions` 从阶段配置传入，否则按阶段差异化调参无法生效 |
| R-07 | **禁止 `analysis_phase_logs` 中 `stage` 字段写入 `'FALLBACK'`** | `stage` 字段记录的是业务阶段（如 `CHUNK_EXTRACTION`），fallback 身份通过 `is_fallback=true` 标记 |
| R-08 | **禁止 `callFn` 返回值省略 `usage` 字段** | `AiCallExecutor` 依赖 `AiCallFnResult<T>.usage` 写入日志，省略会导致 token 统计缺失 |
| R-09 | **禁止在 Job 运行期间实时读取模型策略配置**（除非 `fallbackRefreshOnRetry=true`） | Job 启动时 `preloadStrategy()` 快照一次，运行中使用快照，避免配置漂移导致半完成 Job 行为不可预期 |
| R-10 | **禁止保留旧的 `resolveAnalysisModelConfig` / `resolveValidationModelConfig` / `createRuntimeAiClient` 方法** | 这些方法是单模型一刀切架构的产物，必须完全删除，所有模型解析走 `ModelStrategyResolver` |
