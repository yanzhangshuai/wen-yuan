2. 核心代码定位结果
2.1 模型调用入口
文件：src/server/providers/ai/index.ts（71行）

定义 AiProviderClient 统一接口，仅有一个方法：generateJson(prompt: string): Promise<string>
createAiProviderClient() 工厂函数按 provider 名称创建客户端
支持4种Provider：gemini / deepseek / qwen / doubao
为什么关键：所有AI调用都经过此入口，模型切换完全依赖数据库配置，代码层面无差异化处理
2.2 Prompt模板定义
文件：src/server/modules/analysis/services/prompts.ts（~580行）

buildRosterDiscoveryPrompt() — Phase 1 人物名册发现（完整章节输入）
buildChapterAnalysisPrompt() — Phase 2 章节分段分析（分片输入，核心提取）
buildTitleResolutionPrompt() — Phase 5 称号溯源（批量推断历史真名）
buildTitleArbitrationPrompt() — Phase 3 灰区仲裁（判断称谓是否人格化）
buildChapterValidationPrompt() — 章节验证
buildBookValidationPrompt() — 全书验证
为什么关键：Prompt质量直接决定模型效果上限，是选型的核心评估依据
2.3 书籍解析主流程
文件：src/server/modules/analysis/services/ChapterAnalysisService.ts（~950行）

analyzeChapter() — 单章节解析主流程
splitContentIntoChunks() — 文本分片（MAX_CHUNK_LENGTH=2000字符, OVERLAP=200字符）
persistResult() — 事务持久化（mentions/biographies/relationships）
resolvePersonaTitles() — Phase 5 称号真名溯源
runGrayZoneArbitration() — Phase 3 灰区仲裁
为什么关键：定义了所有AI调用的输入构造、并发策略、重试逻辑和结果持久化
2.4 配置来源
文件：src/server/modules/analysis/config/pipeline.ts

关键参数：
chunkAiConcurrency: 3（同时解析分片数）
chapterConcurrency: 1（章节间串行）
aliasRegistryMinConfidence: 0.75
personaResolveMinScore: 0.72
chapterValidationRetries: 1
dynamicTitleResolutionEnabled: false（Phase 2关闭）
llmTitleArbitrationEnabled: false（Phase 3关闭）
模型配置：完全存储在数据库 ai_models 表中（provider/modelId/baseUrl/apiKey/isEnabled/isDefault），支持运行时切换。

3.2 单章节AI调用次数分析
假设一个章节约 8000 字：

Phase 1 名册发现：1次（完整章节，~8000字输入）
Phase 2 分片分析：4次（2000字/片，3路并发）
章节验证：1次

4. 项目真实任务判断
4.1 当前项目究竟在解决什么问题
文渊（wen-yuan）是一个中国古典文学人物知识图谱构建系统，核心目标是：

从中文古典小说（以明清白话小说为主，如《儒林外史》）中自动提取人物实体、生平事件、人物关系
构建结构化的人物关系图谱并支持人工审核
4.2 任务类型
整本书级别的逐章解析（支持全书/章节范围/指定章节三种粒度）
多阶段流水线：名册发现 → 分片提取 → 实体对齐 → 验证修复 → 称号溯源
输入：用户上传的电子文本（经章节拆分后的纯文本），无 OCR 环节
输出：结构化 JSON → 写入 PostgreSQL（personas / mentions / biography_records / relationships / alias_mappings）
4.3 当前最重要的约束
中文古典文学理解能力：文言文/半白话文、官衔称谓、历史人物辨识
JSON 输出稳定性：所有AI调用都要求返回严格JSON，解析失败直接报错
称谓消歧能力：同一人物有多种称呼（名/字/号/官职/尊称），需正确归并
成本控制：单本书 350+ 次AI调用，若用昂贵模型，成本显著
4.4 当前最大的瓶颈
所有AI步骤使用同一个模型：Phase 1/2/5/验证/仲裁全部走同一个 book.aiModelId 或默认模型，无法按任务类型差异化分配
分片粒度偏小（2000字符≈1000汉字），导致调用次数多、上下文碎片化
章节串行（chapterConcurrency: 1），整书解析耗时长
输入为电子文本而非OCR，但代码中无OCR相关逻辑
5. 当前方案问题分析
5.1 模型选择问题
问题1：单模型一刀切

代码中 resolveAnalysisModelConfig() 和 resolveValidationModelConfig() 逻辑完全相同，验证步骤复用了与解析相同的模型
Phase 1（名册发现）需要长上下文 + 强 NER 能力，Phase 2（分片分析）只需中等能力但调用次数最多，Phase 5（历史溯源）需要领域知识——三者能力需求截然不同，不应共享同一模型
建议：在 AnalysisJob 或 Book 配置中支持按阶段指定模型
问题2：无 fallback 机制

analyzeChunkWithRetry() 只做同模型重试（最多2次），不尝试切换到备用模型
章节级重试 (CHAPTER_MAX_RETRIES: 2) 也只是同模型重试
唯一的降级处理是验证失败时跳过验证继续（validation.degraded），而非换模型重试
5.2 Prompt 设计问题
问题1：Prompt 过于庞大

buildChapterAnalysisPrompt() 包含 12 条 Strict Rules + JSON Schema 示例 + Known Entities 上下文 + 正文，Prompt 本身（不含正文）约 2000-3000 tokens
对于每个 2000 字符的分片，Prompt 开销占比高
问题2：Prompt 缺少 system/user 分离

所有 Provider 都把 Prompt 作为单条 user message 发送
特别是 DeepSeek/Qwen/Doubao 支持 system message，但未利用，可能影响规则遵循效果
问题3：Prompt 中的泛化称谓示例硬编码

通过 GENERIC_TITLES_EXAMPLE 动态生成（取前15个），但示例数据偏少，AI 需要自行泛化
5.3 流程设计问题
问题1：分片过小（2000字符 ≈ 1000汉字）

当前大模型普遍支持 8K-128K 上下文，2000字符分片过于保守
导致调用次数多、上下文断裂（overlapping 200字符不足以覆盖人物指代链）
章节合并 mergeChunkResults() 只是简单 flatMap，无去重/冲突处理
建议：对支持长上下文的模型，可增大到 6000-8000 字符/片，减少 2/3 调用次数
问题2：Phase 1 输入完整章节但无长度保护

discoverChapterRoster() 将整章内容（可能 2-5 万字）直接发送，无分片
若使用上下文窗口较短的模型（如 DeepSeek V3 的 64K），长章节可能被截断
缺失信息：代码中未设置 maxInputTokens 限制
问题3：章节间串行 + 无 pipeline 重叠

Phase 1 和 Phase 2 之间是串行关系（Phase 2 依赖 Phase 1 结果）
但不同章节之间理论上可以并行（chapterConcurrency: 1 过于保守）
5.4 成本问题
问题：对调用次数无感知

无 token 计数、无成本估算、无预算限制
350+ 次调用/本书，若使用 GPT-4 级别模型，单本书成本可达 $5-15
建议：增加 token 计数日志和成本估算
5.5 稳定性问题
问题1：JSON 解析容错有限

repairJson() 函数存在（在 types/analysis.ts），但仅做基础修复
模型输出包含 Markdown 代码块（```json 包裹）时可能失败
Prompt 中已写 "Do not use markdown code blocks"，但不同模型遵循程度不同
问题2：maxOutputTokens 不统一

GeminiClient: 8192
OpenAiCompatibleClient: 8192 (max_tokens)
DeepSeekClient: 未设置 max_tokens（依赖 API 默认值）
风险：DeepSeek 默认可能截断输出
5.6 工程实现问题
问题1：重复的模型配置读取代码

ChapterAnalysisService.resolveAnalysisModelConfig() 和 ValidationAgentService.resolveValidationModelConfig() 是几乎完全相同的代码
应抽取为共享工具函数
问题2：Provider Client 缺少 temperature/maxTokens 参数化

temperature 硬编码为 0.2（在各 Client 中）
不同任务（提取 vs 验证 vs 创造性溯源）可能需要不同 temperature

7. 逐模型分析
7.1 DeepSeek R1
优势：推理能力极强，复杂逻辑推理和交叉校验能力突出；价格低
劣势：推理速度慢（思维链输出导致延迟高）；不适合高频调用；输出可能包含思考过程干扰JSON解析
适合：Phase 5 称号溯源（需历史推理）、全书验证（需全局逻辑推理），调用次数少（≤5次/书）
不适合：Phase 2 分片提取（每本书 200+ 次调用，速度瓶颈严重）
风险：R1 的 "思维链" 输出可能在 response_format: json_object 下不稳定，需实测
7.2 DeepSeek V3
优势：速度快、成本极低（输入 ¥1/M tokens，输出 ¥2/M tokens⚠️）；中文能力好；JSON 模式支持良好
劣势：64K 上下文对长章节名册发现有风险；古典文言文能力不如通义千问
适合：Phase 2 分片提取的主力模型（调用量大、成本敏感）；灰区仲裁
不适合：超长章节的 Phase 1（需 >64K 上下文时）
风险：DeepSeek API 稳定性时有波动（历史记录有限流），需做好重试
7.3 通义千问 Max
优势：中文能力业界顶级；128K 长上下文；JSON 输出稳定；结构化抽取能力强；阿里云基础设施稳定
劣势：成本中等（约 ¥20-40/M tokens⚠️）；速度不如 V3/Flash
适合：Phase 1 名册发现（需完整章节理解 + 强 NER + 长上下文）；Phase 5 称号溯源（需历史文化知识）
不适合：Phase 2 大量分片调用（成本过高）
风险：低，API 稳定性好
7.4 通义千问 Plus
优势：成本低（约 ¥4-8/M tokens⚠️）；速度快；中文能力良好；128K 上下文
劣势：复杂推理能力不如 Max；长文本理解深度不如 Max
适合：Phase 2 分片提取（成本/性能平衡好）；章节验证
不适合：需要深度理解的 Phase 1 长章节
风险：低
7.5 豆包 Pro
优势：成本极低（可能是最便宜的选择之一）；速度快
劣势：中文古典文学理解能力偏弱；JSON 输出稳定性不如头部模型；32K 上下文限制
适合：简单分类任务（灰区仲裁——但当前该功能关闭）；可作为 fallback
不适合：核心提取步骤（Phase 1/2）；验证
风险：能力不足可能导致幻觉率升高、JSON 解析失败率升高
7.6 Gemini Flash
优势：超长上下文（1M tokens）；速度极快；JSON mode 原生支持（responseMimeType: "application/json"）；成本低
劣势：中文古典文学理解明显弱于国产模型；文言文/半白话文场景可能出现理解偏差；需科学上网（API 可达性问题）
适合：Phase 1 名册发现（利用超长上下文一次读完整章节）但需降低对文言文理解的预期
不适合：需要深度中文理解的 Phase 2 提取；Phase 5 历史人物溯源
风险：API 可达性在中国大陆不稳定（已配置 generativelanguage.googleapis.com）
7.7 GLM-4（补充）
优势：中文文学理解能力极强（智谱在古典文学NLP领域有专项研究）；128K上下文；OpenAI兼容接口
劣势：成本中等；社区生态不如 DeepSeek/通义
适合：Phase 1 名册发现；Phase 5 称号溯源（特别是涉及历史人物辨识）
不适合：大量分片调用（成本）
风险：需评估实际JSON稳定性
8. 推荐方案
最优单模型方案
通义千问 Max

理由：中文能力最强 + 128K长上下文 + JSON稳定 + API稳定 + 结构化抽取优秀
适合场景：初期功能验证、小规模解析
风险：成本偏高（大量分片调用时）
最优性价比方案
DeepSeek V3

理由：成本极低 + 速度快 + 中文能力良好 + 64K上下文对当前2000字符分片绰绰有余
适合场景：批量解析、成本敏感场景
风险：API 偶尔限流；长章节 Phase 1 可能上下文不足
最优稳定性方案
通义千问 Plus（主力） + 通义千问 Max（fallback）

理由：同厂商双模型，API行为一致，fallback切换零成本
适合场景：生产环境长期运行
最适合批量处理方案
DeepSeek V3（Phase 2 主力） + 通义千问 Plus（Phase 1/验证）

理由：批量场景下 V3 的成本优势压倒性
Top 3 推荐排名
🥇 通义千问 Max（综合最优，适合验证和小规模生产）
🥈 DeepSeek V3（性价比最优，适合批量生产）
🥉 通义千问 Plus（中间路线，适合长期运行）
